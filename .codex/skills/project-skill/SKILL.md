# 项目开发技能

当任务涉及代码编写、代码评审、调试、重构、配置调整、测试补充或构建修复时，使用本技能。

## 来源与适配

本技能结合本项目 `docs/` 中的环境、工作流、开发标准，并改写保留 `forrestchang/andrej-karpathy-skills` 的通用工程行为规则。

来源仓库: https://github.com/forrestchang/andrej-karpathy-skills
许可: MIT

## 快速检查

开始前：

- 是否已经读过 `AGENTS.md` 和 `AGENTS.override.md`？
- 是否已经确认相关代码、配置、测试和文档？
- 是否存在多个合理解释？如果存在，先列出并确认。
- 是否存在更简单的实现路径？如果存在，优先提出。

写文件前：

- 输出计划。
- 列出涉及文件。
- 说明逻辑修改点。
- 说明 Docker 卷挂载路径；不涉及写“不涉及”。
- 说明验证方式和成功标准。

完成后：

- 运行相关测试、构建、类型检查或人工验证。
- 对照原始目标说明是否达成。
- 说明未验证项、原因和残余风险。

## 四项工程行为

### 1. 三思而后行

不要静默选择一种解释并直接实现。先管理不确定性：

- 明确说明假设。
- 明确说明歧义。
- 明确说明技术取舍。
- 必要时提出澄清问题。
- 如果用户要求的方案更复杂或风险更高，给出更简单方案及原因。

### 2. 简洁优先

实现只服务于当前目标：

- 不添加请求之外的功能。
- 不为单次使用代码创建抽象。
- 不添加未要求的灵活性或配置项。
- 不为理论上不可能的场景写复杂处理。
- 代码明显臃肿时，先压缩方案。

### 3. 精准改动

控制 diff，保护团队协作效率：

- 不修改无关文件。
- 不顺手重排格式、重写注释或调整命名。
- 不重构没有坏的代码。
- 匹配项目现有风格。
- 只清理本次改动造成的无用代码。

### 4. 目标驱动

把任务转成可验证目标：

- “修复 bug”要落实为“复现 bug -> 修复 -> 验证不再复现”。
- “添加校验”要落实为“无效输入失败、有效输入通过、错误信息符合预期”。
- “重构”要落实为“重构前后测试通过，外部行为不变”。
- 多步骤任务每一步都要有独立验证点。

## 环境规范

Python：

- 使用仓库本地 `.venv`。
- 使用 Python 3.10+。
- 使用 `uv` 管理依赖。
- 禁止直接使用 `pip`。

Node.js：

- 使用 Node.js 20+。
- 优先使用 `pnpm`。

Docker：

- Docker 文件放在 `docker/`。
- 环境变量放在 `docker/.env`。
- 卷挂载放在 `docker/volume/`。
- 端口映射遵循 `50000 + 容器端口`。
- 下载源优先使用国内源，例外必须说明。

## 安全规则

- 不输出或写入 API key、MCP header、数据库凭据和敏感工具输入。
- 不把个人路径、个人 token 或私有账号配置写入团队文件。
- 必需工作流不得依赖远程服务。
- 配置值不得硬编码到业务逻辑。

## 反模式

- 未确认需求就实现数据导出、权限、批量操作或破坏性操作。
- 为简单函数引入策略模式、插件系统或复杂配置。
- 修一个 bug 时顺手重写相邻模块。
- 改动格式导致 diff 难以评审。
- 没有复现问题就直接修 bug。
- 用“应该可以”代替测试、构建或明确的人工验证。
