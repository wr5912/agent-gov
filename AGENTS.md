# Agent 指令

本文件是团队级 Codex 项目说明。所有在本项目内工作的 Agent 都必须先阅读并遵守本文件。

## 读取顺序

1. 先阅读本文件。
2. 如存在 `AGENTS.override.md`，继续读取并应用其中的项目覆盖说明。
3. 读取 `.codex/rules/project.rules` 中的统一规则。
4. 涉及代码编写、重构、调试或评审时，启用 `.codex/skills/project-skill/SKILL.md` 中的项目技能。

## 协作语言

- 所有对话、文档、代码注释和提交说明默认使用中文。
- 保留代码标识符、命令、日志关键字和第三方 API 名称的原文。

## 工作流

所有非琐碎代码变更必须遵循 Analyze -> Plan -> Execute -> Verify。

### Analyze

- 先阅读相关代码、报错、需求、已有文档和配置。
- 能从仓库确认的事实，先通过只读方式确认。
- 显式说明关键假设、歧义和技术权衡。
- 需求不清或存在安全/数据风险时，先提问，不要猜测实现。

### Plan

执行前必须输出修改计划，并等待用户确认。计划至少包含：

1. 涉及的文件列表。
2. 逻辑修改点。
3. 与四项准则的对应关系。
4. Docker 卷挂载路径；不涉及时写明“不涉及”。
5. 验证方式和成功标准。

### Execute

- 仅在用户确认后写文件。
- 只改完成目标所需的文件和逻辑。
- 优先复用现有模式、工具和约定。
- 不引入无当下价值的依赖、抽象、配置或“未来可能需要”的功能。
- 如果发现无关问题，记录并说明，不要顺手重构或删除。

### Verify

- 对照原始目标验证结果。
- 运行相关测试、构建、类型检查或端到端检查。
- 如无法验证，说明原因、残余风险和后续路径。
- 涉及 Docker 持久化时，检查 `docker/volume/` 数据路径。

## 四项核心准则

### 三思而后行

- 不做未验证假设。
- 多种解释并存时，先列出解释和取舍。
- 有更简单或风险更低的方案时，明确提出。
- 困惑时停止推进，说明不清楚的点并请求澄清。

### 简洁优先

- 用最少代码解决当前问题。
- 不为一次性逻辑创建抽象。
- 不添加未被要求的灵活性、配置项或扩展点。
- 如果实现明显臃肿，先收缩方案再继续。

### 精准改动

- 每一行修改都应能追溯到用户目标。
- 匹配项目现有风格，即使个人偏好不同。
- 不改动与任务无关的格式、注释、结构或命名。
- 只清理本次改动造成的无用导入、变量、函数和文件。

### 目标驱动

- 把任务转成可验证的成功标准。
- 修 bug 时优先复现问题，再实现修复。
- 重构时先确认现有行为，再证明行为未变。
- 多步骤任务要小步交付，每步都有验证点。

## 环境规范

### Python

- 必须使用仓库本地 `.venv`。
- Python 版本必须为 3.10+。
- 包管理器使用 `uv`，禁止直接使用 `pip`。
- 创建环境使用 `uv venv .venv`。
- 安装依赖使用 `uv pip install <package>` 或 `uv pip install -r requirements.txt`。
- 运行 Python 工具优先使用 `.venv/bin/python`，除非当前 shell 已激活 `.venv`。
- 不要使用系统 Python 或 Conda Python 执行项目 runtime、测试、依赖检查或脚本。

### Node.js

- Node.js 版本使用 20+。
- 前端包管理器优先使用 `pnpm`，已有 npm 脚本可按项目现状继续使用。
- 常用命令为 `pnpm install`、`pnpm dev`、`pnpm build`。

### Docker

- Docker 文件统一放在 `docker/`。
- 环境变量放在 `docker/.env`。
- 卷挂载统一使用 `docker/volume/` 下的目录，禁止挂载到项目路径外。
- 容器内部端口小于 10000 时，宿主机端口映射为 `50000 + 原端口`，例如 `5432 -> 55432`。
- Dockerfile、CI 安装脚本和镜像构建脚本中，所有可配置下载源必须优先使用国内源。
- apt 源优先使用阿里云镜像，npm registry 优先使用淘宝源 `https://registry.npmmirror.com`，Python/pip/uv 源优先使用阿里云 PyPI 镜像或项目已配置镜像。
- 如某依赖没有可用国内源，必须在 Dockerfile 注释或变更说明中写明原因。

## 安全与产品不变量

- 离线模式是产品不变量；必需工作流不得依赖远程服务。
- 不得将API key提交到git仓库。
- 不要硬编码配置值。环境变量放 `docker/.env`，应用配置放 `config/*.yaml` 或 `config/*.json`。

## Git 提交

提交信息使用：

```text
<type>: <描述>
```

常用类型：

- `feat`: 新功能
- `fix`: 修复 bug
- `refactor`: 重构
- `docs`: 文档更新
- `test`: 测试
- `chore`: 构建或工具变更
- `perf`: 性能优化
- `ci`: CI/CD 变更
