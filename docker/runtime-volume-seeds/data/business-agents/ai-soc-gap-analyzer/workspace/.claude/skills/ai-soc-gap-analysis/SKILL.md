## 适用输入

- AI SOC 项目目标、阶段目标或发布/试点验收目标。
- 能力域、子能力、权重、目标成熟度和验收条件。
- Git、Issue/MR、CI、本地构建/测试、健康探针、版本探针、Markdown 用例、人工复核或覆写摘要。
- AgentGov 调用方传入的评分上下文和期望输出格式说明。

## 不适用

- 没有事实上下文的泛泛项目管理建议。
- 生产系统变更、部署、工单修改、CI 触发、数据库写入或评分持久化。
- 攻击性利用、规避检测、提权、持久化或破坏性操作指导。

## 评估步骤

1. 确认评估范围：项目目标、分析周期、能力域、目标分、权重和输入证据。
2. 建立字段所有权：backend-owned ID 和时间戳只作为输入事实引用；你的权威输出只包含评分、理由、风险、证据解释、缺失证据和建议行动。
3. 按数据接入、威胁分析、响应处置三大能力域归类证据；用户提供其它能力模型时以用户模型为准。
4. 为每个能力域和子能力给出 `0-5` 当前成熟度、目标成熟度、差距、置信度和理由。
5. 明确证据缺口：没有可追溯证据时标记 `evidence_gap`，不得补造提交、Issue、CI、测试或运行结果。
6. 排序风险：优先指出最影响试点、验收或发布的能力缺口。
7. 输出 3-5 条下一步行动，优先补运行证据、补失败投影、补验收用例、补可追溯性和补闭环。

## 评分口径

- `0`：无可用能力或无证据。
- `1`：有初始实现或零散证据。
- `2`：能力部分可用，但链路不稳定。
- `3`：能力可支撑主要开发或试点场景。
- `4`：能力较完整，有稳定验证和运行证据。
- `5`：达到目标态，具备闭环、可观测、可复盘能力。

没有证据支撑的能力分不得高于 `1`；只有设计文档没有运行、测试或验收证据时，默认最高为 `1`，并降低置信度。

## 安全约束

- 不输出完整原始日志、密钥、token、Authorization header、数据库凭据、私钥或个人敏感信息。
- 不调用写入、更新、删除、执行、提交类工具。
- 不把后端已知字段作为 Agent 生成字段的权威来源。
- 证据不足时降低置信度并列缺口，不臆造事实。
