> 版本：v2.7（完整自包含稿）

## 0. 治理对象预检（先建模，后画图）

> 本节是地基。任一格说不清，就先补清楚再改下游界面。

### 0.1 治理对象矩阵

| 维度 | 结论 |
| --- | --- |
| 被治理对象 | 一支**业务 Agent 舰队**（N 个业务 Agent），以及支撑闭环的**治理 Agent（governor）本身**与沉淀的**资产**。main-agent 只是舰队成员 / 初始样板，不是边界。事项级治理单元是 `ImprovementItem`，归属某个业务 Agent。 |
| 治理执行者 | 组合：① 后端规则 / 状态机（生命周期合法转移、发布门禁、反馈归属、`improvement_stage` 单一来源）；② 治理 Agent governor（归因 / 方案 / 回归用例治理，对用户隐藏执行细节）；③ 人工确认（归因 / 方案 / 发布关键判断点）。 |
| 资产类型 | 四类：数据资产（run / trace / feedback / eval result / release event）、方法论资产（归因法 / 优化 SOP / 发布策略）、执行资产（prompt / skill / profile / 回归用例）、审计资产（改进事项全链路 + diff / 快照 / 事件）。跨 Agent 方法论资产复利为 Wave 3 预留。 |
| 生命周期 | 业务 Agent：draft / active / evaluating / deprecated / archived。ImprovementItem：7 段链路（见 §7），由后端状态机统一管理合法转移。 |
| 反馈归属 | 反馈 → ImprovementItem → 归属某业务 Agent + agent version + 场景。`agent_id` 是稳定归属主键。 |
| 当前实现边界（新分支基线） | 闭环引擎（feedback / attribution / optimization_plan / eval / release）存在但为用户触发；事项层尚未统一；无自动编排；无统一术语 / 主题。 |
| 目标能力边界 | 统一 `ImprovementItem` 领域实体（本期）+ 治理工作台外壳（本期）+ 多 Agent scoping（本期）+ 闭环引擎对接（W2）+ 自动化编排 / 相似度归并（W2）+ 资产复利中心（W3 预留）。 |

### 0.2 闭环链路（终点是资产 Registry）

```text
业务 Agent 舰队
  → 运行(run / trace)
  → 反馈(feedback)
  → 归并 → 改进事项(improvement_item)         ← 事项级单一领域实体（本期建立）
  → 归因(attribution)
  → 优化方案(optimization_plan)
  → 执行(execution / agent version)
  → 回归 + 评估(regression / eval)
  → 版本 / 发布(release)
  → 资产 Registry(回归 / 方法论 / 执行 / 审计资产)   ← 复利为 Wave 3 预留
```

### 0.3 风险自检

- 把当前实现误当长期边界？→ §0.1 已舰队化，main-agent 仅为舰队成员。
- 把数据资产误当全部资产？→ 四类资产齐列，复利预留。
- 混淆业务 Agent 与治理 Agent？→ §0.1 分列；governor 在 §0.9 作一等角色建模。
- 缺反馈路由与资产归属？→ 反馈必落到某业务 Agent 名下的 ImprovementItem。

### 0.4 能力边界分层（greenfield 口径，诚实标注本期 vs 后续）

| 能力 | 本期(W1) | 后续 | 说明 |
| --- | --- | --- | --- |
| 统一术语表 + 主题 token 落地 | ✅ W1 | — | 前后端同名同义、状态色单一来源 |
| `ImprovementItem` 事项级领域实体（持久化 + 状态机 + Agent scoping + `/api/improvements`） | ✅ W1 | — | 真实新实体，非投影 |
| 治理工作台前端外壳（导航 / 改进 surface / 每态一主动作 / data-testid / 获取上下文） | ✅ W1 | — | 消费 `/api/improvements` |

### 0.5 严禁双轨：前端、后端、API、测试必须同名同义（强制硬性）

```text
严禁同一业务概念在不同层使用不同名称。
严禁 UI 为"用户友好"另起一套概念、后端为"实现方便"保留另一套概念。
严禁 API / DTO / Event / ContextPackage / Playwright selector 使用与领域模型不一致的别名。
```

以「改进事项」为例，前端、后端、API、上下文导出、测试必须统一为同一领域实体：

```text
中文展示名：改进事项
英文领域名：ImprovementItem
代码词根：improvement
唯一 ID：improvement_id
阶段字段：improvement_stage
状态字段：improvement_status
API 资源：/improvements
测试词根：improvement-*
```

直接弃用红线（新分支 greenfield，无兼容包袱）：

```text
旧物理表名 / 旧字段 / 旧服务名 / 旧 API 资源 / 旧事件名 / 旧测试选择器一律直接弃用。
不设计旧名兼容机制、不保留 legacy alias、不做双写双读、不保留旧入口。
新实现 / 新接口 / 新事件 / 新上下文包 / 新测试中不得出现任何旧名称。
发现旧名 → 唯一处理方式：直接改名，不做兼容。
```

> 注：本期是新分支 + 全新 schema，"直接重建"不触及历史运行态数据迁移；若未来需在既有库上演进，则该红线须配一次性迁移脚本（非双写双读）。

| 中文展示名 | 英文领域名 | 代码 / API 词根 | 统一 ID | 统一状态 / 阶段 | API / DTO | Playwright 词根 | 禁用别名 / 混用 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 业务 Agent | BusinessAgent | `agent` | `agent_id` | `agent_lifecycle` | `/agents` | `agent-*` | 不得与 Subagent 混用；`/agents` 在本平台＝BusinessAgent，与旧实现 `/api/agents`(=subagent) 语义相反，旧路由直接废弃 |
| Subagent | Subagent | `subagent` | `subagent_id` | `subagent_status` | `/subagents` 或配置内字段 | `subagent-*` | 不得叫业务 Agent |

### 0.7 术语对齐治理流程

```text
新增概念？
  → 先查术语表
  → 表中已有：必须复用既有术语
  → 表中没有：提交术语补充，明确 中文名 / 英文名 / 代码词根 / ID / 状态 / 禁用别名
  → 术语评审通过后，才能进入接口设计和页面设计
```

PR / 设计稿必须说明：涉及哪些统一术语、是否新增、是否有旧名（如有须已直接弃用）、是否影响 API / DTO / ContextPackage / selector、是否存在前后端双轨风险。验收红线：同概念跨层名称不一致 → 退回；新实现出现旧名 → 退回；前端造概念但后端无对应领域实体 → 退回。

### 0.8 视觉主题与颜色规范：可信治理工作台

默认主题 `Governance Light`。气质：干净 / 克制 / 可信 / 工程化 / 可审计 / 低干扰 / 高可读。禁止默认做成全黑科技风 / 赛博朋克 / 大面积渐变 / 荧光色 / 粒子动效 / AI 魔法感——AgentGov 是治理、调试、确认、回归、发布门禁类产品，目标不是"看起来像 AI"，而是"让用户稳定判断、低成本确认、减少误读"。

主题策略：

```text
浅色主工作台 / 中性灰信息层级 / 蓝色主动作 / 橙色待确认 /
红色阻塞·失败 / 绿色通过·完成 / 紫色仅表示 AI 推断·自动化 /
深色仅用于 Trace · Raw · Log · Diff · 代码类调试信息
```

| --- | --- | --- | --- |

按钮：主按钮（蓝底白字，仅当前状态唯一主动作用）、次级按钮（白底灰边，用于 修改 / 查看证据 / 获取上下文 / 重新整理 / 更多）、危险按钮（红字弱红底，不常驻主界面、入"更多"或二次确认）。深色区（`.debug-panel`：`--bg-debug` 底 / `--text-debug` 字）仅用于 Trace 摘要 / Raw JSON / Logs / Diff / Playwright 复现 / 开发者 Debug，正文工作台不得默认深色。

视觉红线：

```text
不得把 AI / 自动化设计成炫光、渐变、粒子或魔法按钮。
不得把风险、等待、成功、AI 推断混用同一种颜色。
不得用颜色替代文字状态（状态必须同时有文字 + data-state + aria-label）。
不得在不同页面为同一状态使用不同色。
不得在 CSS 中绕过 token 直接写散落颜色，不得为同一状态维护两套视觉语义。
```

### 0.9 跨代深度：舰队 / 治理 Agent / 资产复利对象建模（复利不在本期）

```text
舰队治理        一个 AgentGov 实例治理 N 个业务 Agent；全站按 agent_id scoping；
                改进事项 / 反馈 / 发布 / 资产均归属到具体业务 Agent。
治理 Agent      governor 是一等被建模角色：它既"执行治理"（归因 / 方案 / 回归），
                其自身也"可被治理"（版本 / eval / 回归）——本期对用户隐藏，仅审计可见。
资产复利(W3预留) 一个 Agent 沉淀的方法论资产（归因法 / 优化 SOP）可被舰队内其它 Agent 继承复用；
                这是跨代护城河，本期只建模与留出 /assets 出口，不实现写侧复利。
```

### 0.10 离线治理不变量（产品支柱）

```text
必需治理工作流（创建反馈 / 改进事项 / 确认 / 发布门禁 / 获取上下文）不得依赖远程服务。
平台始终提供本地化 LLM 模型；顶栏模型展示不得暗示核心流强依赖远程。
```

## 1. 设计总原则：把复杂留在系统内，把简单留给用户

> 用户只负责指出问题、确认理解、做关键判断；系统负责规范化、归因、生成方案、补充回归、推进发布。

前台 UI 不暴露过多系统内部对象和流水线动作，而围绕「改进事项」组织用户任务。下面是反复出现的过度设计根因与本方案的纠偏规则。

### 1.1 根因一：把系统内部动作暴露成用户按钮

例如把 `生成候选回归用例 / 关联改进事项 / 启动归因 / 确认并开始处理 / 查看自动化任务队列` 做成主按钮。这些对系统重要，但不是用户的自然任务。

规则：

```text
用户只看到"整理反馈、确认理解、确认归因、查看证据、查看发布状态"等自然动作；
归因、生成方案、生成回归用例、关联改进事项等由系统按自动化策略决定。
```

### 1.2 根因二：把"保存数据"和"推进流程"混成同一层交互

`确认保存` 是数据确认动作；`开始处理` 是治理流程动作。二者不应并列为主按钮。

```text
用户确认保存反馈
        ↓
系统按自动化策略继续处理
        ↓
必要时再向用户请求确认
```

因此创建反馈确认态只保留 `[修改] [重新整理] [确认保存]`。

### 1.3 根因三：一次状态给了多个主动作

若一页同时出现 `重新归因 / 重新生成方案 / 执行优化 / 运行回归 / 发布 / 生成用例`，用户会被迫理解系统状态机。

```text
每个状态只给一个主动作，其他动作进入"更多 / 高级"。
```

### 1.4 根因四：对象管理思维太重

不要围绕 `Feedback / Attribution / OptimizationPlan / RegressionAsset / Release / 自动化任务 / Trace` 等系统对象设计页面。用户心智是：哪里不对？系统理解对了吗？现在卡在哪？我要确认什么？能不能发布？UI 应从"对象管理型后台"收敛为"用户确认型治理工作台"。

### 1.5 根因五：按系统对象拆页面而不是按用户任务组织

反馈页与优化页拆成两个独立入口，会逼用户在页面之间手动找关联对象。

```text
反馈不是终点；反馈保存后自动生成或归并到"改进事项"；
用户后续都围绕"改进事项"继续处理。
```

### 1.6 最终约束（所有页面、接口、字段、测试共同遵守）

```text
1.  用户不填系统字段，只描述问题和确认判断。
2.  默认展示结论，不展示元数据。
3.  每个状态只保留一个主动作。
4.  完整链路存在于系统内，但默认只展示当前阶段和下一步。
5.  Trace / Raw JSON / 自动化任务 / Task Queue 只作为下钻信息，不作为主界面。
6.  反馈、归因、方案、回归、发布在前台统一收敛到"改进事项"。
7.  对外协作只提供"获取上下文"，不内置一键提交 Issue 或外部工单流。
8.  关键 UI 必须有稳定选择器和可断言状态，方便 AI 用 Playwright 自动化测试。
9.  每个改进事项 / 反馈 / 发布项都必须可追溯到某个业务 Agent。
10. 凡是依赖"系统自动推进 / 自动归并"的界面，必须可降级为"用户触发"，并明确标注该能力是否已启用。
```

## 8. 关联多个反馈的改进事项

### 8.1 设计意图

一个改进事项不一定只来自一条反馈。多个用户、多个 Run、多个时间点可能反复暴露同一问题：

```text
反馈 1：这个横向移动告警其实是误报。
反馈 2：AI 没注意到告警时间和事件时间不一致。
反馈 3：sec-ops-data 返回的数据像是模拟数据，不能支撑告警判断。
```

它们表面不同，但可能都指向同一根因：

```text
sec-ops-data MCP 返回的事件数据时间窗口不可靠，导致 Agent 对横向移动告警误判。
```

系统不应为每条反馈各生成一个独立改进事项，否则造成重复归因、重复生成方案、重复回归、局部过拟合、优化任务碎片化。更好的方式：

```text
多条相似反馈 → 系统归并 → 一个改进事项 → 一次归因 / 方案 / 回归 / 发布
```

### 8.2 用户看到的是"改进事项"，不是内部聚合细节

前台不让用户理解 `feedback_ids / similarity_score / 内部聚合键`。用户只需看到：这个改进事项由几条反馈归并而来、它们共同指向什么问题、系统为什么认为可以合并、用户是否认可这个合并。

## 15. 防过度设计与术语 / 颜色双轨检查清单

每新增字段 / 按钮 / 入口，先问：

```text

2.  系统能不能根据上下文自动推断？

6.  这个对象 / 动作归属到哪个业务 Agent？能否被 Agent scoping？
7.  它依赖的后端能力是否已建成？未建成时主按钮是否已降级为用户触发并标注？

9.  UI / API / DTO / 后端模型 / 事件 / 上下文包 / Playwright selector 是否同名同义？
10. 是否引入了禁用别名、历史旧名或双轨对象？
```

```text
  - 阶段模型：改进事项级 improvement_stage 单一来源派生

Wave 2 · 后端立项（目标能力）
  - ImprovementItem ↔ 既有闭环引擎（attribution / optimization_plan / eval / release）深度对接
  - 自动化策略编排引擎（自动推进 + 关键点请求确认）；未启用前 UI 用 W1 用户触发兜底
  - 相似度自动归并 / 拆分 / 重归因；未启用前走手动关联 + 禁用纠错按钮

Wave 3 · 资产闭环
  - 资产 Registry 复利中心前台化（回归 / 方法论 / 执行 / 审计资产）
  - 跨 Agent 方法论资产继承复用

后续加固
  - ContextPackage 脱敏（当前面向开发优化人员，不做）
```

每个新写操作按钮（拆分 / 移出 / 标记合并不准 / 强制发布）落地前，确认后端有对应幂等操作 + 重归因 / 回滚策略，否则置灰并提示能力未启用。
