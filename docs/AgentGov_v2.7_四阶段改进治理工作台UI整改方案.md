# AgentGov_v2.7_四阶段改进治理工作台UI整改方案

## 1. 文档目的

本文档用于说明 AgentGov 改进治理工作台的最终版 UI 方案，覆盖以下内容：

- 四阶段主链路定义
- 四个阶段各自的页面结构与交互规则
- 决策卡设计规范
- 面板级“查看详情 / 管理”入口规范
- 处理记录统一规范
- 四张效果图对应的页面含义说明

本方案聚焦解决以下问题：

1. 页面“信息堆叠”感过强；
2. 决策卡中按钮过多、查看类动作与流程动作混杂；
3. 摘要信息与详细信息缺少统一的下钻入口；
4. 各阶段底部“处理记录”不统一，且混杂了全链路信息；
5. 测试阶段对测试用例资产的治理入口不足。

---

## 2. 最终主链路

系统主链路统一为四个阶段：

1. **反馈整理**
2. **归因分析**
3. **优化执行**
4. **测试发布**

对应的用户心智模型分别是：

- 反馈整理：整理反馈、确认问题对象是否成立；
- 归因分析：确认根因是否可信、是否进入方案生成；
- 优化执行：确认方案是否可执行、是否允许实施；
- 测试发布：准备并执行回归测试，验证效果，再决定是否进入发布。

这四个阶段作为顶部主链路固定呈现，形成统一的流程感和闭环感。

---

## 3. 整体页面骨架

四个阶段统一使用同一套工作台骨架：

### 3.1 左侧：改进事项队列

用于展示当前可处理的改进事项集合。

包含：
- 标题：改进事项
- 快捷筛选：全部 / 待确认 / 处理中 / 待归因（或待回归）/ 已完成
- 列表项：
  - 事项标题
  - 所属 Agent / 数据域
  - 反馈数 / Run 数
  - 当前状态标签

### 3.2 右侧：当前改进事项工作台

统一由五层结构构成：

1. 顶部阶段进度条
2. 页面标题与元信息
3. **决策卡（当前阶段唯一主决策中心）**
4. 阶段工作面板（摘要、证据、详情、计划、资产）
5. 底部处理记录（只展示本阶段）

## 4. 决策卡设计规范（最终版）

### 4.1 核心原则

决策卡只负责回答一个问题：

> **当前阶段，用户需要做什么决定？**

它不承担查看详情、展开明细、浏览证据全文等职责。

### 4.2 决策卡允许出现的按钮类型

#### A. 主决策按钮
例如：
- 确认整理结果，进入归因分析
- 确认归因，生成优化方案
- 确认并执行优化
- 执行回归测试

#### B. 改变流程方向的按钮
例如：
- 重新整理
- 重新归因
- 返回反馈整理
- 返回归因分析
- 返回优化执行

#### C. 改变输入事实或结论的按钮
例如：
- 管理来源与归并
- 修改方案

### 4.3 不再放在决策卡中的按钮
以下按钮统一从决策卡下沉至面板：

- 查看证据
- 查看 Trace
- 查看完整归因
- 查看方案
- 查看 Diff
- 查看执行计划
- 查看测试计划
- 查看测试结果
- 查看日志

### 4.4 最终结构原则

> **决策卡瘦身，面板补入口，抽屉承载完整信息。**

## 5. 面板动作规范（最终版新增）

为了在不污染决策卡的前提下保留完整的信息可钻取能力，所有主面板统一采用“标题 + 右侧动作”的模式。

### 5.1 动作类型

#### 1）查看详情
适用于结构化摘要面板，例如：
- 整理结果
- 归因结论
- 优化方案
- 影响范围
- 发布门禁预览

#### 2）管理
适用于会改变对象关系或资产内容的面板，例如：
- 来源反馈
- 测试用例
- 测试数据集
- 回归资产

#### 3）查看完整内容
适用于天然内容较长或结构复杂的面板，例如：
- Diff / 变更预览
- Trace 摘要
- 执行日志
- 测试报告

#### 4）展开全部
适用于列表型面板，例如：
- 证据链
- 覆盖场景
- 处理记录补充条目

### 5.2 位置规范

所有这些入口统一放在：

> **面板标题右上角**

而不是重新放回决策卡。

## 6. 处理记录统一规范（本次重点整改）

这是本版方案的一个关键新增规范。

### 6.1 原则

每个阶段页面中的“处理记录”必须：

1. **统一名称**：统一叫“处理记录”；
2. **统一位置**：统一位于页面底部；
3. **统一样式**：统一采用横向时间线样式；
4. **统一粒度**：统一展示 4～6 个本阶段关键节点；
5. **统一职责**：**默认只展示本阶段处理记录**；
6. **统一入口**：右上角统一提供“查看完整链路”。

### 6.2 为什么只展示本阶段

阶段页底部处理记录的作用是：

> 告诉用户“当前这个阶段，是怎么推进到现在的”。

它不是全链路历史回放，不应该把前后所有阶段的节点都堆在一起。

### 6.3 全链路记录如何处理

全链路信息仍然需要，但不与本阶段 Timeline 混用。

方案为：
- 顶部阶段条负责展示全局进度；
- 底部处理记录只展示本阶段；
- “查看完整链路”进入抽屉或独立页，查看完整生命周期。

### 6.4 四个阶段的处理记录节点建议

#### 反馈整理阶段
- 收到反馈
- 相似归并
- 系统整理
- 证据确认
- 等待人工确认

#### 归因分析阶段
- 进入归因分析
- 收集证据链
- Trace 定位
- 生成归因结论
- 等待人工确认

#### 优化执行阶段
- 进入优化执行
- 生成优化方案
- 风险评估
- 生成执行计划
- 生成回滚方案
- 等待确认执行

#### 测试发布阶段
- 进入测试发布
- 生成测试计划
- 确认测试集
- 待执行回归测试

### 6.5 状态词统一

时间线节点状态统一使用：
- 已完成
- 当前节点
- 进行中
- 失败
- 已回退
- 已跳过

本轮效果图以“已完成 / 当前节点”为主。

## 7. 阶段一：反馈整理

效果图：![](./imgs/反馈整理.png)

### 7.1 阶段目标
将多条反馈整理为一个可治理的问题对象，完成系统整理、问题理解与证据确认，并由人工决定是否进入归因分析。

### 7.2 当前图对应状态
**整理完成，等待确认**

### 7.3 决策卡按钮
- **确认整理结果，进入归因分析**
- 管理来源与归并
- 重新整理

### 7.4 主要面板

#### A. 整理结果（查看详情）
展示摘要：
- 问题模式
- 系统理解
- 影响范围
- 建议下一步

点击“查看详情”后进入抽屉，查看完整整理结果、相似归并依据、人工备注等。

#### B. 证据确认（展开全部）
Checklist 形式展示：
- 来源反馈完整
- 关联 Run 可用
- Trace 可查看
- Agent 归属明确
- 版本影响待后续确认

#### C. 来源反馈（管理来源与归并）
展示来源反馈列表，并支持：
- 查看原文
- 移出当前事项
- 添加 / 调整归并关系（进入管理抽屉）

#### D. 处理记录
只展示反馈整理阶段的本地时间线。

## 8. 阶段二：归因分析

效果图：![](./imgs/归因分析.png)

### 8.1 阶段目标
基于前阶段确认的问题对象与证据，分析根因、影响范围和不确定性，并决定是否进入优化方案生成。

### 8.2 当前图对应状态
**归因完成，等待确认**

### 8.3 决策卡按钮
- **确认归因，生成优化方案**
- 重新归因
- 返回反馈整理

### 8.4 主要面板

#### A. 归因结论（查看详情）
展示摘要：
- 核心根因
- 次要诱因
- 涉及层面
- 建议下一步

点击“查看详情”进入归因结论抽屉，查看完整归因报告。

#### B. 证据链（展开全部）
展示支撑归因的关键证据项。

#### C. 影响范围（查看详情）
展示：
- 业务智能体
- 数据域
- 告警类型
- 风险等级

#### D. Trace 摘要（查看完整 Trace）
展示关键运行链路节点及简要结果。

#### E. 反证与不确定性（查看详情）
展示：
- 反证
- 不确定性
- 验证建议

#### F. 处理记录
只展示归因分析阶段本地时间线。

## 9. 阶段三：优化执行

效果图：![](./imgs/优化执行.png)

### 9.1 阶段目标
根据归因结论生成优化方案，展示变更内容、执行计划与回滚方案，并由人工决定是否执行。

### 9.2 当前图对应状态
**方案生成完成，待确认执行**

### 9.3 决策卡按钮
- **确认并执行优化**
- 修改方案
- 返回归因分析

### 9.4 主要面板

#### A. 优化方案（查看详情）
展示摘要：
- 规则修正
- 上下文补强
- 校验逻辑
- 回归准备

> 注意：面板标题不再叫“优化方案摘要”，而是直接叫“优化方案”。

#### B. Diff / 变更预览（查看完整 Diff）
展示：
- Prompt / 规则 变更量
- Skill / 配置 变更量
- 测试用例 变更量

#### C. 执行计划（查看详情）
展示：
- 修改对象
- 影响范围
- 风险级别
- 审批状态
- 执行步骤

#### D. 回滚方案（查看详情）
展示：
- 当前版本
- 目标版本
- 回滚就绪状态
- 版本快照
- 回滚方式

#### E. 执行记录（查看完整日志）
展示方案生成、审批、风险检查、环境就绪等关键执行前状态。

#### F. 处理记录
只展示优化执行阶段本地时间线。

## 10. 阶段四：测试发布

效果图：![](./imgs/测试发布.png)

### 10.1 阶段目标
围绕本改进事项生成并管理测试资产，执行回归测试，验证改进效果与风险，为后续发布门禁提供依据。

### 10.2 当前图对应状态
**待执行回归测试**

### 10.3 决策卡按钮
- **执行回归测试**
- 返回优化执行

### 10.4 主要面板

#### A. 测试资产与计划（管理测试用例）
这是本阶段最关键的资产面板之一。

展示：
- 默认回归用例
- 新增用例
- 历史基线用例
- 覆盖场景数
- 预计耗时
- 当前测试集

并提供四个直接入口：
- 查看用例
- 修改
- 重新生成
- 纳入测试集

这意味着系统已经支持：
- 查看与本改进事项相关的测试用例
- 修改测试用例
- 重新生成测试用例
- 将测试用例纳入测试数据集

#### B. 回归执行状态（查看执行日志）
当前图是“尚未执行”的空状态，执行后可在同面板查看最新状态与趋势。

#### C. 覆盖场景（查看全部）
展示本次回归测试重点覆盖的关键场景，例如：
- 时间窗口边界
- 跨时区
- DST 切换
- 窗口重叠
- 告警去重
- 乱序事件

#### D. 执行环境 / 基线（查看详情）
展示：
- 执行环境
- 数据版本
- 代码版本
- 对比基线

#### E. 发布门禁预览（查看门禁详情）
展示门禁条件，例如：
- 关键指标不劣于基线
- 严重问题数不增加
- 新增严重问题 = 0
- 通过率 ≥ 95%

同时给出预期门禁评估结果。

#### F. 处理记录
只展示测试发布阶段本地时间线：
- 进入测试发布
- 生成测试计划
- 确认测试集
- 待执行回归测试

## 12. 最终结论

本次最终版方案，完成了四个关键升级：

1. **主链路收敛为四阶段**：反馈整理 → 归因分析 → 优化执行 → 测试发布。
2. **决策卡彻底瘦身**：只保留流程动作，不再承载查看类按钮。
3. **面板补齐入口**：摘要由面板展示，完整信息通过“查看详情 / 管理 / 查看完整”进入抽屉或二级视图。
4. **处理记录彻底统一**：四个阶段统一样式，并且默认只展示本阶段处理记录；全链路通过单独入口查看。

因此，整个系统的阅读路径被统一为：

> **顶部阶段条看全局 → 决策卡做当前判断 → 中部面板看摘要与详情入口 → 底部处理记录看本阶段推进过程。**

这就是最终推荐的 AgentGov 四阶段改进治理工作台 UI 设计方案。
