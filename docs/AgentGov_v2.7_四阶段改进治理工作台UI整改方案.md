# AgentGov_v2.7_四阶段改进治理工作台UI整改方案

> 文档层级：v2.7 改进治理工作台 UI 与流程绝对权威。
> 权威范围：本文与四张效果图 `docs/imgs/反馈整理.png`、`docs/imgs/归因分析.png`、`docs/imgs/优化执行.png`、`docs/imgs/测试发布.png` 共同定义改进治理工作台的用户主链路、页面骨架、决策卡、面板入口、处理记录和验收锚点。
> 覆盖规则：凡已归档的 `docs/archive/design/AgentGov_ASCII_UI_草图方案_v2.7.md`、当前实现基线、旧设计评审、前端旧设计、后端旧设计或测试验收口径与本文冲突，以本文和四张效果图为准。
> 实施边界：本轮先整改 `docs/` 文档；前后端代码整改另行执行。
> 代码整改原则：后续实现默认按四阶段重构或重新设计。当重构收益更大时，不为旧七段阶段、旧反馈工作台四菜单、旧发布顶级入口或旧兼容 facade 增加迁移/兼容层，除非用户单独批准。

## 1. 文档目的

本文档用于说明 AgentGov 改进治理工作台的最终版 UI 方案，覆盖以下内容：

- 四阶段主链路定义
- 四个阶段各自的页面结构与交互规则
- 决策卡设计规范
- 面板级“查看详情 / 管理”入口规范
- 处理记录统一规范
- AI 友好与人类友好的兼容设计
- 测试数据集资产化与沉淀设计
- 四张效果图对应的页面含义说明

本方案聚焦解决以下问题：

1. 页面“信息堆叠”感过强；
2. 决策卡中按钮过多、查看类动作与流程动作混杂；
3. 摘要信息与详细信息缺少统一的下钻入口；
4. 各阶段底部“处理记录”不统一，且混杂了全链路信息；
5. 测试阶段对测试用例资产的治理入口不足。
6. 人类界面与 AI / Playwright / 回归资产生成器消费的上下文契约没有在权威方案中统一；
7. “测试数据集”只作为当前测试集入口出现，尚未明确为可版本化、可审计、可复用的治理资产。

## 2. 最终主链路

系统主链路统一为四个阶段：

1. **反馈整理**
2. **归因分析**
3. **优化执行**
4. **测试发布**

对应的用户心智模型分别是：

- 反馈整理：整理反馈、确认问题对象是否成立；
- 归因分析：确认根因是否可信、是否进入方案生成；
- 优化执行：确认方案是否可执行、是否允许实施；
- 测试发布：准备并执行回归测试，验证效果，再决定是否进入发布。

这四个阶段作为顶部主链路固定呈现，形成统一的流程感和闭环感。

## 3. 整体页面骨架

四个阶段统一使用同一套工作台骨架：

### 3.1 左侧：改进事项队列

用于展示当前可处理的改进事项集合。

包含：
- 标题：改进事项
- 快捷筛选：全部 / 待确认 / 处理中 / 待归因（或待回归）/ 已完成
- 列表项：
  - 事项标题
  - 所属 Agent / 数据域
  - 反馈数 / Run 数
  - 当前状态标签

### 3.2 右侧：当前改进事项工作台

统一由五层结构构成：

1. 顶部阶段进度条
2. 页面标题与元信息
3. **决策卡（当前阶段唯一主决策中心）**
4. 阶段工作面板（摘要、证据、详情、计划、资产）
5. 底部处理记录（只展示本阶段）

## 4. 决策卡设计规范（最终版）

### 4.1 核心原则

决策卡只负责回答一个问题：

> **当前阶段，用户需要做什么决定？**

它不承担查看详情、展开明细、浏览证据全文等职责。

### 4.2 决策卡允许出现的按钮类型

#### A. 主决策按钮
例如：
- 确认整理结果，进入归因分析
- 确认归因，生成优化方案
- 确认并执行优化
- 执行回归测试

#### B. 改变流程方向的按钮
例如：
- 重新整理
- 重新归因
- 返回反馈整理
- 返回归因分析
- 返回优化执行

#### C. 改变输入事实或结论的按钮
例如：
- 管理来源与归并
- 修改方案

### 4.3 不再放在决策卡中的按钮
以下按钮统一从决策卡下沉至面板：

- 查看证据
- 查看 Trace
- 查看完整归因
- 查看方案
- 查看 Diff
- 查看执行计划
- 查看测试计划
- 查看测试结果
- 查看日志

### 4.4 最终结构原则

> **决策卡瘦身，面板补入口，抽屉承载完整信息。**

## 5. 面板动作规范（最终版新增）

为了在不污染决策卡的前提下保留完整的信息可钻取能力，所有主面板统一采用“标题 + 右侧动作”的模式。

### 5.1 动作类型（v2.7 W3 修订：4 类 → 2 类）

> **W3 修订决策（2.8.5）**：原 4 类标签（查看详情 / 管理 / 查看完整内容 / 展开全部）收敛为 **2 类**。依据：「查看完整内容」「展开全部」本质都是只读下钻，与「查看详情」无用户可感知差异，统一为「查看详情」；只有会改变对象关系或资产内容的面板才用「管理」。原 4 类的脚手架实现存在 14 个未接线的死按钮 + 内容与卡片不对应，W3 一并整改为：只读卡统一进 `StageDetailDrawer`（内容与本卡一一对应），可编辑卡进对应管理抽屉。

#### 1）查看详情（所有只读下钻）
点击在面板标题右上角，统一打开 `StageDetailDrawer`，抽屉内容与本面板语义一一对应。适用：整理结果、证据确认、归因结论、影响范围、证据链、Trace 摘要、反证与不确定性、优化方案、Diff / 变更预览、执行计划、回滚方案、执行记录、回归执行状态、覆盖场景、执行环境 / 基线、发布门禁预览。

#### 2）管理（可编辑）
适用于会改变对象关系或资产内容的面板，打开对应管理抽屉。适用：来源反馈、测试用例 / 测试数据集 / 回归资产。

### 5.2 位置规范

所有这些入口统一放在：

> **面板标题右上角**

而不是重新放回决策卡。

## 6. 处理记录统一规范（本次重点整改）

这是本版方案的一个关键新增规范。

### 6.1 原则

每个阶段页面中的“处理记录”必须：

1. **统一名称**：统一叫“处理记录”；
2. **统一位置**：统一位于页面底部；
3. **统一样式**：统一采用横向时间线样式；
4. **统一粒度**：统一展示 4～6 个本阶段关键节点；
5. **统一职责**：**默认只展示本阶段处理记录**；
6. **统一入口**：右上角统一提供“查看完整链路”。

### 6.2 为什么只展示本阶段

阶段页底部处理记录的作用是：

> 告诉用户“当前这个阶段，是怎么推进到现在的”。

它不是全链路历史回放，不应该把前后所有阶段的节点都堆在一起。

### 6.3 全链路记录如何处理

全链路信息仍然需要，但不与本阶段 Timeline 混用。

方案为：
- 顶部阶段条负责展示全局进度；
- 底部处理记录只展示本阶段；
- “查看完整链路”进入抽屉或独立页，查看完整生命周期。

### 6.4 四个阶段的处理记录节点建议

#### 反馈整理阶段
- 收到反馈
- 相似归并
- 系统整理
- 证据确认
- 等待人工确认

#### 归因分析阶段
- 进入归因分析
- 收集证据链
- Trace 定位
- 生成归因结论
- 等待人工确认

#### 优化执行阶段
- 进入优化执行
- 生成优化方案
- 风险评估
- 生成执行计划
- 生成回滚方案
- 等待确认执行

#### 测试发布阶段
- 进入测试发布
- 生成测试计划
- 确认测试集
- 待执行回归测试

### 6.5 状态词统一

时间线节点状态统一使用：
- 已完成
- 当前节点
- 进行中
- 失败
- 已回退
- 已跳过

本轮效果图以“已完成 / 当前节点”为主。

## 12. AI 友好与人类友好兼容设计

改进治理工作台必须同时服务两类消费者：

1. **人类用户**：开发者、治理负责人、测试负责人需要低负担判断“当前阶段要做什么决定”。
2. **AI / 自动化工具**：Codex、Claude Code、Playwright、回归资产生成器和治理 Agent 需要稳定、完整、可复现的结构化上下文。

两类消费者使用同一事实源，但不能混用同一展示方式。

### 12.1 人类友好层

人类友好层只展示当前阶段决策所需的信息：

- 顶部阶段条展示全局进度；
- 决策卡只保留当前主决策、方向改变和事实修正动作；
- 中部面板展示摘要、证据、计划、资产和详情入口；
- 底部处理记录只展示本阶段推进过程；
- 完整链路、Trace、Diff、日志、完整测试报告进入抽屉或二级视图。

人类界面不得把底层 job id、raw JSON、完整 Trace、自动化日志、模型原始输出、Playwright selector 和数据库字段直接堆在主界面。

### 12.2 AI 友好层

AI 友好层通过“获取上下文”提供结构化导出，不通过扩大主界面信息密度实现。

“获取上下文”入口出现在：

- 页面标题区域；
- 决策卡右侧辅助区；
- 关键面板标题右上角；
- 回归失败、发布阻塞、证据不足等需要交给 AI 继续分析的位置。

上下文类型统一为四类：

| 类型 | 消费者 | 内容形态 | 用途 |
| --- | --- | --- | --- |
| 问题摘要 | 人类、Issue、PR、群聊 | Markdown | 让人快速理解当前事项和阻塞点 |
| AI 分析上下文 | Codex、Claude Code、ChatGPT、治理 Agent | Markdown + 稳定字段 | 继续分析归因、方案、风险、测试缺口 |
| Playwright 复现信息 | 自动化测试、AI 生成测试 | 路径 + 状态 + 断言 + selector | 生成或修复 UI 验收脚本 |
| 完整 JSON 上下文 | Agent、回归资产生成器、调试工具 | 结构化 JSON | 跨工具稳定消费、审计和复现 |

### 12.3 ContextPackage 最小字段

完整 JSON 上下文必须至少包含：

```text
context_version
source.page
source.url
source.timestamp
improvement.improvement_id
improvement.title
improvement.stage
improvement.status
improvement.agent_id
current_decision.primary_action
current_decision.allowed_actions
source_feedback[]
normalized_feedback
attribution
optimization_plan
execution_record
test_dataset_refs[]
regression_run_refs[]
release_gate_preview
evidence_refs[]
trace_refs[]
asset_refs[]
missing_reasons[]
```

字段缺失时输出 `missing_reasons[]`，不得用空对象冒充完整证据。

### 12.4 同源约束

AI 友好层与人类友好层必须同源：

- 页面显示的阶段、主动作、面板摘要和 ContextPackage 中的阶段、主动作、摘要一致；
- API、DTO、ContextPackage、Playwright selector 和 UI 文案使用同一术语表；
- AI 导出可以更完整，但不能引入 UI 中不存在的事实结论；
- AI 导出可以暴露稳定 ID，但人类主界面默认显示标题、状态和摘要；
- Playwright 断言必须包含负向断言，例如“查看类动作不出现在决策卡中”“完整 Trace 不出现在主界面”。

### 12.5 边界

“AI 友好”不等于把页面做成调试控制台，也不等于让 AI 直接跳过人工确认。

默认边界为：

- AI 可以消费上下文、生成分析、提出方案、生成测试候选和复现脚本；
- 后端规则和状态机负责校验合法转移、权限、发布门禁和资产写入；
- 人类保留归因确认、方案执行、回归测试和发布相关关键决策；
- 高风险动作不得只因 AI 输出存在而自动推进。

## 13. 测试数据集资产化与沉淀设计

测试发布阶段中的“测试数据集”不是临时勾选列表，而是可版本化、可审计、可复用的治理资产。

### 13.1 核心对象

| 对象 | 定位 | 是否资产 |
| --- | --- | --- |
| 测试用例 | 单条可执行评估/回归用例 | 是，执行资产 |
| 测试数据集 | 一组测试用例 + 选择策略 + 适用范围 + 基线 | 是，执行资产和数据资产的组合 |
| 回归运行 | 某个版本/变更集上执行测试数据集后的结果 | 是，数据资产和审计资产 |
| 回归资产 | 被确认可长期防退化的用例、数据集或场景包 | 是，长期执行资产 |

### 13.2 TestDataset 最小字段

测试数据集至少包含：

```text
test_dataset_id
title
description
agent_id
improvement_id
source_feedback_ids[]
source_case_ids[]
scenario_tags[]
eval_case_ids[]
selection_policy
baseline_agent_version_id
candidate_agent_version_id
status
promotion_status
quality_gate_policy
provenance
created_by
created_at
updated_at
archived_at
```

### 13.3 生命周期

测试数据集生命周期统一为：

```text
draft -> candidate -> approved -> active -> archived
```

- `draft`：系统或人工刚生成，尚未进入门禁；
- `candidate`：已纳入当前改进事项测试集，等待回归运行验证；
- `approved`：通过人工确认，可作为当前事项发布门禁输入；
- `active`：进入长期回归资产库，可被后续改进事项复用；
- `archived`：保留审计，不再参与新回归选择。

非法回退和跨 Agent 误归属必须被后端拒绝。

### 13.4 测试发布阶段的用户操作

测试发布阶段只暴露与当前决策相关的动作：

- 查看用例：查看当前数据集中的用例摘要、来源和覆盖场景；
- 修改：调整用例内容、期望行为、检查项和标签；
- 重新生成：基于当前反馈、归因、优化方案和历史基线生成候选用例；
- 纳入测试集：把候选用例加入当前 `TestDataset`；
- 固化为回归资产：回归通过并人工确认后，将数据集或关键用例晋级为长期资产。

这些动作放在“测试资产与计划”面板或其管理抽屉中，不放入决策卡。决策卡只保留“执行回归测试”和必要的返回动作。

### 13.5 资产关系

测试数据集必须能回答以下问题：

- 它属于哪个业务 Agent？
- 它由哪些反馈、归因结论和优化方案产生？
- 它覆盖哪些场景、能力域和风险？
- 它基于哪个 Agent 版本或候选变更集执行？
- 它最近一次回归运行结果如何？
- 它是否已经晋级为长期回归资产？
- 后续哪个改进事项复用了它？

因此，`TestDataset` 必须进入资产 Registry，并与以下对象建立 provenance：

```text
ImprovementItem
Feedback
Attribution
OptimizationPlan
ExecutionRecord / change set
RegressionRun
Release
RegressionAsset
ScenarioPack
```

### 13.6 与旧“回归资产”入口的关系

旧“回归资产”独立主路径不再定义改进治理工作台主流程。

新的关系是：

```text
测试发布阶段
  -> 生成 / 管理 TestDataset
  -> 执行 RegressionRun
  -> 通过发布门禁
  -> 晋级为 RegressionAsset
  -> 进入 AssetRegistry
```

也就是说，测试数据集先服务当前改进事项的测试发布决策，再在验证后沉淀为长期资产。

### 13.7 验收断言

后续代码整改必须验证：

- 当前测试集不是匿名数组，必须有 `test_dataset_id`；
- 测试数据集必须有 `agent_id` 和 `improvement_id` 归属；
- 测试数据集必须记录来源反馈、用例列表、基线版本和候选版本；
- 回归运行必须引用具体 `test_dataset_id`；
- 通过人工确认后，测试数据集或其关键用例可以进入资产 Registry；
- archived 数据集仍可审计，但不参与新回归选择；
- 跨 Agent 数据集误用必须被拒绝。

## 14. 后续代码整改原则（本轮先写入 docs）

代码整改稍后执行，但实现方案必须先遵守以下原则：

1. **用户主链路只保留四阶段**：反馈整理、归因分析、优化执行、测试发布是改进治理工作台唯一用户可见阶段。后端更细状态若仍存在，只能作为内部实现细节，不能反向决定页面结构。
2. **决策卡只承载当前主决策**：查看 Trace、查看 Diff、查看测试计划、查看日志等浏览动作必须下沉到面板入口或抽屉，不回到决策卡。
3. **测试发布阶段收口测试资产与门禁**：测试用例、测试数据集、回归执行、覆盖场景、执行环境、发布门禁预览都归入测试发布阶段，不再用旧“回归资产”独立主路径定义改进治理主流程。
4. **旧设计不作为兼容目标**：旧七段链路、旧 `反馈信息 | 优化批次 | 回归资产 | 版本管理` 菜单、旧发布顶级入口、旧 proposal/job 页面和旧 facade 只可作为迁移来源或历史证据；若与四阶段方案冲突，默认删除、合并或重构，不默认保留兼容。
5. **接口与测试围绕四阶段验收**：后续 API、DTO、前端状态、Playwright 选择器和主流程测试应验证四阶段语义、面板入口和负向断言，例如“查看类按钮不出现在决策卡中”“处理记录只展示本阶段”“完整链路必须通过单独入口查看”。
6. **效果图是 UI parity 基准**：布局、命名、入口位置和信息归属以四张效果图为截图级验收锚点；旧草图、旧评审截图或当前运行态截图不能覆盖它们。
7. **AI / 人类双友好同源**：人类主界面保持低噪声，AI / Playwright / 回归资产生成器通过 ContextPackage 获取完整结构化上下文；两者必须来自同一事实源。
8. **测试数据集资产化**：`TestDataset` 必须作为有 ID、有归属、有生命周期、有 provenance 的治理资产建模，不得只做成当前页面的临时勾选状态。

## 15. 最终结论

本次最终版方案，完成了六个关键升级：

1. **主链路收敛为四阶段**：反馈整理 → 归因分析 → 优化执行 → 测试发布。
2. **决策卡彻底瘦身**：只保留流程动作，不再承载查看类按钮。
3. **面板补齐入口**：摘要由面板展示，完整信息通过“查看详情 / 管理 / 查看完整”进入抽屉或二级视图。
4. **处理记录彻底统一**：四个阶段统一样式，并且默认只展示本阶段处理记录；全链路通过单独入口查看。
5. **AI 友好与人类友好同源兼容**：主界面服务人类决策，ContextPackage 服务 AI、Playwright 和自动化工具消费。
6. **测试数据集资产化**：测试数据集作为可版本化、可审计、可复用资产进入测试发布与资产 Registry 链路。

因此，整个系统的阅读路径被统一为：

> **顶部阶段条看全局 → 决策卡做当前判断 → 中部面板看摘要与详情入口 → 底部处理记录看本阶段推进过程。**

这就是最终推荐的 AgentGov 四阶段改进治理工作台 UI 设计方案。
