## 1. 文档定位

本文是 v2.7 当前实现的全局界面系统性审视报告，不是最终 UI 规范，也不是实施补丁说明。它用于回答三个问题：

1. Trace 按钮是否适合改成右侧抽屉。
2. 抽屉宽度是否应按内容复杂度分档。
3. 当前每个主要界面是否支撑 AgentGov 的治理闭环，而不只是局部可点击。

结论先行：Trace 改成右侧抽屉是更一致的方向，但不能作为孤立改动实施。当前界面还存在抽屉、modal、详情面板、设置弹窗、诊断视图并存的问题；更本质的整改是建立“侧边任务面板 + 中心全局配置弹窗 + 主流程工作台”的界面系统。

## 2. 治理对象预检

| 维度 | 结论 |
| --- | --- |
| 被治理对象 | 整个 AgentGov 平台的用户界面系统，核心对象是多业务 Agent、运行 Trace、反馈、改进事项、发布门禁和资产 Registry。 |
| 治理执行者 | 前端信息架构、后端状态投影、治理 Agent 输出、人工确认共同构成治理执行者。 |
| 资产类型 | 数据资产：run、trace、feedback、evidence、eval result、release event；方法论资产：归因法、优化 SOP、发布策略；执行资产：prompt、skill、profile、playbook、eval case；审计资产：状态、diff、版本和发布记录。 |
| 生命周期 | 业务 Agent 有 draft / active / evaluating / deprecated / archived；改进事项有反馈收集、系统整理、归因分析、优化方案、执行优化、回归测试、发布。 |
| 反馈归属 | 反馈必须归属到业务 Agent、agent version、场景、run / session，并进入改进事项链路。 |
| 当前实现边界 | v2.7 主导航、改进事项、创建反馈抽屉、运行设置抽屉、发布门禁、资产 Registry 已存在；Trace 详情仍是中心 modal；旧反馈优化工作台仍作为诊断入口保留。 |
| 目标能力边界 | UI 应帮助用户理解和推进闭环：运行 -> Trace -> 反馈 -> 改进 -> 归因 -> 优化 -> 执行 -> 回归 -> 发布 -> 资产，而不是把内部对象堆成多个孤立页面。 |

闭环链路：

```text
业务 Agent -> 运行 -> Trace -> 反馈 -> 改进事项 -> 归因 -> 优化方案 -> 执行记录 -> 回归 -> 发布 -> 资产 Registry
```

风险自检：

- 不能把当前 main-agent 样板误当长期产品边界；所有界面都应能按业务 Agent 归属理解。
- 不能把 Trace 当纯调试日志；Trace 是反馈可信度和归因证据链的入口。
- 不能把资产 Registry 当数据清单；它是方法论、执行、回归和审计资产复用中心。
- 不能用“界面能打开”代替“用户能理解下一步和系统状态”。

## 3. 《系统之美》可借鉴的设计原则

这里不把书中观点当作界面装饰，而是转化为 AgentGov 的产品审视框架：

| 系统思维要点 | 对 AgentGov UI 的要求 |
| --- | --- |
| 反馈环 | 用户从运行结果提出反馈后，界面必须显示反馈如何进入归因、优化、回归、发布和资产沉淀。 |
| 延迟 | 自动化策略、治理 Agent 生成、回归运行和发布门禁都有延迟，界面要显示“正在发生什么”和“下一次可判断点”。 |
| 系统边界 | Langfuse、外部工单、Git 服务、业务 Agent workspace 各有边界，界面不能让用户误以为 AgentGov 全权拥有所有对象。 |
| 信息流 | Trace、ContextPackage、反馈、证据、资产来源是信息流，应该能从同一个界面模型追溯。 |
| 杠杆点 | 用户真正高杠杆动作不是到处点“生成”，而是确认反馈理解、确认归因、确认方案、运行回归、决定发布。 |
| 心智模型 | 页面应围绕“治理闭环”组织，而不是围绕数据库表、内部 job 或旧批次引擎组织。 |

这支持一个关键判断：Trace 不应只是中心弹窗里的运行细节，它应成为侧边证据面板，和创建反馈、获取上下文、查看证据等动作形成连续信息流。

## 5. 总体判断

### 5.1 已经成立的方向

- 顶层导航已收敛为 Playground / 改进 / 发布，符合 v2.7 主线。
- “创建反馈”抽屉证明了侧边任务面板适合不中断主上下文的补充动作。
- “运行设置”抽屉证明复杂配置可放入宽抽屉，不必回到旧三栏常驻布局；会话历史不属于配置。
- 改进详情的七段阶段条已表达闭环主线。
- 发布页已有“能不能发”的门禁判断，符合治理杠杆点。
- 资产 Registry 已进入主流程高级入口，并显示来源改进和继承关系。

### 5.2 当前最大系统性偏差

| 优先级 | 偏差 | 影响 |
| --- | --- | --- |
| P0 | Trace 详情仍是中心 modal，而反馈和配置是右侧抽屉 | 用户在“审查运行证据 -> 创建反馈 -> 获取上下文”的连续任务中被迫切换空间模型。 |
| P0 | 抽屉、modal、详情面板没有统一语义矩阵 | 新功能容易继续随手选择展示容器，造成交互割裂。 |
| P0 | 改进、发布、资产页默认态和空白区域没有充分承载下一步 | 用户难以感知系统正在推动闭环。 |
| P1 | 资产 Registry 创建表单常驻，弱化浏览、筛选、追溯来源 | 资产复利中心看起来像录入表单，而不是治理资产库。 |
| P1 | 旧反馈优化诊断视图虽降级但仍像主功能 | 用户可能绕回旧批次引擎，削弱 v2.7 主线。 |
| P1 | 设置 modal 同时包含 Agent 管理、自动化策略、开发者配置和危险操作 | 全局配置合理，但结构需要更清晰的区域和确认机制。 |
| P2 | 中英文和技术词混排缺少明确规则 | 例如 Runtime online、model、Developer / Debug、Langfuse 与中文动作混排，工程人员可接受，但治理平台口径需要更稳定。 |

## 6. 逐界面审视

### 6.1 Playground

当前事实：

- 主区只保留对话、回复动作和输入框。
- 配置、反馈都进入侧边抽屉。
- 回复动作包括创建反馈、查看 Trace、获取上下文、打开 Langfuse、重新运行。

问题：

- `查看 Trace` 和 `打开 Langfuse` 并列，但二者层级不同：前者是内部运行证据，后者是外部观测系统入口。
- “获取上下文”当前是复制动作，缺少用户可见的成功反馈和上下文内容预览。
- Playground 中的侧边动作没有统一 DrawerShell，创建反馈和运行设置抽屉各自实现。

建议：

- 将 `查看 Trace` 改为右侧 Trace 抽屉，抽屉内保留“打开 Langfuse”外链。
- “获取上下文”应至少提供 toast 或侧边只读预览，避免无声复制。
- 建立消息动作分层：证据查看、反馈创建、上下文导出、外部跳转、重新运行。

### 6.2 Trace 详情

当前事实：

- 当前 `查看 Trace` 打开 `ResponseDetailModal`。
- modal 中已有事件统计、AgentActivitySummary、搜索、事件时间线和 JSON 明细。
- 这是一个高信息密度内容，宽度约 920px。

问题：

- 中心 modal 遮蔽主对话，用户无法同时对照回答文本和 Trace 事件。
- Trace 是反馈归因证据链入口，语义上应靠近“创建反馈”和“获取上下文”，而不是像一个独立阻断式弹窗。
- 当前 Trace 与 Langfuse 的边界不清：内部事件详情和外部观测链接并列在消息按钮上。

建议：

- 新增 `TraceDrawer`，复用当前 ResponseDetailModal 的内容能力。
- 默认 `medium` 宽度；当事件数多、JSON 明细长、工具调用多时打开即使用 `wide`。
- 抽屉顶部显示 run、session、agent version、事件数、工具调用数、Langfuse 外链。
- 搜索和时间线保留在抽屉内，JSON 使用深色 debug 区。
- 原中心 modal 仅保留给真正阻断式确认，不再承载 Trace。

### 6.3 创建反馈抽屉

当前事实：

- 右侧 480px 抽屉，展示两阶段创建反馈。
- 自动带入当前 Run / Trace、业务 Agent、Agent 版本、场景上下文。

问题：

- 它证明了右侧抽屉适合不中断上下文，但实现是独立 CSS 和组件，没有通用容器。
- 作为 dialog 的可访问性语义还不完整，缺少统一焦点、Esc 关闭、aria 规则。
- 如果用户先打开 Trace 再创建反馈，两个侧边任务之间缺少统一切换模型。

建议：

- 迁移到统一 `DrawerShell size="narrow"`。
- 把自动带入项抽成标准 context chips，用于 Trace 和反馈一致展示。
- 后续支持从 Trace 抽屉直接发起创建反馈，并携带当前选中事件作为证据。

### 6.4 会话抽屉与运行设置抽屉

当前事实：

- 会话历史、Subagent、Skills、运行参数和 Runtime 状态一度被混放进同一个“运行配置”抽屉。
- 这是本报告早期建议的边界错误：Sessions 是 Playground 运行对象管理，不是运行配置。

问题：

- “配置”入口承载会话历史、运行设置和 Runtime 调试，会变成杂物抽屉。
- 抽屉内部如果复用旧 `Sidebar` / `Inspector`，会把旧三栏布局伪装成新抽屉。
- 测试如果只校验抽屉存在和宽度，无法发现对象边界错误。

建议：

- 拆成 `DrawerShell size="medium"` 的「会话」抽屉和 `DrawerShell size="wide"` 的「运行设置」抽屉。
- 「会话」只承载会话历史、新会话、刷新、删除和切换。
- 「运行设置」只承载 Subagent、Skills、Skills Mode、Max Turns、Alert/Case、Allowed/Disallowed Tools。
- Runtime / Config / Events 只能作为「高级调试信息」默认折叠，不作为运行设置主内容。

### 6.5 改进事项

当前事实：

- 左侧列表 + 右侧详情。
- 详情态有阶段条、下一步、系统理解、系统归因、回归保障、来源引用和高级区域。

问题：

- 未选中态空白过大，列表已有数据时仍让用户额外选择。
- 详情态把完整链路折叠在 `查看完整链路` 下，系统反馈环不够显性。
- “生成原因（初步）”“采购为回归资产”“开始系统整理”等动作之间仍带有内部推进感，用户任务语言可以更稳定。

建议：

- 当列表有数据且没有用户选择时默认选中第一项，或把右侧空态改为任务型入口。
- 始终显示一条压缩闭环 spine：反馈 -> 理解 -> 归因 -> 方案 -> 执行 -> 回归 -> 发布 -> 资产。
- 右侧主动作只保留一个最强下一步，其他动作进入次级区。
- 来源反馈、Trace、ContextPackage 和资产 provenance 应能互相跳转。

### 6.6 发布门禁

当前事实：

- 发布页显示“能不能发”、门禁状态、运行回归、查看变更、强制发布、候选变更和发布记录。

问题：

- 页面主体大面积空白，候选变更只有一行，diff / 回归 / blocker 没有充分展开。
- `强制发布` 是高风险动作，应比普通次级按钮更显式地进入确认流程。
- 发布页没有直接显示该候选变更对应哪些改进事项、回归资产和 Agent version 证据。

建议：

- 采用门禁台布局：左侧候选变更列表，右侧门禁详情、回归结果、diff 摘要、发布记录。
- “强制发布”必须进入确认对话，说明绕过的门禁和审计记录。
- 每个候选变更显示来源改进事项、关联执行记录、回归结果和目标发布版本。

### 6.7 资产 Registry

当前事实：

- 页面展示资产创建表单、资产列表、来源改进、继承来源和跨 Agent 继承操作。

问题：

- 创建表单常驻在首屏顶部，使资产中心看起来像录入页，而不是复利资产库。
- 缺少资产类型筛选、来源改进筛选、Agent scope 解释和 provenance 展开。
- 资产列表的信息层级较弱，方法论、回归、执行、审计资产没有形成明显分类视图。

建议：

- 默认以资产浏览和追溯为主，创建资产移入抽屉或折叠面板。
- 增加类型筛选、来源改进筛选、继承链展示。
- 资产卡片突出：类型、归属 Agent、来源改进、被继承情况、最后更新时间。

### 6.8 设置

当前事实：

- 设置使用中心 modal，宽约 680px。
- 包含业务 Agent 管理、自动化策略、资产 Registry 入口、Developer / Debug。

判断：

- 设置作为全局配置使用中心 modal 是合理的。它不是“侧边证据查看”任务，不必强行改为抽屉。

问题：

- 多类配置堆在一个滚动 modal 内，业务 Agent 生命周期和删除操作风险较高。
- Developer / Debug 与普通业务设置同屏，可能让业务用户接触过多运行时细节。
- “打开资产 Registry”和“反馈优化工作台（旧 / 诊断）”是导航动作，放在设置中合理，但需要更明确的高级入口语义。

建议：

- 设置保留 modal，但内部改为 tabs 或分组导航：业务 Agent、自动化、资产入口、开发者。
- 删除业务 Agent 需要影响面预览和明确确认。
- Developer / Debug 默认折叠或标注高级。

### 6.9 旧反馈优化诊断视图

当前事实：

- 从设置进入，页面顶部有 legacy banner。
- 左侧仍有反馈信息、优化批次、回归资产、版本管理等旧工作台导航。

问题：

- banner 已说明旧入口身份，但整体视觉仍像一等工作台。
- 底部链路仍出现旧批次引擎概念，可能让用户困惑主闭环到底在“改进”还是“反馈优化”。

建议：

- 保留诊断入口，但进一步降级为“开发者诊断视图”。
- 在页面首屏提供“返回改进事项主流程”的强入口。
- 旧引擎术语在该页面保留真实实现名，但必须持续标注与 v2.7 主术语的关系。

## 7. 抽屉与弹窗语义矩阵

| 容器 | 使用场景 | 宽度策略 | 示例 |
| --- | --- | --- | --- |
| `DrawerShell size="narrow"` | 轻量输入、确认、创建动作 | `min(480px, 96vw)` | 创建反馈、轻量资产创建 |
| `DrawerShell size="medium"` | 证据摘要、上下文包、Trace 摘要 | `min(720px, 96vw)` | Trace 摘要、ContextPackage 预览 |
| `DrawerShell size="wide"` | 高密度设置、事件时间线、JSON / Diff | `min(960px, 96vw)` | 运行设置、Trace 明细、diff 预览 |
| Center modal | 全局设置、危险确认、阻断式决策 | 固定中宽或内容宽度 | 设置、强制发布确认、删除 Agent 确认 |
| 主工作台详情面板 | 长时间停留、对象详情、主流程推进 | 页面布局决定 | 改进详情、发布门禁台、资产列表 |

宽度规则：

- 根据内容类型和打开时的复杂度选择宽度档位。
- 打开后宽度保持稳定，不随异步加载结果跳动。
- Trace 可以从 `medium` 开始；事件数、工具调用、JSON 明细超过阈值时打开即使用 `wide`。
- 移动端不在本轮范围内；桌面端先保证信息密度和空间一致。

## 8. 建议整改优先级

### P0：统一证据侧边任务模型

1. 新建通用 `DrawerShell`，迁移创建反馈、会话和运行设置抽屉。
2. 把 `ResponseDetailModal` 改为 `TraceDrawer`。
3. Trace 抽屉内整合内部事件详情、run/session/agent version、Langfuse 外链。
4. Playwright 验证 `查看 Trace` 打开右侧抽屉，而不是中心 modal。

### P2：治理高级入口和设置结构

1. 设置 modal 内部改为分组导航或 tabs。
2. 旧反馈优化诊断视图进一步降级，并提供返回改进主流程入口。

4. 统一中英文术语显示规则：业务动作中文优先，系统名和第三方产品名保留英文。

## 10. 结论

Trace 按钮打开的界面更适合改成侧边抽屉，而且应和“创建反馈”形成同一套侧边任务模型。抽屉宽度也应按内容类型分档，而不是全部固定或实时抖动。

但当前更重要的问题不是单独把 Trace modal 换成 drawer，而是建立完整的界面容器语义：

```text
主工作台承载对象和闭环推进
右侧抽屉承载不中断上下文的证据、输入和配置任务
中心 modal 只承载全局配置或阻断式确认
```

按这个模型整改，AgentGov 的界面才会更符合“系统之美”里强调的反馈环、信息流、边界和杠杆点，也更符合 v2.7“把复杂留在系统内，把简单留给用户”的目标。
