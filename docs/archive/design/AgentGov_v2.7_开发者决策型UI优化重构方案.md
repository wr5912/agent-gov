# AgentGov v2.7 开发者决策型 UI 优化重构方案

> 文档定位：v2.7 设计评审补充方案。本文不替代 `docs/AgentGov_ASCII_UI_草图方案_v2.7.md`，而是在当前实现和近期 UI 评审基础上，把“改进事项应服务开发者决策”具体化到页面结构、下钻入口和验收规则。
>
> 读者：技术主管、前端实现者、产品/交互评审者、负责 Playwright UI 验收的工程师。
>
> 适用范围：改进事项主流程、反馈创建、反馈归并、运行证据、上下文导出和高级治理能力。本文只定义 UI/交互与信息架构目标，不声明后端能力已经完成。

## 1. 可复用核心原则

1. 先展示用户当前待决策问题，再展示系统对象。
2. 改进事项是场景级问题、小类问题或可治理的问题模式；单条反馈只是发生实例和来源证据。
3. 每个阶段只保留一个主决策和一个主动作。
4. 每个决策必须说明判断依据、确认后会发生什么、不会发生什么。
5. Provenance 一等化：开发者必须能知道改进事项如何从反馈创建、归并或拆分而来。
6. 细节入口按用户疑问分组：查看依据、查看全部反馈、添加反馈、调整归并关系、查看处理链路、获取上下文、高级设置。
7. AI/系统生成内容必须标明来源和确认状态：用户原话、系统整理、治理 Agent 生成、后端权威字段、草稿、已确认。
8. 处理链路、Trace、Raw JSON、自动化任务和高级配置默认收起。
9. UI 验收不能只断言元素存在，还要断言首屏是否有明确待决策问题、是否只有一个主动作、是否有依据入口和后果说明。

### 1.1 治理对象边界

| 对象 | 设计定义 | UI 应如何表达 |
| --- | --- | --- |
| 反馈 | 某一次运行、对话、Trace 或外部入口暴露出的具体问题实例 | 展示为来源证据，可查看原文、Run、Session、版本和场景 |
| 改进事项 | 多条反馈共同指向的场景级问题、小类问题或可治理的问题模式 | 展示共同指向、影响范围、归并依据、分歧点和当前待决策 |
| 归因结果 | 对该问题模式的原因判断，不只解释某一次反馈 | 展示责任边界、证据摘要和确认状态 |
| 优化方案 | 针对该问题模式的一次修复方向 | 展示预期改变、影响范围和回归要求 |
| 回归资产 | 防止该问题模式复发的验证资产 | 展示来源改进事项和适用场景 |

判定一个反馈是否应该加入已有改进事项时，不比较“这次回答是不是完全一样”，而比较它是否属于同一可治理问题模式：是否共享归因、是否能复用同一优化方案、是否应进入同一组回归资产。

### 1.2 入口语义矩阵

| 入口 | 推荐出现位置 | 唯一职责 | 不应承载 |
| --- | --- | --- | --- |
| 查看依据 | 当前待决策卡片 | 服务当前待决策问题，展示最小证据摘要 | 不展示完整反馈表，不做反馈集合管理 |
| 查看全部反馈 | “这个事项怎么来的”来源反馈摘要区 | 查看当前改进事项的来源反馈集合、原文、Run、Trace、版本和状态 | 不替代当前决策依据，不承担上下文导出 |
| 添加反馈 | “这个事项怎么来的”来源反馈摘要区 | 把新的反馈实例加入当前改进事项所代表的问题模式；默认先选择候选反馈，再审阅详情和确认添加 | 不静默推进阶段，不自动重写归因或方案 |
| 调整归并关系 | 当前待决策卡片或来源反馈摘要区 | 处理改进事项之间的合并、移出、拆分和误归并纠错 | 不作为普通详情入口 |
| 查看处理链路 | 后续处理链路区 | 查看反馈、理解、归因、方案、执行、回归、发布的阶段进展 | 不把所有阶段内容平铺到首屏 |
| 获取上下文 | 辅助信息区 | 导出给 AI 或外部协作用的结构化上下文 | 不作为普通详情、证据查看或反馈管理入口 |
| 高级设置 | 辅助信息区或高级区 | 自动化策略、闭环对象关联、危险操作等低频治理能力 | 不放当前阶段主决策 |

同一页面不得同时出现多个语义等价的来源反馈入口。若需要浏览来源反馈集合，统一使用“查看全部反馈”；若只是支撑当前判断，使用“查看依据”。
在改进事项详情页，来源反馈集合的查看入口只出现一次：放在“这个事项怎么来的”来源反馈摘要区；当前决策卡片只放 `查看依据`，辅助信息区不再重复放来源反馈查看入口。

## 2. 页面一：改进事项详情页

### 当前问题

当前详情页把系统理解、归因、优化方案、执行记录、回归保障、来源反馈、来源引用、Trace、高级自动化等对象平铺在同一纵向流中。开发者点击改进事项后，看得到很多系统对象，但不知道当前到底需要自己判断什么。

另一个关键问题是，页面容易把改进事项表现成“某一条反馈的详情”。这会把治理对象收窄成具体问题实例，导致事项过碎、重复归因、重复方案和重复回归。改进事项应表达一类可治理问题，来源反馈可以是 1 条，也可以是 N 条。

根因不是视觉样式，而是页面缺少“当前待决策”这一中间层。

### 设计目标

- 首屏回答：当前需要开发者确认什么、为什么确认、依据是什么、确认后会发生什么。
- 首屏必须用 `N 条反馈 / M 个 Run / K 个版本` 这类通用表达，兼容一个改进事项关联多条反馈。
- 首屏展示“共同指向”和“分歧点”，帮助开发者判断归并是否准确。
- 只展示当前阶段相关内容；非当前阶段内容进入处理链路或历史详情。
- 保留一个主动作；其他动作降级为辅助。
- 把“这个事项怎么来的”作为一等下钻入口。
- 把“添加反馈”放在来源反馈摘要区，作为当前问题模式吸纳新反馈实例的显性入口；从改进事项详情页进入时必须先选择候选反馈。

### 验收要点

- 首屏必须出现“当前需要你确认”或等价文案。
- 首屏只能有一个主按钮。
- 首屏必须出现“确认后会发生什么”。
- 系统理解、归因、方案、执行、回归不能全部同时平铺在首屏。
- `查看依据`、`查看全部反馈`、`添加反馈`、`这个事项怎么来的`、`查看处理链路`、`获取上下文`语义分离。
- `查看全部反馈` 是来源反馈集合的唯一查看入口；不得在辅助信息区重复放等价来源入口。
- `添加反馈` 必须出现在改进事项详情页的来源反馈摘要区，不得只藏在创建反馈 Drawer 或高级区；点击后进入候选反馈选择态。
- 首屏不得把改进事项写成单条“原始反馈”详情；必须展示来源反馈数量和共同指向。
- 当来源反馈大于 1 条时，必须展示归并依据、分歧点和拆分/移出入口。

## 3. 页面二：创建反馈 Drawer

### 当前问题

当前 Drawer 能创建反馈并生成改进事项，但创建逻辑对开发者不透明：标题如何生成、为什么归属当前 Agent、是否会归并已有事项、确认保存后是否会修改 Agent，都没有在保存前说清楚。

### 设计目标

- 输入态只收集用户自然语言反馈，不要求填写系统字段。
- 整理态解释系统如何理解反馈，以及哪些内容是初步推断。
- 保存前明确说明：保存后会新建改进事项或添加到已有改进事项，不会立即执行优化或发布。
- 给出候选已有改进事项，让开发者能选择“添加到已有事项”或“新建事项”。
- 给出归并策略状态：自动归并开启/关闭、候选相似事项数量和匹配原因。

### 验收要点

- 保存前必须说明“会发生什么 / 不会发生什么”。
- 系统整理字段必须标注“初步”或“待确认”。
- 不要求用户填写 run/session/version 等系统字段。
- 如果后端未启用自动归并，UI 必须如实提示“未自动归并”，但仍应提供手动选择已有改进事项的路径。
- 保存前必须能区分“添加到已有事项”和“新建事项”两条路径。
- 点击 `添加到已有事项` 不得直接写入关联；必须进入目标改进事项选择、目标事项审阅和确认添加流程。

## 4. 页面三：添加反馈到已有改进事项

### 当前问题

后端已经具备 `POST /api/improvements/{id}/feedbacks` 这类把来源反馈加入某个改进事项的能力，当前 UI 主路径却更像“每条反馈先新建一个事项，再靠相似归并纠错”。这不符合“改进事项 = 问题模式，反馈 = 发生实例”的抽象，也会制造过碎事项。

更具体的问题是，原方案把“添加反馈到已有改进事项”直接画成了某条反馈已经被选中的确认页。这个模型只覆盖了从某条反馈或创建反馈流程进入的后半段；从改进事项详情页点击 `添加反馈` 时，目标事项已确定，但待添加反馈尚未确定，必须先选择反馈，再展示选中反馈详情、匹配判断和确认后果。

### 设计目标

- “添加到已有改进事项”必须是一等动作，和“新建改进事项”并列。
- 从改进事项详情页进入时，流程为：选择候选反馈 → 审阅选中反馈详情 → 查看系统匹配判断 → 确认添加。
- 从创建反馈 Drawer 或反馈详情进入时，可以跳过反馈选择态，但必须选择或确认目标改进事项。
- 添加前说明为什么这条反馈适合加入该事项，以及可能的分歧点。
- 添加后不改变事项阶段，不自动重归因；如果需要重归因，应作为后续明确决策。
- 开发者可以从反馈创建 Drawer、改进事项详情页和来源反馈区进入该动作。
- 选择态必须支持搜索、筛选、已关联反馈排除和空态说明，避免开发者不知道能添加什么。

### 状态模型

| 入口来源 | 已知对象 | 必须先做什么 | 后续步骤 |
| --- | --- | --- | --- |
| 改进事项详情页的 `添加反馈` | 目标改进事项 | 选择候选反馈 | 审阅反馈详情 → 匹配判断 → 确认添加 |
| 创建反馈 Drawer 的 `添加到已有事项` | 待保存或刚保存的反馈 | 选择候选改进事项 | 审阅目标事项 → 匹配判断 → 确认添加 |
| 反馈详情页或反馈 Inbox | 待添加反馈 | 选择候选改进事项 | 审阅目标事项 → 匹配判断 → 确认添加 |

### 验收要点

- 创建反馈和改进事项详情页都必须能进入“添加到已有改进事项”路径。
- 从改进事项详情页进入时，必须先出现候选反馈选择态；只有入口已携带 feedback id 时才允许跳过。
- 从创建反馈 Drawer 或反馈详情进入时，必须先出现候选改进事项选择态；只有入口已携带 improvement id 时才允许跳过。
- 选择态必须排除已关联反馈，并说明被排除数量或原因。
- 选择态必须支持搜索、来源/状态/时间筛选和空态。
- 选择反馈后必须展示反馈原文、Run、Session、Agent Version、发生时间等详情。
- 选择目标改进事项后必须展示事项标题、问题模式、当前阶段、来源反馈数量和匹配原因。
- 添加前必须展示共同点、分歧点和确认后果。
- 添加操作不得静默推进阶段或自动改写归因。
- 添加成功后，目标事项的来源反馈数量必须更新，并能看到新增反馈。

## 5. 页面四：反馈保存成功态

### 当前问题

保存成功后只告诉用户“已生成改进事项”，但没有解释为什么新建、是否添加到已有事项、下一步是谁处理、用户是否需要立刻查看。

### 设计目标

- 保存成功态不是普通 toast，而是反馈到改进事项的桥接状态。
- 明确展示处理结果：新建事项、添加到已有事项、自动归并或手动归并。
- 给出下一步说明和直接入口。
- 支持开发者纠错：查看事项、调整归并、关闭。

### 验收要点

- 成功态必须展示处理结果：新建 / 添加到已有事项 / 归并 / 拆分。
- `查看改进事项` 必须选中刚创建、刚添加或归并到的事项。
- 如果子资源写入失败，成功态必须说明“事项或反馈关联已写入，但系统理解/来源反馈补写失败”并给出重试或查看错误入口。

## 6. 页面五：改进事项列表

## 7. 页面六：运行证据 / Trace 面板

### 当前问题

### 设计目标

- Trace 面板服务当前决策，而不是独立观测工具。
- 默认展示证据摘要：关联 run、关键事件、工具调用、异常或缺失。
- 保留打开 Langfuse 的深度入口。
- 标明证据状态：已加载、缺失、拉取失败、与当前事项不匹配。

- 证据面板不能替代 `获取上下文` 导出。

## 8. 页面七：获取上下文抽屉

### 当前问题

获取上下文是重要能力，但容易被误解成“查看详情”。它实际服务 AI/外部协作的复制、下载和再分析。

### 设计目标

- 明确定位：这是导出上下文，不是普通详情阅读。
- 按用途选择上下文类型：问题分析、AI 续写、Playwright 复现、JSON。
- 在预览区标明包含哪些对象、缺失哪些对象。
- 与 `查看依据`、`查看全部反馈` 和 `添加反馈` 分离。

- 抽屉必须声明“用于导出/协作”，避免和详情入口混淆。
- 预览必须标明缺失对象，不能静默输出空字段。
- 上下文内容应包含当前待决策问题，而不仅是对象 dump。

## 9. 页面八：高级设置 / 高级治理区

### 当前问题

“高级”容易变成杂物区：自动化策略、相似归并、闭环对象关联都被塞进去，但它们的用户任务不同。

### 设计目标

- 高级区只放低频、风险较高或系统管理性质的能力。
- 高级区内部继续按任务分组，而不是简单平铺。
- 每个高级动作必须说明风险和影响范围。

### ASCII 草图

```text

│                                                                      │
│ 自动化策略                                                             │
│ 当前模式：人工触发                                                     │
│ 说明：自动推进只改变事项阶段，不直接修改 workspace 或发布版本。          │
│ [切换策略] [自动推进一次]                                               │
│                                                                      │
│ 闭环对象关联                                                           │
│ 已关联：change_set: agc-demo                                           │
│ [添加关联] [查看关联对象]                                               │
│                                                                      │
│ 危险操作                                                               │
│ [归档事项]                                                             │
└──────────────────────────────────────────────────────────────────────┘
```

### 验收要点

- 高级区默认收起。
- 自动化策略、归并关系、闭环对象关联、归档必须分组。
- 危险操作不得和主动作并列。
- 每个高级动作必须说明影响范围。

## 10. 页面九：发布门禁页

### 当前问题

发布页已有门禁概念，但它也可能落入对象展示：候选变更、回归、归因、方案等如果缺少“能不能发布”的判断焦点，开发者仍需自己综合。

### 设计目标

- 首屏回答：当前候选版本能不能发布。
- 门禁按原因组织，而不是按内部对象平铺。
- 阻塞项能反查到改进事项和证据。
- 强制发布必须明确风险、后果和审计记录。

### ASCII 草图

```text
│                                                                      │
```

- 发布页必须有明确发布判断：可发布 / 暂不可发布 / 需要人工确认。
- 阻塞原因必须可点击反查到改进事项或证据。
- 强制发布必须与普通主动作视觉分离。

## 11. 页面十：资产 Registry

### 当前问题

资产 Registry 已显示 provenance，但从治理任务角度看，开发者更关心“这个资产为什么存在、来自哪个改进事项、能否复用到其他 Agent”。

### 设计目标

- 资产列表以查找和追溯为主，不默认进入创建表单。
- 资产详情展示来源改进事项、生成原因、适用 Agent 和继承关系。
- 对回归资产、方法论资产、执行资产、审计资产采用一致 provenance 展示。

### ASCII 草图

```text
┌──────────────────────────── 资产 Registry ───────────────────────────┐
│ 范围：main-agent        类型：[全部]        来源：[改进事项 / 手工]      │

├──────────────────────────────────────────────────────────────────────┤
│ 回归资产：时间窗口不一致不得误判                                      │
│ 来源改进事项：sec-ops-data 时间窗口误判治理                            │
│ 适用 Agent：main-agent        继承：无                                 │
│ [打开来源事项] [查看资产内容]                                          │
│                                                                      │
│ 方法论资产：MCP 数据时间窗口核验 SOP                                  │
│ 来源改进事项：sec-ops-data 时间窗口误判治理                            │
│ 适用 Agent：main-agent        可复用：待评估                           │
```

### 验收要点

- 每个资产必须展示来源：改进事项 / 手工 / 继承。
- 能从资产回跳改进事项。
- 创建资产入口不应挤占浏览和追溯主流程。

## 12. 实施顺序建议

1. 先重构改进事项详情页的信息架构，建立“当前待决策”首屏。
2. 同步调整创建反馈 Drawer、添加到已有改进事项路径和保存成功态，让 provenance 从源头清晰；添加路径必须覆盖“先选反馈”和“先选目标事项”两种入口变体。
3. 改进列表项，加入待决策摘要，使列表成为工作队列。
4. 把 Trace、来源反馈、上下文导出改为服务“当前决策”的下钻入口。
5. 整理高级区，按归并、自动化、闭环对象、危险操作分组。
6. 更新 Playwright 设计一致性脚本，新增首屏决策、主动作唯一、后果说明、provenance 可见、多反馈归并、添加反馈选择态、详情审阅态、空态和确认态等断言。

## 13. 验收矩阵

| 验收项 | 目标 |
| --- | --- |
| 首屏待决策 | 改进事项详情页首屏必须有明确“当前需要你确认” |
| 主动作唯一 | 当前阶段首屏只出现一个主按钮 |
| 依据入口 | 必须有 `查看依据`，且只服务当前待决策问题 |
| 后果说明 | 主动作附近必须说明确认后会发生什么、不会发生什么 |
| 来源解释 | 必须能查看“这个事项怎么来的” |
| 多反馈兼容 | 改进事项详情必须支持 `N 条反馈 / M 个 Run`，展示共同指向和分歧点 |
| 来源反馈入口去重 | 来源反馈集合统一使用 `查看全部反馈`；不得同时出现多个等价入口 |
| 添加反馈入口 | 改进事项详情页必须显性提供 `添加反馈`；也必须能从创建反馈流程添加到已有改进事项 |
| 添加反馈选择态 | 从改进事项详情页进入时，必须先选择候选反馈；从反馈入口进入时，必须先选择候选改进事项 |
| 添加反馈详情审阅 | 选择反馈后必须展示反馈详情；选择目标事项后必须展示目标事项详情和匹配原因 |
| 添加反馈空态 | 无候选反馈或无候选事项时必须展示空态、排除原因和下一步入口 |
| 归并纠错 | 必须能查看或调整归并关系 |
| 细节分层 | Trace、来源反馈、处理链路、上下文导出不混为一个入口 |
| 高级分组 | 自动化策略、归并、关联对象、危险操作分组展示 |
| UI 自动验收 | `verify:design-parity` 增加决策型 UI 断言 |

## 14. 不做事项

- 不把所有系统对象继续平铺，只换视觉样式。
- 不把“获取上下文”当成普通详情入口。
- 不把自动化策略、归并、归档与主决策并列展示。
- 不在 UI 中伪造后端尚未提供的真实自动归因、真实自动归并或真实发布能力。
- 不把改进事项退化成单条反馈详情页；单条反馈只能作为来源证据存在。
- 不在缺少待添加反馈或目标改进事项时直接展示最终确认态。
- 不把已关联反馈、跨 Agent 反馈或不符合筛选条件的反馈静默混入候选列表。
