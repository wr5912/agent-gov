## 1. 决策摘要

导入能力解决了「外部包进平台」，但平台缺少反向的三条通路，导致资产闭环不完整：

- **导出缺失**：没有在线导出任何业务 Agent workspace 的能力。导入方案默认「开发者本地已有源 workspace」，
  但对平台上已经跑起来的 Agent，这个前提不成立。
- **seed 只能同 ID 实例化**：平台预置了多个成熟业务 Agent，但只能用与 seed 同名的 ID 各实例化一次；
  想基于某个成熟 Agent 做变体，只能绕开平台手工复制。唯一可多次实例化的通用模板近乎空壳。
- **归档回平台无路径**：把一个调好的 Agent 沉淀成随产品发布的预置基线，当前只有隐式的仓库操作，无成文流程。

本文补齐这三条，形成闭环：**导出 → 本地修改 → 导入覆盖 → 归档为 seed**。

贯穿全文的思路是一条：**能交给既有机制的，就不新造机制**。三条通路各自都有一个"看起来更直接"的做法，
本文都拒绝了，且理由同源——

- 导出可以「按忽略规则拷贝工作树」，但那要重造受管树边界的全部规则。改用 Git commit 为单位，
  「哪些内容属于这个版本」这个判断就仍然只有 Git 一个答案。
- 从 seed 实例化可以「渲染模板变量替换身份」，但 seed 里根本没有模板变量，且散文式的自我描述也替换不掉。
  改为只归一化后端拥有的注册表 ID，内容修正复用导出与导入——这条闭环反正要建。
- 归档可以做成「一键在线晋升」，但容器里的 seed 目录是只读挂载，而且 origin 是磁盘扫描的派生投影、
  重启就会被校正。改为离线仓库提交，顺带保住了脱敏门。

因此读本文时可以用一个问题检验每处设计：**它有没有制造第二套真相源？** 这也是下列决策的共同依据。

首版采用以下已确认决策：

- 导出以 Git commit 为单位，复用既有的 per-Agent Git archive 能力；工作树导出要求先创建快照，不新造
  「按忽略规则拷贝工作树」的第二套过滤实现。
- 从 seed 实例化到新 ID 时**不做模板变量渲染**。后端只归一化它拥有的注册表 ID 与 name；workspace 内容属包所有者，
  内容层面的身份修正走导出 → 本地修改 → 导入覆盖，复用既有闭环，不新增第二套身份机制。
- 归档为 seed 是**离线动作**，通过仓库提交完成。在线写入 seed 在容器中物理上不可行，且会破坏离线不变量与脱敏门。
- 本文不新增 origin 取值、不改动 seed 只读挂载、不改动受管策略的触发条件。

## 2. 产品目标与非目标

### 2.1 产品目标

1. **闭环完整**：平台上的 Agent 能被导出成与导入方案同形态的包，改完能导回去。
2. **成熟资产可复用**：预置的成熟业务 Agent 能作为新 Agent 的起点，不必从空模板重建。
3. **沉淀有路径**：调好的 Agent 能按成文流程沉淀为随产品发布、可审计、可复现的预置基线。
4. **边界诚实**：在线能做什么、只能离线做什么、离线做完实际生效到什么程度，全部写明，不制造误期待。
5. **不新增身份机制**：导出、实例化、归档共用同一套包形态与受管策略，不各自定义。

### 2.2 非目标

- 不提供把已导入 workspace **在线**晋升为 seed 或预置基线的能力（见 5.1 的物理约束论证）。
- 不新增 origin 取值，不改动 seed 目录的只读挂载与其回归测试锁定的部署不变量。
- 不修改受管策略按 agent_id 绑定的触发条件（该项风险记录在导入方案的安全边界章节，独立推进）。
- 不做模板变量渲染，不把既有 seed 改造成模板变量形式。
- 不导出 SDK transcript、conversation、run、feedback、EvalRun、release archive 或 Langfuse 数据。
- 不导出本地私有覆盖、密钥、缓存和运行态工件；这些本就不在 Git 受管树内。
- 不引入跨租户模板市场或第三方分发。

## 3. 治理对象与所有权

| 维度 | 结论 |
| --- | --- |
| 被治理对象 | 业务 Agent 的 Git 受管 workspace，及其在仓库中的声明式基线副本 |
| 治理执行者 | 后端确定性导出、既有 Git archive、受管策略校验；归档由仓库评审承担 |
| 不参与者 | 治理 Agent 不参与导出、实例化或归档，也不输出其中任何字段 |
| 资产类型 | 导出包属执行资产；commit 属版本资产；归档提交属仓库声明资产 |
| 生命周期 | 导出不改变 Agent 生命周期；从 seed 实例化按既有创建流程产生 active Agent |
| 当前实现边界 | 已有 per-Agent Git archive、同 ID seed 实例化、只读 seed 挂载；没有在线导出、跨 ID seed 实例化和成文归档流程 |
| 目标能力边界 | 在线导出、跨 ID seed 实例化、离线归档流程 |

字段所有权与导入方案一致：注册表 ID、name、origin、commit SHA、digest 属后端；workspace 内容属包所有者；
seed 声明的 MCP server 条目与受管 marker 区段属后端，位于包内但不接受包内改写。

## 4. 导出

### 4.1 版本语义

导出以 **commit** 为单位。这不是任意选择，而是让「受管树边界」成为导出边界的唯一办法：既有 per-Agent
Git store 已提供按 ref 归档为 gzip tar 并返回 sha256 的能力，而 Git archive 只包含该 commit 的已跟踪文件——
本地私有覆盖、密钥、缓存和运行态工件本就不在 commit 里。换言之，「导出不泄露私有内容」这条安全性质由
受管树边界免费提供，不需要导出侧再实现一套过滤，也就不存在两套过滤规则漂移的风险。

导出当前工作树要求调用方先创建快照 commit。不提供「按忽略规则拷贝工作树」的旁路，理由与上一段是同一条：
那条旁路要重造受管树边界的全部规则（包括跳出 workspace 的运行态忽略规则），既贵又会立刻制造第二套过滤实现。
先快照再导出，把「哪些内容属于这个版本」这个判断继续交给 Git，代价只是多一次提交。

### 4.2 接口形态

```text
POST /api/agent-registry/{agent_id}/workspace-exports
  -> {export_id, commit_sha, sha256, size_bytes, created_at}
GET  /api/agent-registry/{agent_id}/workspace-exports/{export_id}/content
  -> application/gzip
```

前缀必须是 `/api/agent-registry`，不能是 `/api/agents`。这是既有事实约束而非风格偏好：per-Agent 子资源的
既有模式是 `/api/agent-registry/{agent_id}/...`（如 lifecycle）；而 `/api/agents` 已被 catalog 路由占用且语义
不同——它列出某业务 Agent 工作区内的 Claude 子 Agent，其 `agent_id` 是**查询参数**。在 `/api/agents` 上新增
路径参数会形成同前缀异语义的双轨。更关键的是 OpenAPI 契约审计的 domain 前缀集合包含 `/api/agent-registry`
而不包含 `/api/agents`，走错前缀会让新端点落在契约审计面之外。

采用两步而非单请求直接下载：产包响应仍走 typed response model，OpenAPI 契约完整；调用方可先校验 sha256
与大小再决定是否取流；大包不阻塞产包请求。

导出端点是仓库内首个二进制响应——当前没有任何路由返回附件或文件流。实现阶段必须确认 OpenAPI 契约审计
能否描述该响应形态，并在无法描述时先给出契约门的处理方式，不得静默绕过。

`agent_id` 必须经既有的业务 Agent 布局解析取得工作区路径，该解析内含目录穿越校验；不得自行拼接路径。

### 4.3 既有归档能力的六个缺口

4.1 说「受管树边界免费提供了安全性质」，那是就**内容**而言的；就**接口**而言，既有归档能力是为发布流程
写的，不是为在线导出写的。下列六项必须补齐后才能复用，不能直接照搬。本清单的作用是充当复用前的检查表，
因此要求完备：漏掉任何一项都会在阶段 A 的退出标准上暴露。

- **无产出侧大小限额**：既有安全限额全部是解压侧预算，用于「别人给我一个包，我解压」。导出是产出侧，
  需要在打包前统计文件数与字节数并设上限。沿用同一组数值可以，但这是新写实现，不是复用代码。
- **不做内容扫描**：归档路径完全不经过内容扫描。若导出需要敏感信息门禁，须在归档之外加一层，
  且只能用只读扫描，不能用会就地改写文件的脱敏函数。
- **产物命名会互相覆盖**：既有归档产物落在 release 目录且命名只含 ref 与 commit 摘要，同一 commit 反复导出
  会覆盖同一文件。在线并发下这是共享可变状态，导出须使用独立的临时产物目录，不复用 release 目录。
- **未加变更保护**：既有归档路径未加 mutation guard。读历史 commit 本身安全，但需确认与回滚、发布类操作并发时的交互。
- **只接受 commit**：与 4.1 的版本语义一致，工作树导出必须先快照。
- **归档根前缀缺失**：既有调用未传 `--prefix`，产出 tar 顶层直接是 `CLAUDE.md`、`.claude/`、`.mcp.json`，
  没有 `workspace/` 根目录，不满足导入方案的包契约（该契约要求包必须恰好包含一个 `workspace/` 根目录）。
  导出须以 `--prefix=workspace/` 归档。这一项单独看只是一个参数，但它承载 8.1 的往返锚点——两文能否共用
  同一包形态，物理上就取决于它；发布路径的既有产物形态不受影响，因为那是另一条调用。

### 4.4 顺带修复的存量缺陷

现有的运行态模板导出脚本**导不出任何业务 Agent**：它的允许源目录集合只认早期的两个顶层工作区名，
其中一个已是废弃布局；而业务 Agent 位于 `data/business-agents/<id>/`，`data` 恰好在该脚本的排除目录名单里。
因此 README 中「从当前运行态保存模板」的说法对业务 Agent 是失效承诺。

该脚本的 staging 优先、高危 finding 阻断的编排值得沿用，但其目录白名单与本能力的 per-Agent 语义不匹配。
实现阶段需修正该脚本的适用范围声明，或明确它只服务治理 Agent 工作区。

## 5. 归档为 seed

### 5.1 在线不可行的物理约束

seed 目录在容器中以只读方式绑定挂载，覆盖整棵目录树（含模板 catalog），且该只读约束有回归测试锁定。
运行态容器写回 seed 会直接失败。

更根本的是身份来源：`origin` 不是可授予的状态，而是**磁盘目录扫描的派生投影**——每次启动重新同步注册表时
会按仓库 seed 目录强制校正已存在行。直接更新数据库把 origin 改成 seed，下次重启即被打回。
seed 身份判定所用的路径是硬编码的，没有环境变量活口。

存在一个必须写明的不对称：本机调试模式下该路径指向真实仓库目录、是可写的。因此「在线晋升」会在本机调通、
在容器中静默失败——这是最坏的一类环境差异，也是不把它做成在线能力的直接理由。

### 5.2 离线归档流程

```text
在线导出包 (第 4 章)
  -> 开发者本地解包并复核脱敏（运行态 workspace 含真实 MCP 地址与凭据形态内容）
  -> 写入仓库 seed 目录 docker/runtime-volume-seeds/data/business-agents/<agent_id>/workspace/
  -> 仓库评审与提交
  -> 重建或重启后生效
```

该流程保住三件事：离线不变量（seed 随代码版本发布、可审计、可复现）；脱敏门（运行态工作区含真实地址与凭据，
在线直写等于把凭据写进本该可提交的声明源）；单向数据流（仓库 → 运行态，不产生双向漂移与多副本不一致）。

### 5.3 必须写明的两个落差

**seed 内容对已存在的卷不生效。** 运行卷引导逻辑在目标 workspace 已存在时整体跳过，绝不逐文件回灌——
这是有意设计，为避免把版本中有意删除的文件复活。因此对一个已经在跑的 Agent，离线归档**只改变它的身份标签，
不改变它的运行内容**。若期望是「归档一份配置快照」，实际得到的是「一个 origin 标签」；配置快照的真正载体是
第 4 章的导出包与 Git 历史，不是 seed 目录。

**归档会激活此前被跳过的 seed 基线漂移检查。** 这条的机制容易被说错，因此写清楚：归档**不会**把该 Agent
「纳入」启动期受管检查面——受管 Agent 集合是「seed 目录中的 Agent」与「注册表中未删除的 Agent」的并集，
任何在跑的 Agent 本就在集合内，与 origin 无关。归档真正改变的是另一件事：受管策略会为「seed 目录中存在
同名工作区」的 Agent 解析出一份 template workspace 作为基线，而用户来源 Agent 此前解析结果为空，因而
pre_tool_guard 比对、MCP 条目对账和安全文本区段检查**整段跳过**。归档后这份基线存在了，这些检查开始生效。

后果是真实的但路径是间接的：检查生效意味着该 Agent 更容易产生受管策略变更，而启动期只对「产生了变更」的
Agent 施加「工作树脏」与「未终结 change set」两道阻断。也就是说，归档不新增阻断规则，但会让原本不触发这些
规则的 Agent 开始触发。归档一个日常会被手工改动的 Agent 前，需要预期它此后必须保持工作树干净、
或与 seed 基线一致。

### 5.4 seed 身份的实际价值

穷举 origin 分支后，seed 身份带来的行为差异只有两项：注册表删除被拒绝，以及设置页展示一个声明基线徽章。
它不带来禁改名、禁导入或生命周期约束。

因此若诉求只是「防止误删」，seed 是错配的手段——正确做法是给用户来源 Agent 一个独立的受保护标志，
而不是挪用一个绑定了仓库布局、seed 基线漂移检查和只读挂载的身份。该方向不在本文范围内，需要时另立方案。

把 5.3 与本节合起来看，归档为 seed 的真实收益与代价并不对称：收益是一个删除锁和一个徽章，代价是仓库提交
流程、5.3 的两项语义落差，以及此后与 seed 基线保持一致的义务。这个判断不阻止归档，但要求归档是一个
有意识的决定，而不是「调好了就顺手归档」的默认动作。

## 6. 从 seed 实例化到新 ID

### 6.1 当前缺口

创建 Agent 时的 seed 复用路径用**新 Agent 自己的 agent_id** 去 seed 目录里查找同名声明，因此只支持
「用 seed X 创建 Agent X」。想基于某个成熟 seed 创建变体时，该查找必然落空，回退到通用空模板。

结果是平台预置的多个成熟业务 Agent 各自只能实例化一次，其积累的 prompt、skill、hook 和 MCP 配置无法被复用为起点。

### 6.2 设计：只归一化后端拥有的身份

实例化接口接受**源 seed ID** 与**目标 agent_id** 两个参数，允许二者不同。后端读取源 seed 工作区作为出生配置，
只归一化它拥有的注册表 ID 与 name，产生的 Agent origin 仍为用户来源。

**不做模板变量渲染**，理由有二：

其一，seed 工作区**不含模板变量**——模板变量只存在于通用模板。对 seed 执行渲染是空操作，实现它只会制造
「已处理身份」的假象。

其二，渲染是半吊子。模板变量只能替换字面 token，替换不掉 seed 工作区里与领域强绑定的散文式自我描述。
做完渲染仍需人工修改内容，却多欠一套机制。而后端归一化注册表身份、内容归包所有者，与导入方案的字段所有权
契约是同一条原则，不是新发明。

因此内容层面的身份修正走既有闭环：**导出新 Agent → 本地修改 → 导入覆盖**。这条闭环本就要建，复用它是零成本。

### 6.3 必须告知用户的语义

从 seed X 实例化出的 Agent Y，其平台身份是 Y，但工作区内容中与 X 相关的自我描述保持原样。这是设计选择而非缺陷：
内容属包所有者，平台不揣测哪些自我描述该改、哪些是真实业务语义。UI 与文档须明确提示「内容按原样复制，
身份相关表述需自行调整」，并给出导出改导入的修正路径，不让用户误以为平台已经改好。

### 6.4 一个必须一并处理的不一致

seed 根目录的环境变量覆盖只被部分代码路径认可：工作区准备与受管策略解析认它，而 seed 身份判定硬编码路径、
不认它。设置该变量时三者会指向不同位置，产生「内容来自 A、身份判定看 B」的分裂。实现跨 ID 实例化前需先收口
该覆盖的单一真相源，否则新增的源 seed 参数会放大这个分裂。

## 7. 安全与信任边界

- 导出返回的是 Git 已跟踪内容。密钥、本地私有覆盖与运行态工件不在受管树内，因此不随导出流出；
  该性质由受管树边界保证，不依赖导出侧再做一次过滤。
- 但受管树内**可能**存在真实的内网地址、endpoint 与账号形态内容。导出面向持有平台 API Key 的可信内部开发者，
  该假设降低的是分发范围，不降低归档前的脱敏复核要求（见 5.2）。
- 归档写入仓库前必须完成脱敏复核。仓库是可提交的声明源，其边界严于运行态工作区。
- 导出与实例化均记录 backend-owned actor、operator 标签、agent、commit 与时间。
- 日志与 metrics 只记录类型、大小、耗时、digest 前缀和相对路径，不记录包正文、prompt、token 或宿主绝对路径。
- 从 seed 实例化会复制源 seed 的全部能力配置。受管策略按 agent_id 绑定而非按能力特征绑定，
  因此跨 ID 实例化一个携带高风险能力的 seed 会绕开该能力对应的治理约束。该风险与其修复方向记录在
  导入方案的安全边界章节，两条入口共用同一个修复，不在本文做局部补丁。

## 8. 测试同步矩阵

### 8.1 导出

- 按 commit 导出的产物 sha256 与大小稳定可复现；产物只含该 commit 已跟踪文件。
- 被忽略的私有文件、密钥、缓存与运行态工件不出现在产物中。
- 工作树脏时导出请求要求先快照，不静默导出一个不对应任何 commit 的树。
- 产出侧限额：超过文件数或字节数上限的工作区被拒绝，且拒绝发生在打包前。
- 同一 commit 并发导出不互相覆盖产物。
- 未知 agent_id、目录穿越形态的 agent_id、非法 ref 均被拒绝且不泄露宿主路径。
- 产物顶层恰好是一个 `workspace/` 根目录（即归档带 `--prefix`），可被导入方案的解包器接受。
  该用例是唯一的跨文档契约锚点，取一个不含真实 endpoint 的工作区作为契约样本——含真实 endpoint 的工作区
  会被导入方案的内容规则拒绝，那是既定契约而非往返失败（见 9.1）。

### 8.2 从 seed 实例化

- 用源 seed X 创建 Agent Y：注册表身份为 Y，origin 为用户来源，工作区内容与 X 的受管树一致。
- 源 seed 不存在、源 seed 为空、源 seed 是符号链接或非目录时明确失败。
- 同 ID 实例化行为不回归。
- 实例化产生的 Agent 通过其自身 agent_id 对应的受管策略校验。
- seed 根目录环境变量覆盖在内容准备、受管策略解析和身份判定三处解析一致。

### 8.3 归档

- 归档流程的文档化验收：按 5.2 流程操作后重启，目标 Agent 的 origin 变为 seed 且删除被拒绝。
- 已存在卷上的归档只改身份不改内容——该用例固化 5.3 的第一个落差，防止实现阶段误加回灌逻辑。
- 归档后 seed 基线漂移检查生效：使归档 Agent 的工作区偏离 seed 基线，受管策略应产生对应变更；
  该用例固化 5.3 的第二条语义。不要断言「归档使 Agent 进入受管集合」——它本就在集合内，该断言不成立。
- 只读挂载不变量的既有回归测试保持通过，不因本能力被放宽。

### 8.4 质量策略与命令

- 内循环：目标 pytest、OpenAPI 类型生成、前端类型检查。
- 主流程：`make main-flow-test`。
- 治理硬门：`make codex-guard`。
- 提交、CI 或发版：`make test`。

## 9. 验收标准

### 9.1 功能验收

- 能从任一业务 Agent 导出与导入方案同形态的包。导回受导入方案的内容规则约束：含真实 endpoint、
  凭据形态内容的工作区必须先完成参数化或脱敏复核才能导回。这是设计意图不是缺陷——导出面向可信内部开发者，
  导入是接入边界，两者的内容门槛本就不同；把它写成无条件承诺会得到一条无法证伪的验收标准。
- 能用源 seed 创建不同 ID 的新 Agent，且新 Agent 通过其自身的受管策略校验。
- 归档流程可按文档执行，重启后身份生效，且执行者对 5.3 的两条语义落差没有误期待。

### 9.2 负向验收

- 不能在线写入 seed 目录，不能通过更新数据库获得 seed 身份。
- 不能导出未跟踪的私有文件、密钥或运行态工件。
- 不能在工作树脏时导出一个不对应任何 commit 的树。
- 不能让导出产物、日志或响应泄露 API Key、宿主绝对路径或包正文。
- 不能让本机调试模式下能通过、容器中失败的路径进入产品能力。

### 9.3 架构验收

- 不新增 origin 取值，不放宽只读挂载，不改动受管策略触发条件。
- 不复制受管树边界规则、忽略规则或安全限额；产出侧限额是新增实现且只有一处。
- 导出、实例化与导入共用同一个包形态定义，无第二套包契约。
- 导出的二进制响应形态在 OpenAPI 契约门中有明确处理，不静默绕过。

## 10. 分阶段交付

### 阶段 A：导出

- 产出侧限额、独立产物目录、按 commit 归档与 sha256 契约。
- 二进制响应形态与 OpenAPI 契约门处理。
- 修正运行态模板导出脚本的适用范围声明。

退出标准：导出产物可复现、不含私有内容、能被导入解包器接受；契约门通过。

### 阶段 B：从 seed 实例化

- 收口 seed 根目录环境变量覆盖的单一真相源。
- 实例化接口接受源 seed ID 与目标 agent_id。
- UI 与文档明确内容按原样复制的语义与修正路径。

退出标准：跨 ID 实例化产生的 Agent 通过其自身受管策略校验；同 ID 行为不回归。

### 阶段 C：归档流程成文

- 在集成指南补齐归档流程、脱敏复核清单与两个落差说明。
- 补齐归档与只读挂载的负向验收。

退出标准：归档流程可被未参与本方案的开发者按文档独立完成，且不产生对 seed 内容生效范围的误期待。

## 11. 运行环境、数据与发布边界

- 容器模式继续使用 `docker/.env` 和宿主 `${HOME}/volume-agent-gov`；本机调试使用 `docker/.env.local-debug`
  和 `/tmp/local-debug-volume-agent-gov`，二者是按运行环境选择，不是覆盖关系。
- 不增加新的 env 选择规则或 Docker volume。导出产物位于现有 data root 的受控临时边界，并按保留期清理。
- 在线导出只读取 `data/business-agents/<agent_id>/workspace` 的 Git 内容；不修改
  `docker/runtime-volume-seeds/`、`claude-root/`、`version/` 目录布局或 Langfuse 数据。
- 归档写入仓库 seed 目录，属仓库提交动作，不经由运行态容器。
- 本方案不 bump `VERSION`、不创建 tag。

## 12. 文档治理

本文与导入方案是同一资产闭环的两个方向，拆分而非合并，因为读者与决策点不同：导入面向「把外部包接进平台」
的接入者与安全评审；本文面向「把平台资产取出与沉淀」的开发者与发布评审。合并会使单文档超出可评审规模。

两文共用同一个 workspace 包形态与受管策略边界。包契约的单一真相源是导入方案的 workspace 包契约章节，
本文引用而不复述；若二者出现分歧，以导入方案为准并修正本文。
