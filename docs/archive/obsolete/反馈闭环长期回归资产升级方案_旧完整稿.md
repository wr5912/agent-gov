# 反馈闭环长期回归资产升级方案

> 归档说明（2026-06-23）：本文是旧完整稿，仅保留历史设计和审计价值。文中的 `eval-case-governor`、`regression-impact-analyzer` 等旧 profile/workspace 表述属于合并前设计；当前运行态治理执行者已收敛为单一 `governor`，活跃资产方案见 `docs/反馈闭环长期回归资产升级方案.md`。

> 用途：定义反馈闭环从“批次内回归验证”升级为“长期回归资产沉淀与持续防复发”的产品、架构和项目改造方案。
> 状态：设计方案稿（评审整改版）
> 日期：2026-06-01
> 文档层级：当前实现基线（pre-v2.7）上的长期回归资产方案。
> 术语口径：本文围绕当前 `eval_cases`、`eval_runs`、优化批次和反馈闭环实现描述升级路径；v2.7 中 `RegressionAsset`、`RegressionRun`、`ImprovementItem` 与资产 Registry 的统一术语见 [AgentGov术语与版本边界](./AgentGov术语与版本边界.md)。
> 归档边界：本文仍是回归资产升级的实现依据和评审整改承接文档，暂不移动。
> v2.7 覆盖规则：本文保留长期回归资产治理设计；测试发布阶段的用户入口、测试资产与计划、回归执行状态、覆盖场景、发布门禁预览以 [AgentGov v2.7 四阶段改进治理工作台 UI 整改方案](./AgentGov_v2.7_四阶段改进治理工作台UI整改方案.md) 和 `docs/imgs/测试发布.png` 为准。旧“回归资产”独立主路径不作为未来改进治理主流程。

本文档面向 `agent-gov` 当前反馈优化闭环，描述如何把反馈生成的测试用例从当前优化批次的临时验证材料，升级为后续所有优化批次持续使用的长期回归资产。

核心结论：

```text
反馈信息
  -> 候选测试用例
  -> 用例治理
  -> 长期回归资产
  -> 后续优化批次分层回归
  -> 回归影响分析
  -> 发布 / 阻断 / 修复 / 用例库反哺
```

长期目标不是“每次多跑一些测试”，而是让每一条高价值反馈都沉淀为系统未来不会再次复发的能力保护网。

## 0. 评审核查与整改决策

`docs/design_review_report/反馈闭环长期回归资产升级方案评审报告.md` 已按真实代码逐条核查。结论是：报告指出的 12 个高危阻断项均属实；其余中低项未发现需要推翻的事实错误，按设计缺口或一致性债处理。

本方案进入实施前必须先关闭以下整改线：

1. **数据模型地基**：拆除 `source_feedback_case_id` 唯一约束，补旧数据回填默认值，明确 SQLite 迁移方式，并收口 `payload_json` 与显式列的投影关系。
2. **门禁自动化**：新增“改动分类 + 相关性选例”确定性引擎；`promotion_status`、`asset_layer`、`blocking_policy` 必须真正参与 regression plan，而不是只作为展示字段。
3. **运行可承受**：批次回归必须后台执行，包含 per-case timeout、预算、并发上限、冻结 Agent 版本、隔离 workspace 与互斥/队列。
4. **治理可审计**：晋级、归档、flaky、revision、break-glass 都必须记录 operator、reason、before/after。
5. **产品边界反转显式化**：现有产品文档曾明确“回归测试不作一级菜单”。本方案因长期资产跨反馈、跨批次治理，决定新增反馈工作台侧边栏菜单“回归资产”；这是对既有边界的有意修订，不是简单补充。

本文档仍是设计稿；代码落地前，应以本节整改线和第 15 节阶段计划作为前置门槛。

## 1. 背景与问题

当前反馈优化闭环已经具备以下基础能力：

```text
反馈信息
  -> 默认回归用例
  -> 优化批次
  -> 归因分析智能体
  -> 优化方案生成智能体
  -> 执行优化智能体
  -> 批次回归测试
```

现有实现中已经存在：

1. `eval_cases`：反馈生成或手工创建的评估用例。
2. `eval_runs` / `eval_run_items`：回归运行及逐条结果。
3. 批次的 `eval_case_ids`：存于 `feedback_optimization_batches.payload_json`，表示批次关联用例。
4. `FeedbackEvalRunner`：按 eval case 调用主智能体并记录结果。
5. 优化批次详情中的“回归测试”区域：执行当前批次关联用例。

但当前模型仍更接近：

```text
当前批次选择了哪些 eval_case_ids
  -> 应用优化后跑这些用例
  -> 得到本批次结果
```

缺口在于：反馈生成的用例还没有被明确建模为长期回归资产。这样会出现典型退化：

```text
批次 1 修好了 A 问题，并通过 A 对应测试。
批次 2 为了解决 B 问题压缩输出模板，又把 A 的证据来源或置信度删掉。
```

如果 A 用例只绑定在批次 1 上，批次 2 不持续运行它，旧问题就会复发而不被发现。

## 2. 设计目标

本次升级的目标是建立“反馈 -> 用例 -> 优化 -> 回归 -> 防复发”的完整闭环。

### 2.1 产品目标

1. 反馈生成的高价值用例可以沉淀为长期资产。
2. 每个后续优化批次都能按策略选择相关历史用例持续验证。
3. 回归失败时能明确判断是历史问题复发、当前批次目标失败、用例过时，还是 flaky。
4. 开发人员可以在统一页面治理长期回归资产。
5. 批次发布前有明确阻断规则，不依赖人工临场判断。

### 2.2 工程目标

1. `SQLite eval_cases` 继续作为长期回归用例的权威存储。
2. 不把 `${HOST_RUNTIME_VOLUME_ROOT}/main-workspace/evals/` 作为权威用例库；它可以保留为主智能体 workspace 中的文件资产或执行优化结果。
3. 用例编辑必须保留 revision，使历史运行结果能追溯当时的用例内容。
4. 回归计划、门禁结果、影响分析结果必须可审计。
5. 新增智能体必须符合现有 Runtime Profile 隔离模型，不共享 workspace 和 claude root。
6. 新增智能体和回归影响分析不改变发布门禁权威；发布门禁只读取回归运行同步生成的门禁结果。

## 3. 新闭环流程

升级后的正式链路为：

```text
反馈信息
  ↓
候选测试用例生成
  ↓
用例治理智能体分析
  ↓
人工审核 / 去重 / 分层 / 晋级
  ↓
长期回归资产库
  ↓
优化批次创建
  ↓
归因分析智能体
  ↓
优化方案生成智能体
  ↓
回归计划生成
  ↓
执行优化智能体
  ↓
当前批次专项测试
  ↓
分层历史回归测试
  ↓
回归影响分析智能体
  ↓
发布 / 阻断 / 继续优化 / 更新用例库
```

这里有两个核心变化：

1. 用例在进入优化批次前，先成为“候选资产”，经治理后才进入长期回归集。
2. 批次回归不再只跑 `batch.eval_case_ids`，而是跑一个可审计的 `regression_plan`。

## 4. 用例资产模型

### 4.1 权威存储

长期回归用例的权威存储为 SQLite 中的 `eval_cases` 及相关表。

选择 SQLite 主存的原因：

1. 当前系统已经围绕 `eval_cases` / `eval_runs` 建立 API、前端和回归运行链路。
2. 批次、反馈、运行结果、Agent 版本均在 SQLite 中有可关联 ID。
3. 资产治理需要过滤、排序、状态管理、修订历史和运行统计，数据库更适合作为主存。
4. 文件用例容易引入 DB 与文件双写漂移，不适合作为 v1 权威来源。

#### 4.1.1 一条反馈可沉淀多条用例

长期资产化后，同一条反馈可以生成多条回归用例，例如原始复现、变体、反例和干扰场景。因此：

1. `eval_cases.source_feedback_case_id` 只能作为来源关联字段，不能继续保持唯一约束。
2. 数据迁移必须显式删除 `ix_eval_cases_source_feedback_case_unique`，并把 ORM 元数据中的唯一索引改为普通索引。
3. 幂等去重改用 `source_feedback_case_id + variant_role + content_hash`，其中 `variant_role` 取值建议为 `original_reproduction`、`variant`、`counterexample`、`distractor`、`manual`。
4. `content_hash` 以规范化 JSON 计算，字段顺序必须稳定，至少包含 prompt、expected_behavior、checks_json、labels、asset metadata 和 source metadata。

### 4.2 用例层级

每条 eval case 增加 `asset_layer`，用于表达它在长期回归资产库中的位置。

| 层级 | 含义 | 默认运行策略 |
| --- | --- | --- |
| `candidate` | 新生成、待治理的候选用例 | 不进入发布门禁 |
| `batch_specific` | 当前批次专项用例 | 当前批次必须跑 |
| `smoke` | 冒烟用例，数量少，覆盖基础能力 | 每个批次必须跑 |
| `core_regression` | 核心能力回归 | 每个非小改动批次必须跑 |
| `scenario_pack` | 场景包，如告警研判、漏洞查询、报告生成 | 相关批次选择性跑 |
| `safety` | 安全边界，如越权、危险操作、权限策略 | 安全相关改动必须跑，发布前必须跑 |
| `historical_bug` | 历史反馈沉淀的防复发用例 | 相关批次必须跑，高危项阻断 |
| `exploratory` | 边界探索、实验性用例 | 不默认阻断 |

### 4.3 晋级状态

每条用例增加 `promotion_status`：

| 状态 | 含义 |
| --- | --- |
| `candidate` | 刚生成，尚未审核 |
| `needs_review` | 治理智能体或规则认为需要人工确认 |
| `approved` | 已确认进入长期回归资产 |
| `rejected` | 不进入长期资产 |
| `superseded` | 被更好的用例替代 |
| `archived` | 已归档，不参与自动选择 |

治理建议中的 `decision=merge` 不作为 `promotion_status` 的独立终态。执行合并时，被合并用例写为 `promotion_status=superseded`，并记录 `superseded_by_eval_case_id` 指向保留用例。

默认策略：

```text
反馈生成的用例默认进入 candidate。
人工确认后才能进入 smoke/core_regression/scenario_pack/safety/historical_bug。
```

这样可以避免反馈用例无脑进入长期集，导致膨胀、重复、冲突和脆弱。

`status` 与 `promotion_status` 的权威边界必须分开：

| 字段 | 语义 | 选择逻辑中的作用 |
| --- | --- | --- |
| `status` | 记录生命周期，仅保留 `active`、`draft`、`archived` | `archived` 永不自动选入 |
| `promotion_status` | 长期资产治理状态 | 只有 `approved` 可进入长期 regression plan |

默认回填策略：

| 存量来源 | 回填 `asset_layer` | 回填 `promotion_status` | 回填 `blocking_policy` |
| --- | --- | --- | --- |
| 反馈来源且 `status=active` | `historical_bug` | `approved` | `blocking_if_relevant` |
| 批次手工来源且 `status=active` | `batch_specific` | `approved` | `blocking` |
| `status=draft` | `candidate` | `candidate` | `non_blocking` |
| `status=archived` | 保留可推断层级 | `archived` | `non_blocking` |

### 4.4 阻断策略

每条用例增加 `blocking_policy`：

| 策略 | 含义 |
| --- | --- |
| `blocking` | 失败必须阻断发布 |
| `blocking_if_relevant` | 当前批次命中相关性时失败阻断 |
| `review_required` | 失败后需要人工判断 |
| `non_blocking` | 失败只记录，不阻断 |
| `flaky_non_blocking` | 已标记 flaky，不直接阻断 |

推荐默认：

```text
safety -> blocking
smoke -> blocking
core_regression -> blocking
historical_bug(high) -> blocking_if_relevant
scenario_pack -> blocking_if_relevant
batch_specific -> blocking
exploratory -> non_blocking
candidate -> non_blocking
```

`flaky_non_blocking` 不是安全后门。`safety` 与 `historical_bug(high)` 不允许仅因 flaky 标记自动降级为放行；它们失败时至少进入 `review_required`，需要带审计的人工确认或 break-glass 才能继续。

### 4.5 防复发目标

每条长期用例必须记录 `regression_goal`。

示例：

```json
{
  "eval_case_id": "evc-...",
  "asset_layer": "historical_bug",
  "source_feedback_case_id": "fbc-...",
  "origin_batch_id": "fob-...",
  "scenario_pack": "alert_triage",
  "bug_type": "missing_evidence_chain",
  "regression_goal": "防止 Agent 在横向移动分析中遗漏进程链、证据来源和置信度。",
  "severity": "high",
  "blocking_policy": "blocking_if_relevant",
  "promotion_status": "approved"
}
```

回归失败时，系统应该展示：

```text
历史问题复发：横向移动分析遗漏证据来源和置信度。
来源反馈：fbc-...
首次修复批次：fob-...
当前失败批次：fob-...
```

而不是只展示“某条测试 failed”。

## 5. 用例修订与历史追溯

当前实现允许直接覆盖 `prompt`、`expected_behavior`、`checks_json`、`labels`、`status`。长期资产化后，直接覆盖会带来审计问题：历史 eval_run 看到的是旧内容还是新内容不明确。

因此需要引入 revision。

### 5.1 修订规则

以下字段变更必须创建 revision：

```text
prompt
expected_behavior
checks_json
labels
asset_layer
blocking_policy
scenario_pack
regression_goal
severity
promotion_status
```

`eval_cases` 保存当前版本字段，`eval_case_revisions` 保存历史快照。

### 5.2 建议新增表

```text
eval_case_revisions
  revision_id
  eval_case_id
  revision_number
  created_at
  created_by
  change_reason
  content_hash
  snapshot_json
```

`snapshot_json` 至少包含：

```text
prompt
expected_behavior
checks_json
labels
asset metadata
source metadata
```

落地规则：

1. 创建 eval case 时必须在同一事务内创建首条 revision，并回填 `eval_cases.current_revision_id`。
2. `created_by` 来自治理写操作的 operator；v1 可先使用 `X-Operator` 请求头或等价内部操作者字段，但不得写死为 `system`。
3. `change_reason` 是编辑、晋级、归档、标记 flaky、取消 flaky、break-glass 等写操作的必填原因。
4. `content_hash` 只针对 snapshot 的规范化内容计算，不包含 revision_id、created_at 等运行时字段。
5. 所有改内容或资产元数据的写操作必须经同一个 revision 服务入口；`PATCH`、promote、archive、mark-flaky 只能复用该入口，不能旁路直写列或 payload。

### 5.3 运行结果快照

`eval_run_items.payload_json` 应固化：

```text
eval_case_revision_id
eval_case_snapshot
regression_goal
asset_layer
blocking_policy
scenario_pack
```

回归运行启动后，每条 item 必须引用当时的 `eval_case_revision_id`，并复制该 revision 的 snapshot。这样后续编辑用例不会改变历史运行结果的解释依据。

这样即使用例后续被编辑，历史运行也能还原当时判断依据。

## 6. 回归计划与门禁

### 6.1 回归计划

批次回归前生成 `regression_plan`，不再只使用批次创建时的 `eval_case_ids`。

`regression_plan` 包含：

```text
plan_id
batch_id
created_at
selection_reason
case_groups
required_case_ids
optional_case_ids
blocked_case_ids
gate_policy
applied_agent_version_id
selection_fingerprint
max_cases
max_concurrency
per_case_timeout_seconds
max_budget_usd
```

`selection_fingerprint` 由 batch_id、applied_agent_version_id、选择策略版本、改动分类和候选用例集合计算。相同指纹的 plan 必须复用，避免重复点击产生多个不一致 plan。

用例分组建议：

```text
current_batch_cases
smoke_cases
core_cases
safety_cases
relevant_historical_cases
scenario_pack_cases
exploratory_cases
```

运行约束：

1. `regression_plan` 生成时冻结 `applied_agent_version_id`；回归运行期间不得切换基线版本。
2. 回归运行使用隔离 workspace 或只读执行策略，不能污染主 Agent 的工作目录。
3. 每个 case 必须有超时；整次 run 必须有预算、并发和最大用例数上限。
4. 批次回归不在单个 HTTP 请求中串行等待全部主 Agent 调用完成，而是创建后台 run job，由前端轮询状态。

### 6.2 选择策略

按批次改动影响选择回归层：

| 改动类型 | 必跑用例 |
| --- | --- |
| 小型文案或局部 skill 调整 | `smoke` + `batch_specific` + 相关 `historical_bug` |
| 主智能体指令改动 | `smoke` + `core_regression` + 相关 `historical_bug` |
| skill 改动 | `smoke` + 对应 `scenario_pack` + 相关 `historical_bug` |
| MCP 配置或工具描述改动 | `smoke` + 相关工具场景包 + `safety` 子集 |
| 权限、安全、外部调用策略改动 | `safety` + `core_regression` + 相关历史用例 |
| 发布前验证 | `smoke` + `core_regression` + `safety` + 相关场景包 + 高危历史用例 |

#### 6.2.1 改动分类与相关性判定

相关性不能依赖人工临场判断，也不能等回归影响分析智能体跑完后才决定选哪些用例。回归计划生成前必须先运行确定性的改动分类与相关性判定：

| 输入 | 判定规则 |
| --- | --- |
| execution plan 的 `target_paths` | 路径前缀映射到 `change_type`，例如 agent 指令、skill、MCP、权限、安全策略 |
| applied diff | 用于确认实际改动文件和目录，生成 `changed_path_groups` |
| eval case `scenario_pack` | 与 `change_type` 精确匹配时选入 |
| eval case `labels` | 与改动标签有交集时选入 |
| `severity=high` 的 `historical_bug` | 无法判定相关性时 fail-safe 选入 |
| `safety` / `smoke` | 按层级默认选入，不依赖相关性 |

该引擎的输出写入 `regression_plan.selection_summary_json`，至少包含 `change_type`、`matched_rules`、`selected_case_ids`、`excluded_case_ids` 和 `uncertain_case_ids`。

### 6.3 阻断规则

回归运行完成后生成 `gate_result`。

| 失败类型 | 是否阻断 |
| --- | --- |
| `safety` 用例失败 | 必须阻断 |
| `smoke` 用例失败 | 必须阻断 |
| 当前批次专项用例失败 | 必须阻断 |
| 高危历史问题复发 | 必须阻断 |
| 相关 `core_regression` 失败 | 默认阻断 |
| 相关 `scenario_pack` 失败 | 按 `blocking_policy` 判断 |
| `exploratory` 失败 | 不直接阻断 |
| 已标记 flaky 的普通用例失败 | 不直接阻断，但记录并进入治理 |
| `safety` 或高危 `historical_bug` 被标记 flaky 后失败 | 至少 `review_required`，不得自动放行 |
| 用例过时 | 由 impact analyzer 或人工治理分类；未完成分类前按既有阻断策略处理 |

`gate_result.status` 取值：

| 状态 | 含义 | 批次推进 |
| --- | --- | --- |
| `blocked` | 存在必须阻断失败 | 批次进入 blocked |
| `review_required` | 存在需要人工判断的失败或 flaky 高危用例 | 批次进入 needs_human_review |
| `passed_with_notes` | 只有 non-blocking 失败或探索失败 | 可继续，但必须展示说明 |
| `passed` | 没有失败 | 可继续 |

必须阻断类失败允许受控 break-glass，但必须记录 operator、reason、到期复核时间、before/after gate_result，并在批次详情中显式展示。

`gate_result` 示例：

```json
{
  "status": "blocked",
  "summary": {
    "total": 18,
    "passed": 15,
    "failed": 3,
    "blocking_failed": 2,
    "non_blocking_failed": 1
  },
  "blocking_failures": [
    {
      "eval_case_id": "evc-001",
      "asset_layer": "historical_bug",
      "regression_goal": "防止遗漏证据来源和置信度",
      "reason": "高危历史问题复发"
    }
  ],
  "non_blocking_failures": [
    {
      "eval_case_id": "evc-009",
      "asset_layer": "exploratory",
      "reason": "探索用例失败，不阻断"
    }
  ]
}
```

## 7. 新增智能体

### 7.1 用例治理智能体：`eval-case-governor`

职责：

1. 判断候选用例是否值得进入长期回归资产。
2. 检测与已有用例的重复、冲突和覆盖重叠。
3. 建议用例层级、场景包、严重度和阻断策略。
4. 判断用例是否稳定、是否可自动判分、是否需要人工审核。
5. 给出变体、反例、干扰用例建议，降低过拟合。

输入：

```text

source_feedback
feedback_case
attribution_summary
proposal_summary
similar_eval_cases
recent_eval_run_stats
```

输出模型建议：

```text
FeedbackEvalCaseGenerationOutput
```

关键字段：

```text
decision: promote | keep_candidate | reject | merge | needs_human_review
recommended_layer
recommended_scenario_pack
recommended_blocking_policy
severity
regression_goal
duplicate_of
conflicts
stability_risk
judgeability
recommended_variants
human_review_reason
```

治理智能体只给建议，不自动把用例放入长期阻断层。进入长期阻断层必须有人确认。

### 7.2 回归影响分析智能体：`regression-impact-analyzer`

职责：

1. 分析批次回归失败是否与本批次改动相关。
2. 判断失败是真实退化、用例过时、判分规则不合理还是 flaky。
3. 识别历史反馈问题是否复发。
4. 给出发布建议：阻断、重跑、修复、更新用例、回滚。
5. 产出可展示给开发人员的失败归因摘要。

输入：

```text

optimization_plan
execution_result
regression_plan
eval_run
failed_eval_run_items
eval_case_snapshots
agent_version_diff
```

输出 schema 建议：

```text
RegressionImpactAnalysisOutput
```

关键字段：

```text
decision: block_release | allow_with_notes | rerun | update_cases | rollback_or_fix
failure_classifications
related_to_batch_change
historical_regression_recurred
case_update_recommendations
batch_fix_recommendations
release_note
```

`regression-impact-analyzer` 的结果不替代确定性 gate。它用于解释和建议：

```text

impact_analysis 是解释、归因和后续行动建议。
```

它也不能作为回归计划选例的前置来源；选例只依赖第 6.2.1 节的确定性改动分类与相关性判定。

### 10.3 迁移机制

当前 `runtime_db.py` 主要使用 `Base.metadata.create_all(engine)` 和 `schema_migrations` 记录初始版本。`create_all` 不会为已有表自动添加新列。

因此本次真正落地前必须补一层轻量迁移机制：

1. 在 `schema_migrations` 中登记每个迁移版本。
2. 启动时按版本执行幂等迁移；执行顺序必须是 `create_all` 建基础表后，再运行手写迁移补已有表。
3. 加列用 `ALTER TABLE ADD COLUMN`，先通过 `PRAGMA table_info` 检测。
4. 去除 `source_feedback_case_id` 唯一约束必须同步删除 ORM 唯一 Index 声明，并执行 `DROP INDEX IF EXISTS ix_eval_cases_source_feedback_case_unique` 后重建普通索引。
5. 修改约束或重建表时必须处理 SQLite 外键，明确 `PRAGMA foreign_keys` 策略、copy/drop/rename 顺序和失败回滚。
6. 每个迁移必须可重复检测，不依赖外部服务。
7. 测试覆盖旧库升级、重复运行迁移、唯一约束拆除后同一 feedback_case 多变体入库。

这是本项目需要先改造的基础设施之一。

## 15. 分阶段落地计划

### 阶段 0：方案收口与产品边界确认

目标：

1. 采纳评审报告中的阻断项，修订方案为可实施版本。
2. 明确“回归资产”是反馈工作台侧边栏菜单。
3. 将 UI ASCII 草图纳入方案，作为前端实现验收依据。

验收：

```text
方案不再把反馈工作台侧边栏误写成 App 顶层导航。
数据模型、门禁、运行、安全和 API 阻断项都有明确落点。
```

### 阶段 1：数据模型与修订基础

目标：

1. 拆除 `source_feedback_case_id` 唯一约束，支持一条反馈多用例。
2. 增加 revision 表和写入逻辑。
3. `eval_run_item` 固化 eval case revision 和 snapshot。
4. 补 SQLite 迁移机制和旧数据回填。
5. 收口 payload_json 与显式列投影写入。

验收：

```text
编辑长期用例会创建 revision。
历史 eval_run_item 能还原当时用例内容。
旧数据库启动后能自动迁移到新 schema。
同一 feedback_case 能创建 original/variant/counterexample 多条用例。
```

### 阶段 2：回归资产页

目标：

1. 新增反馈工作台侧边栏“回归资产”菜单。
2. 支持过滤、查看、编辑、归档、标记 flaky。
3. 支持查看修订历史和历史运行结果。
4. 页面布局符合第 8.2 和第 14.6 节 ASCII 草图。

验收：

```text
开发人员可以不进入具体批次，也能统一治理长期回归集。
批次详情只保留 batch_specific 新建和发布门禁运行入口，长期资产治理跳转到回归资产页。
```

### 阶段 3：用例治理智能体

目标：

1. 新增 `eval-case-governor` profile。
2. 新增 governance review API。
3. 支持人工确认晋级。
4. 支持重复、冲突、分层、阻断策略建议。

验收：

```text
反馈生成用例默认进入 candidate。
人工确认后进入长期回归资产层。
```

### 阶段 4a：改动分类与相关性选例引擎

目标：

1. 基于 target_paths、applied diff、labels、scenario_pack 生成 change_type。
2. 将 `selected_case_ids`、`uncertain_case_ids`、`matched_rules` 写入 regression plan。
3. 无法判定时 fail-safe 选入高危历史用例。

验收：

```text
每批能自动选入相关历史用例，且选择原因可审计。
```

### 阶段 4：回归计划与门禁

阶段 4 必须在阶段 4a 的改动分类与相关性选例引擎完成后实施。

目标：

1. 新增 regression plan 生成逻辑。
2. 批次回归按 plan 执行。
3. 生成 gate_result。
4. 批次状态根据门禁结果推进或阻断。

验收：

```text
后续批次会自动选入相关历史用例。
blocking 用例失败会阻断批次完成。
```

### 阶段 5：回归影响分析智能体

目标：

1. 新增 `regression-impact-analyzer` profile。
2. 分析失败类型和批次相关性。
3. 给出发布、修复、用例更新建议。
4. 在批次详情展示结论。

验收：

```text
回归失败不只显示 failed，还能说明是否历史问题复发、是否本批次引入、下一步该修复还是更新用例。
```

### 阶段 6：分层策略优化

目标：

1. 完善 scenario pack。
2. 引入失败率、运行耗时、最近运行时间统计。
3. 优化每批选择策略。
4. 支持发布前近全量回归。
5. 引入后台 job、轮询、并发上限、预算和 per-case timeout。

验收：

```text
回归集增长后仍可控，不因全量用例膨胀拖慢每个小批次。
发布前近全量回归不会阻塞单个 HTTP 请求，也不会污染主 Agent workspace。
```

## 18. 风险与约束

### 18.1 用例膨胀

风险：所有反馈默认进入长期回归集会导致用例数量膨胀。

约束：

```text
默认 candidate，人工确认后晋级。
```

### 18.2 用例过拟合

风险：Agent 只学会通过固定样本。

约束：

```text
高价值反馈必须考虑变体、反例和干扰样本。
```

### 18.3 双写漂移

风险：DB 与 `evals/` 文件同时作为权威来源会漂移。

约束：

```text
SQLite 为权威；文件只作为 workspace 资产或导出视图。
```

### 18.4 发布门禁等待增强分析

风险：实现时把 impact analyzer 的增强结论误接成发布门禁前置，导致批次推进被分析 job 阻塞。

约束：

```text
发布门禁只读取回归运行同步生成的 gate_result，impact analyzer 只提供解释和治理建议。
```

### 18.5 编排代码重复

风险：新增两个智能体后继续复制 job 编排模板。

约束：

```text
新增智能体前先抽象 job spec。
```

### 18.6 回归运行成本和基线污染

风险：长期资产增多后，单批回归成本、时延和主 workspace 污染不可控。

约束：

```text
后台 job + per-case timeout + 并发上限 + 预算上限 + 冻结 agent_version + 隔离 workspace。
```

### 18.7 高权治理操作缺审计

风险：任意持 key 者可晋级、归档或豁免高危用例，改变门禁结果且无法追责。

约束：

```text
所有治理写操作必须记录 operator、reason、before/after；break-glass 必须带到期复核。
```

### 18.8 历史记录无限增长

风险：revision、eval_run_item snapshot 和 impact analysis 长期累积导致 SQLite 膨胀。

约束：

```text
定义保留窗口、归档策略、content_hash 去重和 SQLite 体量阈值。
```

## 19. 最终验收标准

升级完成后，应满足：

1. 反馈生成的测试用例不会只服务当前批次。
2. 高价值反馈可以人工确认晋级为长期回归资产。
3. 每个后续优化批次能自动选择相关历史用例。
4. 批次回归有明确 `gate_result`。
5. 历史高危问题复发会阻断发布。
6. flaky、探索和过时用例不会无条件阻断，但会进入治理。
7. 开发人员能在“回归资产”页统一管理长期用例库。
8. 历史运行结果能追溯当时的用例 revision。
9. 新增智能体符合现有 profile 隔离和运行边界。
10. 回归资产越积累，后续优化越不容易破坏已修复能力。
11. 回归运行可通过后台 job 执行，具备超时、预算、并发和版本冻结。
12. 晋级、归档、flaky、revision 和 break-glass 全部可审计。
