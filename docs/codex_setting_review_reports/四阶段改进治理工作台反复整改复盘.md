# 四阶段改进治理工作台反复整改复盘

> 日期：2026-07-08
> 角色：Codex/Claude 配置治理复盘。本文记录反复整改暴露出的工程治理失效点，不替代 `docs/AgentGov_四阶段改进治理工作台UI整改方案.md` 的产品与 UI 权威。

## 一句话结论

近期问题不是一组孤立 UI 文案错误，而是同一类系统性失效：业务产物所有权、阶段信息归属、按钮动作副作用、运行环境边界和负向验收没有在执行前被固定，导致每次修复都容易只处理当前可见症状。

## 证据链

| 用户症状 | 失效类型 | 根因 | 应沉淀动作 |
| --- | --- | --- | --- |
| “自动执行优化”文案、重复“重新生成优化方案”入口 | surface-mismatch + execution-gap | 决策卡主动作和阶段卡浏览动作没有明确分层 | 阶段卡只展示业务产物，决策卡承载主动作 |
| A 优化方案面板混入变更项和执行记录 | surface-mismatch | UI 容器缺少业务产物归属检查 | 每个 tab/card/drawer 只承载一个主业务产物 |
| Diff 抽屉提示无文件级 diff | validation-gap + execution-gap | fallback/no-action 没作为产品态显式呈现，用户感知像成功执行 | 执行前检查 plan 状态、执行来源、change_set 和 applied_diff 绑定 |
| Langfuse 链接使用容器内 host/port | env-drift | 没区分 container service URL 和 browser-visible URL | 浏览器链接必须使用可访问地址并保留具体 trace URI |
| 测试用例输入展示“复现场景”描述 | surface-mismatch + validation-gap | backend-owned 原始输入被 Agent/heuristic 生成文本替代 | 原始用户输入由后端权威来源投影，LLM 只能生成业务语义字段 |

## 第一性原理

1. UI 不是信息堆叠容器，而是业务产物投影；不属于当前产物的信息应移到对应 card、drawer 或详情视图。
2. 按钮表达用户主动业务动作；状态推进只能作为副作用，不能成为主动作。
3. backend-owned 字段来自 DB row、route 参数、job input、store context 或确定性 projection；可以给 Agent 做输入上下文，但不能让 Agent 输出后再作为权威值。
4. fallback/no-action 是一等产品状态；不能用成功文案包装没有产生候选版本、变更集、文件 diff 或测试资产的结果。
5. 验收必须包含负向断言：不重复、不混入、不伪造、不泄漏内部地址、不把 fallback 包装成成功。

## 避坑方法论

改四阶段改进治理工作台、反馈闭环 UI、Agent job 产物或 Trace/Diff/测试用例详情前，先完成五行预检：

```text
1. 用户动作/对象：用户点击什么，期望看到哪个业务产物。
2. 证据链：UI state -> API response -> agent_jobs/store -> formatter output -> persisted payload 卡在哪层。
3. 字段所有权：backend-owned、agent-owned、boundary-owned 分别是什么。
4. 容器归属：决策卡、tab、card、drawer 各自承载什么；哪些内容明确不得出现。
5. 验证绑定：目标 pytest/UI verification nodeid 是什么，必须补哪些负向断言。
```
