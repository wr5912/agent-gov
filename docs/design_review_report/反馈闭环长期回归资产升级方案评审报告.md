## 1. 一句话结论

> **方案方向正确、对现状理解准确（16/17 条现状断言经源码核对属实），但尚不可直接进入实施——存在 12 个阻断项。** 其中最关键的是：① `eval_cases.source_feedback_case_id` 上的**唯一索引**与方案核心卖点「一条反馈派生变体/反例/干扰多条用例」**直接冲突**，且全程未排期拆除；② 门禁所依赖的「改动相关性 / 用例选择」判定引擎**全文无定义、无实现阶段承载**，使「每批自动选相关历史用例」「不依赖人工临场判断」无法落地；③ 新增 `promotion_status` 与现有 runner 的 `status=='active'` 选例逻辑**未打通**，晋级资产可能被静默漏跑或新字段沦为死字段；④ 近全量回归的**时延/成本/并发/基线隔离**完全未定义，而每条用例都要真实串行调用一次主 Agent。建议按本报告第 3 节先关闭 12 个阻断项再进入分阶段实施。

## 4. 数据模型与迁移（DM）

> 范围：方案 §4、§5、§10；`app/runtime/runtime_db.py`、`app/runtime/stores/feedback_eval_store.py`

#### [DM-1] 🟧 高 · B 内部矛盾/致命冲突 — `source_feedback_case_id` 唯一索引与「一反馈生成多条用例」直接冲突，全程未提拆约束与迁移
- **位置**：方案 §4.5/§13.2/§3；`runtime_db.py:312`、`feedback_eval_store.py:31,75-91`、`feedback_source_store.py:467`、`feedback_batch_store.py:43`
- **证据**：`runtime_db.py:312` `Index("ix_eval_cases_source_feedback_case_unique", EvalCaseModel.source_feedback_case_id, unique=True)` 强制一个 feedback_case 最多对应一条 eval_case（已实测 SQLite：同值二次插入抛 `UNIQUE constraint failed`）。方案 §13.2（行 869-887）要求对高价值反馈生成「原始复现/变体/反例/干扰」多条用例，§4.5 示例每条都带 `source_feedback_case_id`。三条生成路径都先 `find_eval_case(source_feedback_case_id=...)` 判「已存在则复用」，依赖该 1:1 不变量。方案 §5/§10/§17 全文无 `DROP INDEX`、无放宽约束、无迁移。
- **影响**：治理智能体一旦对同一反馈生成第二条用例，写库即抛 IntegrityError 致整条生成/同步事务回滚（`feedback_eval_store.py:40`、`feedback_batch_store.py:87`），批次创建/用例同步直接失败。方案核心卖点（防过拟合的多用例沉淀）无法落地。
- **建议**：阶段1 迁移中：① 从模型移除 `runtime_db.py:312` 的 `unique=True`（改普通 Index，否则 create_all 会按元数据重建唯一索引）；② 迁移脚本显式 `DROP INDEX IF EXISTS ix_eval_cases_source_feedback_case_unique` 再建非唯一版本；③ 把三处「唯一复用」语义改为基于 `(source_feedback_case_id, variant_role/content_hash)` 去重或返回列表；④ 为 `source_feedback_case_id + variant` 维度补复合唯一约束以保留幂等。

#### [DM-3] 🟧 高 · C 缺口 — §10.3 未定义既有行回填：新列对历史 `eval_cases` 的默认值与回填策略缺失
- **位置**：方案 §10.1（行 663-680）/§10.3/§16.1(8)；`runtime_db.py:299-309`、`feedback_eval_store.py:515-524`
- **证据**：§10.1 为 eval_cases 增加 asset_layer/promotion_status/severity/blocking_policy/scenario_pack/flaky_status/current_revision_id/last_run_at/last_result_status/failure_rate 十列。§10.3 只说「测试覆盖从旧库升级到新 schema 的路径」，§16.1(8) 只要求「旧库迁移后仍可读取原有 eval cases」，均未规定既有行新列填什么。历史用例 payload_json（构造函数 `feedback_eval_store.py:352-374/422-460`）不含这些键。§6.2/§6.3 的选择与门禁规则强依赖 asset_layer/blocking_policy。
- **影响**：迁移后存量用例 asset_layer/promotion_status 为 NULL，§6.2/§6.3 对 NULL 层级无定义行为——可能被全部当作 candidate（不进门禁）静默丢失回归覆盖，或门禁计算 KeyError；failure_rate/last_run_at 为 NULL 时排序/筛选未定义。
- **建议**：§10.3 增加回填步骤：ALTER ADD COLUMN 给 `NOT NULL DEFAULT`（如 asset_layer 默认 `historical_bug`、promotion_status 默认 `approved` 以保留既有用例现役地位、blocking_policy 默认 `non_blocking`），对存量行一次性 UPDATE 回填，明确 NULL 在选择/门禁中的兜底分类，并在 §16.1 加断言。

#### [DM-2] 🟨 中 · C 风险/欠规范 — §10.3 迁移机制未覆盖 SQLite DDL 限制，ORM 元数据与手写迁移未协调
- **位置**：方案 §10.3（行 720-731）；`runtime_db.py:312,381-386`
- **证据**：§10.3 仅写「按版本执行幂等 ALTER TABLE / CREATE TABLE」，全文无 DROP INDEX / 整表重建 / `PRAGMA foreign_keys` / `pragma table_info` / 迁移与 create_all 执行顺序 的任何表述。`ensure_schema` 只 `create_all + 登记 0001`；唯一索引在模型元数据声明，若仅在迁移里 DROP 而不删 `line312` 声明，重启后 create_all 会重建「幽灵约束」。SQLite ALTER 不能改约束/加唯一约束（需整表重建），涉及 FK（`eval_run_items.eval_case_id → eval_cases`，`runtime_db.py:333`）时重建需处理 `PRAGMA foreign_keys`。
- **影响**：按方案描述实现的迁移会出现「删了又被建回」的幽灵约束，使 DM-1 的拆约束在重启后失效；或开发者误以为 ALTER 能改约束写出无效迁移。
- **建议**：§10.3 按操作类型给规范：「加列」用 `ALTER ADD COLUMN`（先查 `pragma table_info` 幂等，SQLite 直接支持）；「去/改唯一索引」用 DROP INDEX 并**同步移除模型元数据 Index 声明**；「改约束/加唯一约束」用整表重建（create+copy+drop+rename，处理 FK）；明确 create_all 与手写迁移的执行顺序；重建/索引迁移在 WAL+busy_timeout 下加测试。

#### [DM-4] 🟨 中 · C 设计缺口 — 新增 10 个显式列与 payload_json 双存储缺单一写入收口，叠加现状旁路双写可致筛选/门禁与详情/快照漂移
- **位置**：方案 §10.1（行 680）/§18.3；`feedback_eval_store.py:93-130,376-390,515-524`
- **证据**：§10.1 行 680 要求 payload_json 继续保存完整字段，与 10 个新列并存。现状已有「双写+列优先」先例：`update_eval_case`（118-125）把 status/labels 同时写 payload 与 row 列，`_eval_case_to_dict`（515-524）读时列覆盖 payload。已有收口 `_update_eval_case_row`（376-390）从 payload 派生列，但 `update_eval_case` **旁路它**单独双写。§18.3 的「双写漂移」仅指 DB-vs-文件，未覆盖**行内列 vs payload_json**的投影一致性。
- **影响**：新增 9 个过滤列后，治理/晋级/flaky/回填多条写路径若未同步两处，列与 payload_json 漂移：筛选/门禁按列走、详情/revision snapshot 按 payload_json 走，同一用例表现不一致，且 revision 快照可能固化错误值，破坏 §5 审计可信度。
- **建议**：§10 明确所有 eval_case 写操作统一收口到「payload 为唯一真相源、列为其投影」模式（禁止 `update_eval_case` 旁路逐列双写）；读取侧文档化（列仅用于过滤/排序，详情与 snapshot 一律以 payload_json 为准）；加测试断言「写后每个显式列 == payload_json 对应键」。

#### [DM-5] 🟨 中 · C 缺口 — revision 模型的先有鸡先有蛋、content_hash 口径、snapshot 覆盖面与 eval_run_items 固化均未定义清楚
- **位置**：方案 §5.1/§5.2/§5.3（行 247-302）/§10.1；`runtime_db.py:328-337`、`feedback_eval_store.py:204-251`、`feedback_store.py:216`
- **证据**：`current_revision_id` 仅出现于方案行 674、无初始化时序；`content_hash` 仅出现于行 274、无计算口径（仓库已有可复用的规范化 hash 助手 `feedback_store.py:216` `_sha256_json`，用 `sort_keys=True`）；§5.1 触发字段全集未被 §5.2 `snapshot_json`（行 281-286）覆盖；`EvalRunItemModel`/`append_eval_run_item`（`feedback_eval_store.py:204`）payload 不含 revision 引用。
- **影响**：current_revision_id 与 revision 互指若不在同事务一致写入，可能悬挂引用或首版用例无 revision；content_hash 口径不定会令判重/判变更不可靠（同义不同序 JSON 生成不同 hash）；snapshot 覆盖不全则「还原当时判断依据」不成立——门禁所依据的 blocking_policy/severity 若不在快照里，历史运行无法复盘为何阻断。
- **建议**：① 创建用例时同事务写首条 revision(number=1) 并回填 `current_revision_id`；② `content_hash = sha256(规范化排序后的 snapshot 子集)`，复用 `feedback_store.py:216` 口径并说明用途；③ 令 §5.2 snapshot 字段集合成为 §5.1 触发字段全集的超集（逐字段列明）；④ `append_eval_run_item`（`feedback_eval_store.py:204`）增 `eval_case_revision_id` 参数，由 `feedback_eval_runner.py:65` 传入运行时所用 revision。

#### [DM-6] 🟨 中 · C 缺口 — `failure_rate/last_run_at/last_result_status` 派生统计列的更新责任、时机与写放大未定义
- **位置**：方案 §10.1（行 674-677）/§8.2/§15 阶段6；`feedback_eval_store.py:204-280`
- **证据**：§10.1 增三列、§8.2 用其筛选/展示，但谁/何时更新、一致性级别、写放大均未说明。现状 `finish_eval_run`（253-280）与 `append_eval_run_item`（204-251）都**不回写** EvalCaseModel；要维护 failure_rate 需每次运行后对命中的每条 eval_case 做一次 UPDATE。建列在 §15 阶段1，写值却在阶段6（行 1036），跨阶段错配。
- **影响**：「写时更新」在 WAL 单写者 SQLite（`runtime_db.py:367`）下放大写竞争并与 DM-4 双写叠加；「读时计算」又与「加显式列避免 JSON 扫描」初衷矛盾。方案未选定，落地时统计列易长期 NULL 或滞后。
- **建议**：派生列更新归属 `finish_eval_run` 收口，在 run 完成同一事务内批量 UPDATE，failure_rate 按滚动窗口/运行次数定义口径；或显式声明为「物化缓存列、可懒重建、接受短暂滞后」；把建列与写值放到同一阶段。

#### [DM-7] 🟨 中 · C 缺口 — §10.2 新表 `regression_plans/regression_impact_analyses` 缺检索字段索引
- **位置**：方案 §10.2（行 682-718）；`runtime_db.py:191,332-333`（既有索引风格）
- **证据**：`regression_plans.batch_id`、`regression_impact_analyses.eval_run_id/batch_id` 均为 §9.2/§9.3 GET 查询键，但方案全文未声明索引（grep `index/索引/Index` 零命中）。现状对同类列一律建索引（`runtime_db.py:332-333` eval_run_id/eval_case_id `index=True`）。
- **影响**：不建索引时按批次/运行查 plan 与 impact 退化为全表扫描，随回归资产累积变慢。
- **建议**：§10.2 为上述列显式 `index=True`（对齐 `runtime_db.py:332-333`），可补 status/created_at 索引。`*_case_ids_json` 列若会同时进 payload_json 则明确单一真相源（建议只用独立列）。

## 5. 门禁与资产模型（GATE）

> 范围：方案 §4、§6、§9.3；自洽性审查为主，并对照 `feedback_eval_runner.py` 的判分/状态语义

#### [GATE-2] 🟧 高 · C 关键缺口 — §6.2「选择层」以「改动类型/相关性」为枢轴却未定义判定来源与算法，阻断阶段4自动选例验收
- **位置**：方案 §4.2/§4.4、§6.1 case_groups（行 333）、§6.2 选择策略表（行 344-349）、§6.3
