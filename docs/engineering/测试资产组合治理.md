# 测试资产组合治理

## 目标与边界

测试治理的目标不是减少测试数量，而是用更低的执行与维护成本持续获得可信、及时、可定位的质量反馈。`tests/quality_policy.json` 是机器可读的单一入口；本文解释稳定工程契约，不复制具体 nodeid 清单。

覆盖率只证明代码被执行过，不单独证明缺陷检测能力。组合决策同时考虑：

- **Speed**：PR 与主干反馈的 p50、p95。
- **Maintainability**：重复断言、实现细节耦合和 owner 缺失。
- **Utilization**：runner 时间、CPU 分钟、容器和模型资源。
- **Reliability**：flaky、重跑后通过、隔离时长和证据完整性。
- **Fidelity**：组件、契约、浏览器、真实容器和 live 模型的生产接近度。

## 资产模型

每个 pytest 叶子 nodeid 必须且只能得到一个有效分类。策略用文件 glob 建立基础分类，参数化用例在真实 pytest collection 后展开；规则零匹配、重叠或漏分类均阻断治理门。

分类同时包含：

| 维度 | 值 |
| --- | --- |
| 执行层级 | static、unit、component、contract、integration、e2e、performance、security、resilience |
| 保护目的 | requirement-acceptance、defect-regression、security-boundary、compatibility-migration、data-integrity、architecture-contract、implementation-detail |
| 生命周期 | KEEP、PROMOTE、DEMOTE、REFACTOR、MERGE、QUARANTINE、DELETE-CANDIDATE |
| 运行约束 | owner、capabilities（可多值）、lanes、parallelism、非空 resource classes |

逻辑 owner 以业务域登记，联系人可以暂时相同，但责任边界不能退化为文件作者。当前域包括 runtime-platform、improvement-governance、agent-lifecycle、security-response、frontend-experience、engineering-governance 和 integrations。

`QUARANTINE` 必须有 owner、原因、issue、隔离日期、到期日和修复方案，最长 14 天。`DELETE-CANDIDATE` 至少需要两类证据，例如业务路径已删除且有更强替代测试。没有可执行测试的重要风险登记为 `GAP`，不能用永久 skip 或无断言测试伪装资产。

## 执行通道

| 通道 | 当前职责 | 门禁 |
| --- | --- | --- |
| governance-static | Codex/docs/OpenAPI/版本治理、类型检查、runtime seed 脱敏 | PR/push 阻塞 |
| main-full | 全量离线后端、coverage、JUnit 与证据包 | PR/push 阻塞 |
| frontend-ui | 生成类型、构建、Chromium 预检和六条主流程 UI 验收 | PR/push 阻塞 |
| quality-gate | 汇总三个阻塞通道并验证同次运行证据 | 稳定分支保护名称 |
| nightly-full | 固定 `-n0` 串行哨兵、长时和跨版本回归 | 定时 |
| xdist-shadow | n2/n4、load/worksteal 与串行结果配对 | 仅 shadow |
| pr-fast | Test Impact Analysis 选择结果与全量结果配对 | 仅 shadow |
| mutation | 高风险状态机的限时 mutation score | 每周/发布 |
| container-live-acceptance | 手工触发的真实 Compose、模型、Runtime 路径和 Responses UI 验收 | 受保护 CI environment；不是 Release 或自动部署 |

`make main-flow-test` 保持本地完整语义，同时执行绑定的后端与 UI。CI 中 `main-full` 已覆盖后端绑定，因此 `main-flow-ui-test` 只能作为组合式 UI 子通道使用，不能单独宣称主流程通过。

分支保护只要求稳定的 `quality-gate` job 名。新工作流在 `master` 连续 3 次绿色前不启用该远端锁；达到条件后由仓库管理员在 GitHub 分支保护中登记，不能用仓库内配置或一次本地通过冒充远端保护已生效。

## 可信证据

`scripts/run_test_lane.py` 在 `artifacts/test-quality/<lane>/` 生成：

- `junit.xml`：每个 testcase 带精确 `agentgov_nodeid` 属性；
- `coverage.json`：同一 pytest 进程产生的 coverage；
- `evidence.json`：提交 SHA、tracked dirty 状态、GitHub run/attempt/job、策略和依赖哈希、全局/选择集 digest、版本、耗时、worker 配置、逐叶子结果及 artifact 哈希。

校验器交叉解析 JUnit、coverage 和 evidence，拒绝旧 SHA、错误 run attempt、部分选择、额外测试、skipped/xfail/failed 叶子、内容篡改、符号链接和路径逃逸。诊断 artifact 无论成功失败都可上传；只有成功 lane 才上传 trusted evidence，最终 `quality-gate` 只接受同一 SHA、run 和 attempt 的 trusted evidence。

## Shadow 晋级

TIA 和 xdist 初始模式均为 `shadow`，不得直接缩小 PR 阻塞范围或替换串行全量。晋级至少同时满足：

1. 至少 20 组同 SHA 配对样本，且样本跨越至少 14 个日历日；
2. collection 与逐叶子 outcome 完全一致；coverage 的统计宇宙相同、两边独立通过门槛，line/branch 差值绝对值不超过 0.10 个百分点；
3. TIA 零漏测，xdist 零并行特有失败、资源泄漏或顺序污染；
4. xdist p50 至少提速 20%，CPU 分钟增幅不超过 50%；
5. 多个配置合格时选择 p95 最低者；差距不超过 5% 时选择 worker 更少者。

未知路径、依赖、CI、测试策略或共享治理配置变更必须 fail-closed 回退 `main-full`。合并后和夜间始终保留完整串行哨兵，选择性测试不能取代全量回归。

## Mutation 与指标

`make mutation-test` 读取同一策略中的目标、测试选择、时间预算和最低 score，并核对 `pyproject.toml` 的 mutmut 配置，防止两个入口漂移。当前只对高风险集中状态机运行，后续扩展必须先给出风险和成本依据。

持续评审至少关注：PR p50/p95、flaky 与重跑后通过率、隔离数量和平均时长、最慢 10% 耗时占比、失败到定位的时间、owner/capability 缺失率、重复资产组、真实缺陷发现比例、线上逃逸缺陷和高风险变更 mutation score。

## 常用命令

```bash
# 严格 schema、真实 collection、分类与主流程绑定
.venv/bin/python scripts/check_test_quality_policy.py --manifest-only

# 完整离线提交门
make test

# 产品主流程
make main-flow-test

# 每周或发布 mutation lane
make mutation-test

# 临时 Compose env + 临时 runtime 根的真实健康 E2E
make container-health-e2e COMPOSE_ENV_FILE=/path/to/ephemeral-compose.env
```

真实容器验收不得读取 `docker/.env.local-debug`，不得写 `${HOME}/volume-agent-gov`。CI 从 `docker/.env.example` 生成单一临时 env 文件，并由 E2E 脚本创建独立 Compose project、动态端口和临时 runtime 根，结束时清理容器与卷。
