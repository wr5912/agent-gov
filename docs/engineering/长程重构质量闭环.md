# 长程重构质量闭环

本文沉淀最近一轮 Pydantic 化、JsonObject 边界治理、store 投影层收口和前端真实浏览器验证中的经验。目标不是记录一次修复，而是给后续长程整改提供可复用的方法。

## 核心结论

长程重构的主要风险不是单个实现错误，而是边界漂移、验证滞后和治理规则没有落到机器检查。后续阶段必须先定义边界，再写代码；先确定验证矩阵，再声称完成；每发现一类可重复问题，都要优先转成测试、生成物校验或治理硬门。

## 最近问题的通用化

| 问题类型 | 表现 | 可复用处理方式 |
| --- | --- | --- |
| 边界命名误导 | `JsonObject` 放在 `records` 目录，被误解为 record 专属类型 | 中性基础类型放在中性目录，records 目录只放 record 模型 |
| 模型语义混用 | 投影模型继承持久化 row 模型，历史快照缺字段时触发 `500` | 持久化 record、投影 record、response schema 必须按语义拆开 |
| 验证数据不足 | 单测通过，但真实 `${HOME}/volume-agent-gov` 或迁移前 `docker/volume` 历史数据访问批次列表失败 | 涉及持久化投影时必须用真实历史数据跑 API 和 UI smoke |
| 前端生成物漂移 | 后端 schema 改动后，前端类型或 OpenAPI 可能滞后 | 后端公开契约变更必须跑 OpenAPI 导出和前端类型生成 |
| 治理靠记忆 | 口头要求不新增弱 `dict` 边界，后续仍可能复发 | 能机器化的规则进入治理脚本；不能机器化的进入 GSD checklist 和 rules |

## 四层沉淀

1. 文档层：本文解释为什么要这样做，以及哪些问题已经发生过。
2. GSD 方法层：`.planning/METHODOLOGY.md` 提供后续 discuss/plan 阶段应读取的方法论 lenses。
3. 项目指引层：唯一根 `AGENTS.md` 与 `.codex/guidance/` 约束后续计划、执行和验证；`.codex/rules/` 仅承载命令执行策略。
4. 机器硬门层：`scripts/check_codex_governance.py --mode fail`、`make test`、OpenAPI/type 生成和浏览器 smoke 负责拦截可判定回归。

## 长程重构阶段流程

### Analyze

- 明确这次是否属于替换旧设计模式。
- 列出所有边界：records、store、response schema、OpenAPI、frontend types、UI 使用、配置、Docker 路径、持久化数据。
- 判断每个边界的单一真相来源。不能把“现有代码就是这样”当作边界理由。
- 标出历史数据和旧快照：SQLite、`${HOME}/volume-agent-gov/data`、迁移前 `docker/volume/data`、agent job payload、OpenAPI 旧字段、前端 localStorage。
- 识别哪些问题可以沉淀为治理硬门，哪些只能先沉淀为人工 checklist。

### Plan

计划必须包含一张边界表：

| 边界 | 单一真相来源 | 允许的开放 JSON | 删除/迁移/保留 | 验证 |
| --- | --- | --- | --- | --- |
| 内部 records | Pydantic record | 仅真实开放字段，如 `raw_output_json` | 删除旧别名或旧入口 | record 单测、治理硬门 |
| store 投影 | typed projection record | 仅边界字段 | 不新增公共裸 `JsonObject` 返回 | 真实 SQLite 数据列表/详情 |
| API response | Pydantic response schema | 必须说明原因 | OpenAPI 与前端类型同步 | OpenAPI 导出、schema 测试 |
| 前端 UI | OpenAPI 派生类型 + UI 领域类型 | 仅 UI 局部状态 | 删除漂移手写字段覆盖 | build、浏览器 smoke |

计划还必须写明：

- 公开 API / OpenAPI / 前端生成类型是否保留、改名、删除或迁移。
- 配置、环境变量、Docker 路径和 `config/*` 是否受影响。
- 持久化数据、SQLite schema、`${HOME}/volume-agent-gov` 默认路径和 `docker/volume/` 历史路径如何验证。
- README、docs、测试和治理硬门如何验证新契约。
- 内部兼容 facade、shim、旧工作流分支、不可达 UI 和一次性调试脚本是否删除；如保留，必须给出期限或清理条件。

### Execute

- 不让投影模型继承持久化 row 模型，除非两者字段和生命周期约束完全一致。
- 不在 store 公共方法上新增弱 `JsonObject` 返回；新增返回形态应先定义 record 或 response。
- 不把 `dict[str, Any]`、`JsonObject` 当作“先临时用一下”的默认选择。
- 公开 response schema 和前端类型必须由后端契约驱动，不允许前端手写字段长期覆盖真实 API。
- 每次发现同类问题的第三个实例时，优先停下来抽出公共类型、集中注册表或治理检查。

### Verify

默认验证矩阵：

| 场景 | 命令或动作 | 成功标准 |
| --- | --- | --- |
| 治理硬门 | `.venv/bin/python scripts/check_codex_governance.py --mode fail` | 无 `FAIL` |
| 后端全量 | `make test` | pytest 全部通过，且包含治理硬门 |
| OpenAPI | `.venv/bin/python scripts/export_openapi.py` | 生成文件与代码契约一致 |
| 前端类型 | `pnpm --dir frontend generate:api-types` | `frontend/src/types/api.ts` 已同步 |
| 前端构建 | `pnpm --dir frontend build` | TypeScript 和 Vite build 通过 |
| 真实页面 | feedback browser smoke 或 Playwright 脚本 | console error、failed request、4xx/5xx API 计数为 0 |
| 真实数据 | 使用 `${HOME}/volume-agent-gov/data`、迁移前 `docker/volume/data` 或对应历史数据 | 列表、详情、投影和旧快照不触发 500 |

如果阶段涉及浏览器可见页面，不能只用 API 测试代替页面验证。如果阶段涉及持久化投影，不能只用新建临时数据代替历史数据。

## 停止条件

遇到以下情况必须停止并重新计划：

- 单测通过但真实历史数据失败。
- 生成类型与 UI 实际字段不一致。
- 为了通过 Pydantic 校验而放宽成大范围开放 JSON。
- 新增兼容 shim 但没有删除条件。
- 治理硬门无法覆盖已识别的重复问题，且计划没有补测试或人工 checklist。

## 沉淀到硬门的规则

阶段结束时，把发现的问题分类：

- 可静态识别：进入 `scripts/check_codex_governance.py`。
- 可行为验证：进入 pytest、OpenAPI 测试或 browser smoke。
- 只能人工判断：进入 `.planning/METHODOLOGY.md` 和 `docs/engineering/GSD长程重构阶段清单.md`。

只写在总结里的经验不算沉淀；至少要被一个后续流程读取或被一个命令验证。
