# 反馈优化闭环多 Agent 架构

> 用途：指导 Claude Agent Runtime / 网络安全运营 AI 助手的反馈优化闭环编码实现。
> 状态：当前实现校准稿
> 版本：v1.1
> 日期：2026-05-26

本版本按当前代码实现校准。文档中的“当前实现”描述的是本仓库已落地能力；标注为“预留”或“待实现”的内容不得作为已完成验收项。

## 0. 替代声明与 Breaking Changes

本文档完全替代早期反馈优化闭环设计文档，早期设计文档删除后不再作为开发、测试、接口或产品验收依据。

不兼容变更如下：

```text
1. 旧版 POST /api/feedback 立即返回 attribution/proposal 的行为废弃。
2. 旧版 POST /api/feedback/events 直接触发规则归因或 proposal 的行为废弃。
3. 旧版 label -> attribution_type -> proposal 的确定性规则链废弃，不作为正式归因来源。
4. 旧版 /data/feedback/*.jsonl 和 /data/optimization-proposals/*.jsonl 不再作为新版接口契约。
5. 旧版 volume/workspace 和 volume/claude-root 的主目录语义废弃，新版固定使用 main-workspace 和 claude-roots/*。
6. 旧版前端“提交反馈后直接展示归因和 proposal 摘要”的交互废弃。
7. 旧数据不做自动迁移；如需保留，只能作为人工参考，不参与新版闭环计算。
8. 基于旧版反馈优化实现产生的后端接口、规则归因代码、前端入口、测试和数据目录必须在新版落地时一并清理。
9. 当前权威运行态数据存储为 /data/runtime.sqlite3；/data/feedback-signals/ 等目录仅为兼容路径或历史说明，不再作为权威存储契约。
```

新版唯一闭环如下：

```text
chat run
  -> feedback signal / SOC event
  -> feedback case
  -> evidence package
  -> attribution job
  -> attribution output
  -> optimization batch
  -> proposal-generator 生成批次 optimization plan
  -> 开发人员点击具体任务执行
  -> optimization task
  -> execution-optimizer plan
  -> backend-controlled patch application
  -> pre/post execution snapshots
  -> main-workspace version
  -> batch regression evaluation
```

## 1. 文档目标

本文档用于统一反馈优化闭环多 Agent 架构的实现口径，避免在编码过程中出现目录命名、运行态隔离、版本管理、权限边界、数据流、API、前端展示等方面的歧义、矛盾和冲突。

本文档明确以下问题：

1. 多个 Agent 如何在同一个容器内运行。
2. 主智能体、归因分析智能体、优化方案生成智能体的职责边界。
3. 为什么不共享 workspace 和 claude-root。
4. 如何通过 Runtime Profile 指定不同的 `cwd`、`HOME`、`CLAUDE_CONFIG_DIR`。
5. 哪些目录进入版本管理，哪些目录只作为运行态。
6. 反馈、Trace、工具调用、SOC 操作如何固化为 evidence package。
7. 归因分析和优化方案如何形成结构化结果。
8. 哪些建议可以进入优化任务，哪些必须成为外部治理建议或人工分析项。
9. 前后端需要实现哪些 API、数据结构和验证逻辑。

---

## 2. 核心结论

最终采用以下架构：

```text
同一个 API 容器
  同一个 FastAPI Runtime 服务
    主智能体Claude Code 子进程
    归因分析智能体Claude Code 子进程
    优化方案生成智能体Claude Code 子进程
    执行优化智能体Claude Code 子进程
    用例治理智能体Claude Code 子进程
    回归影响分析智能体Claude Code 子进程
  同一个 agent job worker 服务
    从 /data/runtime.sqlite3 领取 queued agent_jobs
    调用对应 Runtime Profile
    校验并投影输出到领域表
```

不是：

```text
六个 Agent = 六个容器
```

而是：

```text
当前已实现六个智能体 = 六套 Runtime Profile；需要模型执行的反馈闭环任务统一先创建 `agent_jobs`，由 worker 异步执行。开发人员在优化方案中点击具体任务的“执行”即表示确认，随后 execution-optimizer 生成执行方案，后端完成校验并受控应用。
```

核心规则如下：

```text
1. 六个 Agent 可以运行在同一个镜像中。
2. API 进程只负责创建和查询 `agent_jobs`；worker 进程负责领取和执行队列任务。
3. 六个 Agent 不共享 workspace。
4. 六个 Agent 不共享 claude-root。
5. 六个 Agent 必须使用不同的 cwd。
6. 六个 Agent 必须使用不同的 HOME。
7. 六个 Agent 必须使用不同的 CLAUDE_CONFIG_DIR。
8. 六个 Agent 可以共享 /data，但必须按 job_id、case_id、agent role 分区。
9. 主智能体是被反馈、被优化、被版本管理的主要对象。
10. 归因分析智能体和优化方案生成智能体只输出结构化结果，不直接修改主智能体workspace。
11. execution-optimizer 只输出结构化执行方案；实际写文件由后端在任务执行动作后受控完成。
12. attribution、proposal、batch_plan、execution、eval_case_generation、regression_impact_analysis 的创建接口返回 `AgentJobResponse`；调用方通过 `GET /api/agent-jobs/{job_id}` 轮询。
```

## 4. Agent 角色和职责

### 4.1 主智能体：`main`

主智能体是面向最终用户的网络安全运营 AI 助手。

职责：

1. 处理用户聊天。
2. 进行告警研判、日志解释、攻击链分析、处置建议生成。
3. 调用安全运营数据 MCP、知识库 MCP、报告模板 MCP 等工具。
4. 产生可被反馈、追踪、复盘的 run/session/tool call 记录。
5. 作为反馈优化闭环的主要优化对象。

允许：

1. 读取主智能体workspace 内的指令、skills、MCP 配置。
2. 调用被授权的安全运营工具。
3. 生成回答、报告和分析结论。

不允许：

1. 自行修改自己的核心配置文件，除非通过审批后的优化流程。
2. 直接写归因分析智能体或优化方案生成智能体workspace。
3. 直接修改版本快照。

---

### 4.2 归因分析智能体：`attribution-analyzer`

归因分析智能体是反馈闭环中的“质量分析员 / 反馈归因员”。

职责：

1. 读取 feedback case。
2. 读取 evidence package。
3. 分析反馈背后的问题类型。
4. 判断责任边界：主智能体、工具/MCP、Runtime、外部 SOC 流程、数据质量、用户误解等。
5. 输出结构化 attribution output。

允许：

1. 读取归因任务自己的 job 目录。
2. 读取 evidence package。
3. 读取主智能体当前版本快照或只读副本。
4. 调用只读型 trace / feedback / SOC 查询工具。
5. 向 attribution output 路径写结果。

不允许：

1. 直接修改 `main-workspace`。
2. 直接修改主智能体claude-root。
3. 直接生成 optimization task。
4. 直接审批 proposal。
5. 把证据不足的问题包装成确定性结论。

---

### 4.3 优化方案生成智能体：`proposal-generator`

优化方案生成智能体是反馈闭环中的“优化方案设计师”。它面向优化批次工作，把多条归因结果统筹成开发人员可阅读、可执行、可派发的任务列表。

职责：

1. 读取批次 job 输入中已校验的 attribution outputs、eval cases、source refs 和 target_policy。
2. 判断每条归因应形成 workspace 可执行任务、外部系统 webhook 任务，还是 blocked item。
3. 输出足够完整的优化方案内容，由后端规范化为 `feedback-optimization-plan-output/v1`；其中 `tasks` 是可执行任务，`blocked_items` 是不能执行或不能派发的事项。
4. 在每个 task 内给出任务名称、任务描述、任务目标、目标对象、推荐改动、优化后结果的验收标准、验证方式和风险。
5. 对外部系统任务，在对应 task 的 `task_context` 中明确系统、MCP server、tool/API/endpoint、问题 ID、受影响字段或观察到的问题。

允许：

1. 使用后端嵌入的 batch plan input。
2. 输出结构化优化方案。
3. 根据 `regeneration_instruction` 调整本次方案侧重点。

不允许：

1. 直接修改 `main-workspace`。
2. 直接修改 `.mcp.json`、`CLAUDE.md`、skills 等目标文件。
3. 直接审批或执行优化任务。
4. 直接创建新 Agent 版本。
5. 将外部 MCP、SOC 系统、Runtime bug 问题伪装成主智能体skill 修改。
6. 把 `task_context` 作为顶层对象输出；它必须属于具体 task。

---

### 4.4 执行优化智能体：`execution-optimizer`

当前实现已提供 `execution-optimizer` profile、workspace、API 和任务执行后的受控 patch 应用入口。它不是静默自动改文件的智能体：智能体只生成结构化执行方案，后端负责路径、版本、hash 和操作类型校验，并在开发人员点击具体优化任务“执行”后写入主智能体workspace。

执行优化的目标文件范围与主智能体版本快照一致：支持 `main-workspace` 中的受管相对路径，并排除 `.git`、`node_modules`、`dist`、缓存、虚拟环境和编译产物。`execution-input/v1` 会为每个目标文件提供 `target_file_contexts`，包含当前存在状态、文件类型、大小、sha256 和 UTF-8 文本内容；二进制、目录、symlink 或超大文件不会自动应用，必须进入人工复核。

当前优化执行方式是：

1. 批次优化方案中的 workspace task 创建 `optimization task`。
2. `execution-optimizer` 为 optimization task 生成执行方案。
3. 后端校验执行方案的路径、版本、hash 和操作类型。
4. 校验通过后，后端创建执行前快照、应用受控 patch、创建执行后快照。
5. 任务记录 applied_agent_version_id、pre_execution_agent_version_id、execution_job；应用结果、applied_diff 和补偿状态写入 execution_applications。
6. 开发人员手动触发回归评估。

当前受控执行模式必须满足：

```text
1. proposal 状态为 approved。
2. target_path 在允许修改范围内。
3. 当前主智能体版本与 task baseline 一致；不一致时阻止应用。
4. 执行前已创建快照。
5. Agent 不直接写 main-workspace，只输出 operations。
6. 后端只支持受控文本操作，生成 diff、生成新主智能体版本并提供回滚点。
7. 回归验证仍由用户手动触发。
```

```text

6. runtime.sqlite3 是 feedback signal、SOC event、case、evidence package、job、proposal、task、eval 和 session 的权威存储。
7. .runtime-tmp/jobs 是 Agent job 临时 workspace，不是 Agent workspace，也不是长期事实源。
```

## 5.1 代码模块边界

当前后端模块边界遵循：

```text
app/routers/        HTTP 参数、响应模型和薄控制器。
app/services/       跨 store/runtime/profile 的应用服务编排。
app/runtime/        Claude SDK 适配、profile、schema、版本、策略和运行时支撑。
app/runtime/stores/ 反馈闭环 SQLite store facade 与领域 mixin。
app/runtime/response_schemas/ HTTP response schema。
app/runtime/records/          内部 Pydantic record。
app/runtime/integrations/     Langfuse、外部治理等外部适配器。
app/runtime/prompts/          反馈闭环 Agent prompt 构造。
app/runtime/normalizers/      LLM 输出归一化。
```

新增跨多个 store、Runtime Profile 或 workspace 文件系统的用例编排应优先放入
`app/services/`；新增持久化读写能力应放入 `app/runtime/stores/`。

```yaml
      - ./volume/regression-impact-analyzer-workspace:/regression-impact-analyzer-workspace
      - ./volume/claude-roots:/claude-roots
      - ./volume/data:/data
```

禁止在容器级别固定：

```yaml
CLAUDE_CONFIG_DIR: /root/.claude
```

原因：一旦在容器级别固定，四个 Runtime Profile 容易默认共用同一套 Claude Code 运行态。

正确做法：在后端每次启动 Claude Code 子进程时，根据 profile 动态注入 `HOME` 和 `CLAUDE_CONFIG_DIR`。

## 11. Evidence Package 设计

### 11.1 定义

Evidence package 是归因分析的事实源。

它是针对某个 feedback case 固化出来的一组证据对象，包含反馈、run、session、tool call、trace 摘要、SOC 操作事件、主智能体版本信息等。

归因分析智能体不应直接四处查询散落数据，而应优先读取 evidence package。

### 11.2 存储方式

```text
/data/runtime.sqlite3
  evidence_packages.manifest_json
  evidence_files.content_json

/data/.runtime-tmp/jobs/<job_id>/<job_type>/evidence/
  manifest.json
  feedback.json
  runs.json
  sessions.json
  tool_calls.json
  soc_events.json
  trace_summary.json
  main_agent_version.json
  redaction_report.json
  messages.json              # debug evidence 开启时
  agent_activity.json        # debug evidence 开启时
  langfuse_trace_refs.json   # debug evidence 开启时
```

SQLite 是权威存储；临时 evidence 文件只为 Agent job 读取而物化，job 完成后可清理。

### 11.4 规则


## 12. 反馈优化主链路

完整链路如下：

```text
用户聊天 / SOC 操作
  -> run_id / session_id / alert_id / case_id
  -> feedback signal / SOC event / pending correlation
  -> feedback case
  -> evidence package
  -> attribution job
  -> attribution output
  -> optimization batch
  -> proposal-generator 生成 optimization plan
  -> 开发人员点击具体优化任务执行
  -> optimization task
  -> execution-optimizer 生成执行方案
  -> 后端校验后受控应用 patch
  -> 创建执行前/执行后主智能体版本
  -> 手动触发回归验证
```

当前阶段不做静默全自动改文件；`execution-optimizer` 只生成执行方案，后端在任务执行动作后受控写入。

### 12.1 反馈评估集用例治理

反馈处置单沉淀出的 `eval_case` 面向开发人员调试和持续回归，允许在 UI 中直接编辑：

```text
1. 可编辑 prompt、expected_behavior、checks_json、labels 和 status。
2. 当前实现采用直接覆盖策略，不创建 revision，也不复制新的 eval_case_id。
3. status=active 的用例参与自动和批量回归；draft、archived 不参与自动选择。
4. eval_run、eval_run_item、模型回答、检查结果和历史运行摘要保持只读，不允许编辑。
```

直接覆盖会影响后续回归输入，但不会回算或修改历史回归结果。

## 16. 输出校验规则

所有 Agent 输出分为两层：

```text
raw_output_json          # Agent 原始输出，SQLite 字段
validated_output_json    # 通过 schema 校验后的结构化输出，SQLite 字段
error_json               # 校验或运行错误，SQLite 字段
```

规则：

```text
1. 后续流程只能读取 validated_output_json。
2. raw_output_json 只用于排错、审计和手动 revalidate。
3. Runtime 只对 `schema_version` 精确匹配目标 schema 的完整 JSON 走直接校验；其他自然语言、结构化要点或片段 JSON 均进入 DSPyOutputFormatter 规范化。
4. 离线模式不等于无模型模式；部署必须提供本地或内网模型能力，系统不再生成“未配置模型提供商”的 offline/raw 占位输出。
5. DSPyOutputFormatter 只能做格式转换，不能补充证据中没有的业务事实；证据不足时应输出 needs_human_review / insufficient_information。
6. DSPyOutputFormatter 输出后仍必须经过 Runtime 的 Pydantic schema 最终校验。
7. 校验失败或 formatter 失败时 job 状态改为 failed，并写入 error_json。
8. 当前 proposal job 提供手动 revalidate 接口；attribution job 提供显式 regenerate 接口。
9. schema_version 必填。
```

建议使用 Pydantic 定义 schema。

## 17. 反馈信号和 SOC 事件采集

### 17.1 采集定位

反馈信号和 SOC 事件采集只负责写入新版信号池，不直接归因、不直接生成 proposal、不直接创建 optimization task。

```text
POST /api/feedback-signals
  -> 写入 feedback signal pool
  -> 可关联 run/case 时允许进入 feedback case 候选池
  -> 不启动 attribution job
  -> 不生成 proposal

POST /api/soc-events
  -> 写入 SOC event pool
  -> 按关联规则匹配 run/case
  -> 无法关联时进入 pending correlation
  -> 不启动 attribution job
  -> 不生成 proposal
```

### 17.2 Feedback Signal Schema

`POST /api/feedback-signals` 用于聊天显式反馈、人工标注和系统捕捉的隐式反馈。

```json
{
  "signal_id": "可选；不传由 Runtime 生成",
  "source_type": "explicit_feedback | implicit_feedback | analyst_annotation",
  "timestamp": "2026-05-22T10:00:00Z",
  "run_id": "可选",
  "session_id": "可选",
  "alert_id": "可选",
  "case_id": "可选",
  "labels": [],
  "comment": "可选",
  "confidence": "low | medium | high",
  "auto_captured": false,
  "requires_review": false,
  "metadata": {}
}
```

规则：

```text
1. 显式用户反馈默认 auto_captured=false。
2. 隐式反馈必须 auto_captured=true。
3. 隐式反馈默认 requires_review=true。
4. 隐式反馈不得直接生成可执行 workspace 修改建议。
5. 缺少 run_id 时，必须至少提供 session_id 或 alert_id/case_id，供后续关联。
```

### 17.3 SOC Event Schema

`POST /api/soc-events` 用于外部网络安全运营系统推送有反馈价值的业务事件。

```json
{
  "event_id": "soc-case-evt-20260522-000001",
  "source_system": "sec-ops-ui",
  "event_type": "case.verdict_changed",
  "timestamp": "2026-05-22T10:00:00Z",
  "run_id": "可选",
  "session_id": "可选",
  "alert_id": "可选",
  "case_id": "可选",
  "actor_id": "可选",
  "before": {},
  "after": {},
  "entities": {
    "asset_ids": [],
    "iocs": [],
    "hostnames": []
  },
  "confidence": "medium",
  "auto_captured": true,
  "requires_review": true,
  "metadata": {}
}
```

规则：

```text
1. event_id 必须由 source_system 保证幂等。
2. 重复 event_id 返回已存在记录，不重复写入、不重复关联。
3. SOC event 默认 auto_captured=true、requires_review=true。
4. SOC event 只能进入 evidence package，不能直接生成 proposal。
5. before/after 在生产模式应只保留归因所需字段，不保存密钥、凭据、MCP header 或大段原始日志；开发调试模式允许保留更完整证据，依赖环境开关控制。
```

### 17.4 SOC 事件白名单

只采集有明确反馈价值的业务事件。

```text
case.verdict_changed
case.severity_changed
recommendation.accepted
recommendation.rejected
recommendation.modified
evidence.added
tool.manual_query_after_agent
```

不采集：

```text
普通点击
页面浏览
表格排序
筛选切换
鼠标悬停
无业务语义的打开/关闭面板动作
```

### 17.5 关联规则

采集后按以下优先级关联 run/case：

```text
1. run_id 精确关联。
2. session_id + alert_id/case_id 关联。
3. alert_id/case_id + 时间窗口关联。
4. IOC / asset / hostname 等实体相似关联。
5. 仍无法关联则进入 pending correlation。
```

`pending correlation` 不能直接进入 attribution job。必须先经过人工确认或后续事件补齐关联信息，形成 feedback case 后才允许生成 evidence package。

### 17.6 新版数据路径

当前实现的数据层以以下路径为准：

```text
/data/runtime.sqlite3
/data/.runtime-tmp/jobs/
/data/agent-versions/main/current.json
/data/agent-versions/main/versions.jsonl
/data/agent-versions/main/manifests/
/data/agent-versions/main/bundles/
/data/external-governance-webhooks.yaml
```

`/data/feedback-signals/`、`/data/soc-events/`、`/data/pending-correlations/`、`/data/feedback-cases/`、`/data/evidence-packages/`、`/data/optimization-proposals/`、`/data/optimization-tasks/` 等目录仅为兼容路径或历史说明，不是权威事实源。旧版 `/data/feedback/*.jsonl` 和旧版规则生成的 proposal 不参与新版闭环。

## 24. 最终架构决议

最终决议如下：

```text
1. 采用“六套 Runtime Profile，同镜像运行，API 与 worker 分进程”的架构。
2. 主智能体使用 main-workspace。
3. 归因分析智能体使用 attribution-analyzer-workspace。
4. 优化方案生成智能体使用 proposal-generator-workspace。
5. 六个 Agent 使用独立 claude-root、HOME、CLAUDE_CONFIG_DIR。
6. /data 作为共享业务数据区；当前以 runtime.sqlite3 为权威存储，agent_jobs 和临时 job 文件按 job_id 分区。
7. 主智能体是反馈优化的主要版本管理对象。
8. 归因分析智能体、优化方案生成智能体、execution-optimizer、eval-case-governor 和 regression-impact-analyzer 有自身版本，但不纳入主智能体优化版本。
9. feedback signal 和 SOC event 是新版唯一采集入口。
10. 采集入口不直接归因、不直接生成优化方案。
11. evidence package 是归因事实源。
12. Langfuse 是观测和跳转工具，不是唯一事实源。
13. 归因分析智能体和优化方案生成智能体读取 evidence package、job input 和版本 manifest，不直接修改 main-workspace。
14. execution-optimizer 读取任务和版本上下文，只输出执行方案；后端在任务执行动作后受控应用 patch。
15. 人工或外部系统仍可通过 mark-applied 兜底创建版本快照，并手动触发回归验证。
16. 外部 MCP、SOC 流程、Runtime bug 等问题必须通过 external_guidance、runtime_fix 或 needs_human_analysis 表达。
```

## 25. 一句话总结

本架构把反馈优化闭环拆成三类对象：

```text
main-workspace
  = 主智能体的可版本化行为源码包。

claude-roots/*
  = 各 Agent 的隔离运行态，不进入主智能体版本。

/data/runtime.sqlite3 + /data/.runtime-tmp/jobs + /data/agent-versions/main
  = 反馈闭环的权威数据、临时任务工作区和版本记录。
```

通过这种边界，系统可以做到：

```text
可运行、可隔离、可审计、可复现、可审批、可回滚、可持续优化。
```
