# 反馈闭环当前实现基线

> 文档定位：当前实现基线。
> 读者：需要理解当前反馈闭环代码、API、运行态和迁移边界的研发、测试和治理人员。
> 权威边界：当前 API 细节以运行时 OpenAPI 为准；四阶段改进治理工作台 UI 和效果验收以 `docs/AgentGov_四阶段改进治理工作台UI整改方案.md` 为准。

## 治理对象矩阵

| 维度 | 当前基线 |
| --- | --- |
| 被治理对象 | 业务 Agent；默认样板是 `main-agent`，长期边界是多个业务 Agent。 |
| 治理执行者 | 单一 `governor`，承担归因、优化方案、执行和回归用例生成。 |
| 用户主对象 | `ImprovementItem`，即“改进事项”。 |
| 反馈归属 | 反馈必须能追溯到业务 Agent、运行记录、版本、场景或外部事件来源。 |
| 证据来源 | 运行记录、feedback signal、SOC event、feedback case、evidence package 和 Langfuse trace。 |
| 发布资产 | 已实现 typed `TestDataset`、EvalRun 数据集快照、change set、release 和通用 Asset Registry 投影。 |

## 闭环链路

```text
业务 Agent
  -> 运行记录 / Trace
  -> feedback signal / SOC event / 人工标注
  -> feedback case / evidence package
  -> ImprovementItem
  -> normalized feedback
  -> attribution
  -> optimization plan
  -> execution record / change set
  -> regression assessment / typed TestDataset / EvalRun
  -> release gate / release
```

这条链路的目标不是“自动修改提示词”，而是把反馈变成可归因、可确认、可执行、可回归、可发布、可回滚的治理事实。
回归评估确认后可从事项证据链采用 typed `TestDataset`；回归运行绑定数据集修订及不可变用例快照，不再扫描旧全局用例池。通用 `AssetRecord` 继续承载其他 Registry 资产，不作为 TestDataset 的替代表示。

## 当前运行态对象

| 对象 | 当前职责 | 四阶段投影 |
| --- | --- | --- |
| `agent_runs` | 保存一次业务 Agent 运行的轻量记录、版本、会话和 trace 关联。 | 反馈来源和 Trace 证据。 |
| `feedback signal` | 保存用户或系统反馈来源。 | 反馈整理的来源记录。 |
| `SOC event` | 保存外部安全运营事件或监控事件。 | 外部反馈和证据来源。 |
| `pending correlation` | 表示待人工关联的反馈/事件。 | 反馈整理阶段的待确认事项。 |
| `feedback case` | 后端证据容器，承载 evidence、归因 job 和历史兼容数据。 | 改进事项的证据上下文。 |
| `evidence package` | 固化运行上下文、trace、消息、错误、版本和相关证据。 | ContextPackage / TraceSummary 的事实来源。 |
| `ImprovementItem` | 事项级治理实体，聚合反馈并推进四阶段。 | 用户主对象。 |
| `attribution` | 归因结论、责任边界、证据、反证和不确定性。 | 归因分析阶段产物。 |
| `optimization plan` | 事项级优化摘要、变更项和风险等级。 | 优化执行阶段输入。 |
| `execution record` | 执行摘要、应用变更、候选版本、回滚说明和 change set。 | 优化执行阶段产物。 |
| `regression assessment` | 回归用例候选和发布门禁阈值建议。 | 测试发布阶段输入。 |
| `TestDataset` | 保存业务 Agent 归属、事项来源、修订、生命周期、provenance 和有序用例。 | 测试发布阶段的数据集与回归运行唯一用例来源。 |
| `EvalRun` | 固化数据集修订和完整用例快照并执行候选回归。 | 回归运行及发布门禁证据。 |
| `AssetRecord` | 按 `asset_type` 保存 methodology、execution、audit 通用 Registry 资产。 | 与 typed TestDataset 分离，不承载其生命周期或用例真相，也不接受旧 `regression` 类型。 |
| `change set` / `release` | 业务 Agent 文件变更、候选版本、发布和回滚记录。 | 发布门禁与发布结果。 |

## Trace 边界

- 业务 Agent 聊天 Trace 由 Playground 的“查看 Trace”入口展示。
- governor 成功生成的归因、优化方案、执行和回归评估会保存 `generation_trace_id` / `generation_trace_url`；heuristic 或手工产物可以为空。
- 前端在归因、优化方案、执行记录和测试发布面板提供“查看 Trace”入口。
- 后端通过 `GET /api/langfuse/traces/{trace_id}` 代理读取 Langfuse trace，前端不直接持有 Langfuse secret。
- Playground 历史已通过 `/v1/conversations/{conversation_id}/items` 投影 SDK session transcript；SQLite `agent_runs` 只保留反馈、Trace 和运行证据关联，不再作为消息历史恢复源。session mapping 使用持久化 active-run 租约阻止流式 turn 期间并发删除。

## TestDataset 当前契约

- `TestDatasetResponse` 是后端、OpenAPI 和前端生成类型的单一字段真相；稳定 ID 为 `dataset_id`，不存在 Registry asset ID 与 body 内部 ID 双轨。
- 数据集只能从已确认的事项回归证据链采用，后端注入 Agent 归属、provenance、候选版本和有序用例；客户端不能伪造这些字段。事项返工形成新的 execution / 回归评估证据链时采用为新的确定性 dataset 版本，旧数据集及历史 EvalRun 快照不改写。
- 集中生命周期为 `draft -> active -> evaluating -> active`；只有 active 可启动 EvalRun，evaluating 由运行原子占用并在完成、失败或取消时释放。活跃 EvalRun 以 15 分钟 heartbeat 租约防止多进程启动误抢占，过期运行由启动检查和每分钟 reconciliation 失败化并释放数据集。人工可从 active/evaluating 进入 `deprecated` 或 `archived`、从 deprecated 恢复 active；archived 为终态。所有变更使用 expected revision CAS 并形成不可变 revision 记录。
- 候选回归启动前和创建事务内都会校验数据集属于目标 Agent、状态为 active、包含用例，且事项、execution、candidate provenance 与 ChangeSet 完全一致；启动后固化 evaluating revision 与完整 case snapshots。
- EvalRun 只把非空回答和无运行错误视为运行健康检查；`expected_behavior` / `checkpoints` 仍是自然语言时必须投影为 `needs_human_review`，不能自动通过。取消会先把真实 EvalRun 写成失败终态，change set 不构造伪 EvalRun。
- 候选回归的自然语言语义待复核会进入独立 `regression_review_required` 状态。专用 review API 以幂等 `review_id`、operator、reason、固定 scope 和完整逐 case 决策绑定当前 EvalRun；全接受投影为 `passed_with_notes`，任一拒绝投影为 blocked。原始 item/check 不改写，EvalRun、ChangeSet 和事件审计同事务提交，并发相反决策只能一个获胜。
- 跨 Agent 读取、跨数据集 case、归档/空数据集、hostile backend-owned 字段和非法生命周期转移均为负向契约。

## API 与测试锚点

- 完整 API 以运行时 `/openapi.json` 或 `scripts/export_openapi.py` 导出的临时 OpenAPI 为准。
- 改进事项主入口是 `/api/improvements` 及其内容子资源：`normalized-feedback`、`attribution`、`optimization-plan`、`execution`、`regression-assessment`。
- typed 测试数据集入口是 `/api/test-datasets`、`/api/test-datasets/{dataset_id}`、`/api/test-datasets/{dataset_id}/lifecycle`、`/api/test-datasets/{dataset_id}/revisions`，事项采用入口是 `/api/improvements/{improvement_id}/test-dataset/adopt`。
- 旧批次优化、任务、提案、外部治理和回归影响链路已从活跃 API 中移除；旧 SQLite 表由迁移清理。
- 修改反馈闭环、治理模型任务、store 投影、API response 或用户可见 tab 状态时，必须运行 `make main-flow-test` 并同步 `tests/quality_policy.json`。

## 运行隔离与存储

- API 进程负责反馈、改进事项、治理模型调用、版本、评估和运行态对账，不再创建或领取 Agent job 队列。
- 改进事项四阶段生成由 API 内的后端服务直接调用 governor 并写入内容子资源；升级前 `agent_jobs` 仅保留只读历史查询，migration 0042 会把未终态历史行收口为带明确退役错误码的失败终态。
- `/data/runtime.sqlite3` 是反馈、改进事项、执行、评估以及 change set/release 元数据的权威存储；发布、回滚和恢复的 durable operation 会在 API 启动及周期任务中对账，Git 已落地而 DB 尚未完成的中断可自愈。
- `/data/business-agents/<agent_id>/workspace` 的 Git repository 是业务 Agent 文件内容、commit、ref 和 tag 的版本权威；候选 worktree 和发布归档位于同级 `version/` 下。
- `/data/feedback-signals/`、`/data/soc-events/`、`/data/feedback-cases/` 等旧目录只作为兼容或历史说明，不再是权威存储。

## Runtime 与健康边界

- 手工 EvalRun 从绑定数据集读取权威 `agent_id`，并把同一 Agent 归属传给 candidate runtime；请求不能回落到 `main-agent`。
- SDK turn 在请求前写 durable intent，完成或失败时原子收口 session 与 run 投影；连续 finalize CAS 失败会立即中断 turn 或交给租约过期对账，不再冻结 `applying` 或悬挂约一小时。
- provider 版本与能力探测在后台执行并缓存。`/health/live` 只表示 API 进程存活；`/health/ready` 返回缓存的模型 readiness；外部 vLLM 超时会阻断模型请求并给出结构化诊断，但不会让 Compose 把 API/UI 判成启动失败。

## 与长期资产方案的关系

`docs/反馈闭环长期回归资产升级方案.md` 继续承载 typed TestDataset、EvalRun、发布门禁和资产 Registry 的长期设计。旧全局 EvalCase、独立回归资产和场景包链已迁移归档，不再作为演进方向；豁免审批和更完整的 ReleaseGate 风险模型仍按长期方案演进。
