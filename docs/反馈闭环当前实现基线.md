# 反馈闭环当前实现基线

> 文档状态：当前运行态事实说明。
>
> 改进治理工作台的四阶段结构、页面信息归属和效果图以
> [AgentGov 四阶段改进治理工作台 UI 整改方案](./AgentGov_四阶段改进治理工作台UI整改方案.md)
> 为绝对依据；字段级契约以运行时 OpenAPI 和当前代码为准。

## 1. 当前主链路

```text
反馈来源
  -> ImprovementItem
  -> NormalizedFeedback
  -> Attribution
  -> OptimizationPlan
  -> ExecutionRecord / AgentChangeSet
  -> RegressionTestDesign
  -> Workspace tests/test_*.py
  -> AgentTestRun
  -> Release / rollback
```

当前用户主链路只有四个阶段：反馈整理、归因分析、优化执行、测试发布。后端内部状态、Git 操作、
治理 Agent 调用和平台测试运行只服务这四个阶段，不形成平行产品流程。

## 2. 核心对象与所有权

| 对象 | 所有者 | 当前作用 |
| --- | --- | --- |
| `ImprovementItem` | 后端 | 聚合反馈并承载阶段、归属和内容引用。 |
| `NormalizedFeedback` | 用户 + 治理 Agent | 反馈整理阶段的结构化事实。 |
| `Attribution` | 治理 Agent，用户确认 | 归因结论、证据、置信度和待确认项。 |
| `OptimizationPlan` | 治理 Agent，用户确认 | 优化动作、风险、验证建议和执行目标。 |
| `ExecutionRecord` | 后端 | 隔离 worktree 中实际应用的变更、Diff、待发布提交和回滚信息。 |
| `AgentChangeSet` | 后端 | 同一未发布改动的业务容器；可以在返工后指向更新的待发布提交。 |
| 回归测试代码候选 | 治理 Agent + 平台，用户确认 | 治理 Agent 输出完整 pytest 代码、测试意图和断言依据；平台确定路径。确认前是待审查 Diff，不是长期测试资产。 |
| `tests/test_*.py` | 业务 Agent Workspace | 可执行测试资产的唯一真相源，与 Agent 代码和配置一起进入 Git。 |
| `AgentTestRun` | 平台 | 固定 pytest 命令在精确提交上的持久化执行证据。 |
| `AgentTestSchedule` / `AgentTestScheduleEvent` | 平台 | 每 Agent 唯一定时策略及每个计划窗口的触发审计；不保存测试正文。 |
| `Release` | 后端 | 发布、强制发布警告、恢复和回滚记录。 |
| `AssetRecord` | 后端 | 方法论、执行和审计等通用 Registry 资产；不复制 Workspace 测试内容。 |

所有注册业务 Agent（含 `main-agent`）遵循相同的 Workspace、测试、变更集和发布机制。`governor`
是治理执行者，不进入业务 Agent 注册表、Playground 业务选择器或业务发布链。

## 3. 四阶段内容契约

- 反馈整理：归并来源并确认 `NormalizedFeedback`。
- 归因分析：治理 Agent 生成 `Attribution`，用户确认或返回整理阶段修正事实。
- 优化执行：生成并确认 `OptimizationPlan`；后端在隔离 worktree 应用确定性动作并产生
  `ExecutionRecord`、Diff、`AgentChangeSet` 和待发布提交。
- 测试发布：治理 Agent 生成回归测试代码候选；后端校验并确定 `tests/test_feedback_*.py` 路径。
  用户确认待发布变更后，后端只新增测试文件，并把配置和测试收口为相对修复前版本的同一待发布
  commit；用户再显式运行完整 `tests/`，平台才创建待发布检查 `AgentTestRun`。该确认动作不调用定时策略。

内容生成、确认、执行和测试物化会校验同一事项、业务 Agent、执行记录和变更集的依赖关系。
客户端不能提交后端拥有的 job ID、提交 SHA、状态、运行结果或 provenance 来伪造阶段完成。

## 4. Workspace pytest 测试资产

每个业务 Agent 的测试随 Workspace 管理：

```text
workspace/tests/
├── README.md
├── conftest.py       # 可选
└── test_*.py
```

首版只接受 `tests/` 下的扁平 Python 测试文件。导入包缺少 `tests/`、说明文件或测试文件时允许导入，
但返回结构化 warning；没有可运行测试的版本不能满足普通发布条件。新建和覆盖导入都要求包根目录
`agent.yaml.agent.id` 有效，并与 API URL 中的目标 `agent_id` 逐字一致；缺失、无效、格式错误
或来源 ID 不一致会在任何 Workspace、注册表、Git 或会话变更前被明确拒绝。

`RegressionTestDesign` 只表达候选测试语义。用户确认后，后端才把它确定性物化为不可覆盖、不可删除、
不可弱化的测试文件。测试文件、fixture 和辅助资产随 Workspace Git 提交，平台不再维护数据库测试集、
全局用例池或第二份测试正文。

完整工程契约见
[业务 Agent Workspace 原生 pytest 测试资产实现方案](./engineering/业务AgentWorkspace原生pytest测试资产实现方案.md)。

## 5. 平台测试运行

平台只执行固定命令：

```bash
python -m pytest -q -p agentgov_testkit.pytest_plugin tests
```

创建运行时省略 `commit_sha`，后端会在请求内固定当时的当前提交；不会等到队列消费时再解释“最新”。
客户端不能覆盖命令、工作目录、测试状态或报告。`AgentTestRun` 保存精确 `agent_id`、`commit_sha`、
派生 `suite_digest`、可选 `change_set_id`、stdout、stderr、pytest item 和错误详情。

运行状态为：

```text
queued -> running -> passed | failed | error | cancelled
running --服务关闭或重启--> interrupted
```

服务启动时把遗留 `running` 收口为 `interrupted`，把 `queued` 重新入队。测试临时会话属于 API 进程，
重启后不会伪装成可恢复会话，而是返回明确的不可用错误。

资产复利中心默认以左侧业务 Agent 导航、右侧详情区展示当前有效 commit 的测试文件、suite 诊断、
最近运行、运行历史和定时策略；源码从 Workspace Git 只读加载，不写入通用资产表。每个 Agent 最多一条五字段 Cron + IANA
时区策略，最短间隔 15 分钟。定时触发读取触发时当前有效 commit，创建 `source=scheduled`、
`change_set_id=null` 的运行；它不读取候选 worktree，也不推进发布状态。错过多个窗口只补一次，同
Agent/commit 已有活跃运行时记为合并而不重复执行；Agent 归档或删除时停用策略但保留历史审计。

## 6. 版本与发布条件

Git `commit_sha` 是被测版本的唯一权威标识；`suite_digest` 是从该提交测试文件派生的完整性摘要，
`change_set_id` 只用于关联同一未发布改动。不得把三者拼成新的人工版本身份。

普通发布至少要求：

- 归因、执行和待发布版本 provenance 完整；
- 当前 `AgentChangeSet.candidate_commit_sha` 存在；
- 同一业务 Agent、同一待发布 `commit_sha` 存在 `passed` 的平台测试运行；
- 当前没有其他发布阻塞项。

旧提交上的通过记录不能放行更新后的待发布提交。失败、错误、取消、中断、空测试目录或只有测试设计
也不能放行。UI 使用“修复前版本”和“待发布版本”，不把“基线/候选”作为用户标签。

反馈闭环待发布版本必须在当前精确提交上运行完整 `workspace/tests/`，已有失败和新增失败都必须整改；
该链路不能强制绕过测试条件，反馈发布工作台不提供强制发布入口。历史 release 仍展示原阻塞项、原因、
操作人和警告。未关联反馈、由版本治理 API 手工创建的待发布版本可通过受保护 API 强制发布；provenance 不完整始终不得绕过。

## 7. 公开 API 分组

- 四阶段内容：`/api/improvements/{improvement_id}/normalized-feedback`、`attribution`、
  `optimization-plan`、`execution`、`regression-test-design` 及其 generate/confirm 动作。
- 测试资产检查：`GET /api/agent-test-assets`、`GET /api/agent-registry/{agent_id}/test-suite` 和
  `GET /api/agent-registry/{agent_id}/test-suite/file`。
- 平台测试运行：`POST /api/agent-test-runs`、`POST /api/agent-change-sets/{change_set_id}/test-runs`、`GET /api/agent-test-runs`、
  `GET /api/agent-test-runs/history`、`GET /api/agent-test-runs/{test_run_id}`、`POST /api/agent-test-runs/{test_run_id}/cancel`。
- 测试定时策略：`GET/PUT /api/agent-registry/{agent_id}/test-schedule`、
  `GET /api/agent-registry/{agent_id}/test-schedule/events`。
- testkit 会话：`POST /api/agent-test-sessions`、`POST /api/agent-test-sessions/{id}/messages`、
  `DELETE /api/agent-test-sessions/{id}`。
- 版本治理：`/api/agent-change-sets`、`/api/agent-releases` 及 publish/restore/rollback 动作。

接口字段、状态码和请求体以运行时 `/openapi.json` 为准，不从本文复制 schema。

## 8. 存储与迁移

- `${HOME}/volume-agent-gov/data/runtime.sqlite3` 保存事项内容、变更集、平台测试运行、发布和审计记录。
- `${HOME}/volume-agent-gov/data/business-agents/<agent_id>/workspace` 保存业务 Agent Git Workspace；
  worktree 与发布归档位于同一 Agent 的 `version/` 下。
- SQLite migration `0048` 先归档旧测试数据集和评估运行原始行，再删除旧表、字段和活跃 API 依赖；
  migration `0049` 收口四阶段测试产物命名；migration `0051` 原样归档旧自然语言测试设计，删除旧表并
  建立测试代码、测试意图和断言依据的新契约，不猜测转换旧设计；migration `0052` 增加测试定时策略、
  调度事件和运行触发来源字段。
- 历史迁移、归档文档和负向测试可保留旧名称，但它们不是兼容入口，也不得重新进入主流程。

## 9. 明确不再成立的设计

- 数据库测试集、全局测试用例池和逐用例人工复核不是当前测试资产模型。
- 测试内容不写入通用 `AssetRecord.body`；资产复利中心只从 Workspace Git 只读投影源码，并从运行库
  投影运行与调度证据。
- 平台不接收上传后的任意 pytest 命令，不自动安装测试依赖，也不把本地断言回传解释成权威结果。
- `main-agent` 没有特殊测试或发布通道。
- 治理 Agent 不作为业务 Agent 被导入、选择、测试或发布。
