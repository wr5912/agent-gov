> 文档定位：pre-v2.7 当前实现基线。
> 读者：需要理解当前反馈闭环代码、API、运行态和迁移来源的研发、测试和治理人员。

> 权威边界：当前 API 细节以运行时 OpenAPI 为准；v2.7 改进治理工作台 UI、四阶段主链路和效果图验收以 `docs/AgentGov_v2.7_四阶段改进治理工作台UI整改方案.md` 为准。

## 治理对象矩阵

| 维度 | 当前基线 |
| --- | --- |
| 被治理对象 | 当前默认业务 Agent 是 `main-agent`；长期边界是多个业务 Agent，不是只治理 `main-agent`。 |
| 治理执行者 | 后端规则、人工确认和治理 Agent 组合执行；归因、方案、执行、用例治理和回归影响分析职责在运行态由 `governor` 按 job type 承担。 |
| 资产类型 | 同时包含数据资产、方法论资产、执行资产、版本资产和审计资产。 |
| 生命周期 | 反馈、批次、任务、eval case、eval run、change set、release 均有自己的状态机或治理状态。 |
| 反馈归属 | 反馈必须归属到业务 Agent、运行记录、版本、场景或外部事件来源，不能只作为孤立文本保存。 |
| 当前实现边界 | 当前闭环仍以反馈信息、优化批次、回归资产、版本治理等 pre-v2.7 对象承载。 |
| 目标能力边界 | 后续 v2.7 应投影为改进事项四阶段闭环和资产 Registry，不再让旧菜单决定用户主流程。 |

## 闭环链路

```text
业务 Agent
  -> 运行记录 / trace
  -> feedback signal / SOC event / 人工标注
  -> feedback case / evidence package
  -> attribution job
  -> optimization batch / optimization plan
  -> optimization task / execution plan
  -> 受控应用 / change set
  -> eval run / regression gate
  -> release
  -> regression asset / TestDataset / Asset Registry
```

这条链路的关键不是“自动修改提示词”，而是把反馈变成可归因、可审批、可回归、可发布、可回滚的治理事实。

## 当前运行态对象

| 对象 | 当前职责 | v2.7 投影 |
| --- | --- | --- |
| `agent_runs` | 保存一次业务 Agent 运行的轻量记录、版本、会话和 trace 关联。 | 改进事项的运行证据来源。 |
| `feedback signal` | 保存用户或系统反馈来源。 | `Feedback` 来源之一。 |
| `SOC event` | 保存外部安全运营事件或监控事件。 | 外部反馈和证据来源。 |
| `pending correlation` | 表示待人工关联的反馈/事件。 | 反馈整理阶段的待确认事项。 |
| `feedback case` | 当前后端单反馈处理容器，承载 evidence、归因和历史兼容数据。 | 改进事项的证据与归因上下文。 |
| `evidence package` | 固化运行上下文、trace、消息、错误、版本和相关证据。 | ContextPackage / TraceSummary 的事实来源。 |
| `agent_jobs` | 归因、方案、执行、用例治理、回归影响分析等异步任务队列。 | 治理 Agent 执行记录。 |
| `optimization batch` | 多条反馈合并生成方案和回归验证的当前容器。 | `ImprovementItem` 聚合和阶段推进能力。 |
| `optimization task` | 承接方案中的可执行任务、执行计划和应用结果。 | 优化执行阶段的执行记录。 |
| `eval case` / `regression asset` | 反馈衍生测试用例、长期回归资产和治理状态。 | 测试发布阶段的 TestDataset / RegressionAsset。 |
| `change set` / `release` | 业务 Agent 文件变更、候选版本、发布和回滚记录。 | 测试发布阶段的版本门禁与发布结果。 |

## 运行隔离与存储

当前实现运行在同一套 FastAPI Runtime 和 worker 体系下：

- API 进程负责创建反馈、批次、任务、版本和 `agent_jobs`。
- worker 进程领取 `agent_jobs`，按 job type 构造 prompt context 并执行治理 Agent。
- `main-agent` 是当前默认业务 Agent 样板；治理职责由 `governor` 承担，不等同于被治理对象。
- `/data/runtime.sqlite3` 是反馈、作业、执行、评估、版本和发布的权威存储。
- `/data/business-agents/<agent_id>/workspace` 是当前业务 Agent Git 版本源；候选 worktree 和发布归档位于同级 `version/` 下。预制 `main-agent` 使用 `/data/business-agents/main-agent/version/{worktrees,releases}`。
- `/data/feedback-signals/`、`/data/soc-events/`、`/data/feedback-cases/` 等旧目录只作为兼容或历史说明，不再是权威存储。

## 当前用户界面边界

pre-v2.7 当前实现仍存在这些用户可见能力来源：

- 反馈信息：查看反馈来源、补充标注、生成或编辑评估用例、加入优化批次。
- 优化批次：聚合多条反馈、运行归因、生成优化方案、执行任务和回归验证。
- 回归资产：管理反馈衍生 eval case、晋级长期资产、标记 flaky、归档或 supersede。
- 版本治理：查看 change set、diff、发布、恢复和版本事件。

这些名称只说明当前运行态和迁移来源。后续改进治理工作台必须投影为四阶段：反馈整理、归因分析、优化执行、测试发布。旧菜单、旧七段链路和旧发布顶级入口不得作为 v2.7 新 UI 的兼容目标。

## API 与测试锚点

- 完整 API 以运行时 `/openapi.json` 或 `scripts/export_openapi.py` 导出的临时 OpenAPI 为准，本文不复制完整路由清单。
- 前端类型由 OpenAPI 生成，不能让文档里的旧字段表反向覆盖当前 schema。
- 修改反馈闭环、agent job、store 投影、API response 或用户可见 tab 状态时，必须运行 `make main-flow-test` 并同步 `tests/coverage_policy.json`。
- 影响全局契约、提交或发版时，必须运行 `make test` 和治理硬门。

## 迁移结论

当前实现基线的价值是解释“现在代码和运行态为什么这样”。未来 v2.7 的价值是定义“下一版用户应该如何完成治理决策”。两者不能混成一个权威口径：当前实现只作为迁移来源，四阶段方案才是改进治理工作台后续 UI 与代码整改的依据。
