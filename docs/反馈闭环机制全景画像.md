> 用途：用一张全景图说明当前反馈闭环从“如何反馈”到“收集反馈信息”，再到“基于反馈持续优化并演进 main 智能体”的完整机制。  
> 状态：现状画像 + 演进目标  
> 日期：2026-06-01

## 1. 一句话定义

反馈闭环的本质是：

```text
用户或外部系统指出 main 智能体表现中的问题
  -> Runtime 把问题和当时运行上下文固化为反馈证据
  -> 归因智能体判断问题来自哪里
  -> 方案智能体把多个问题合并成可执行优化计划
  -> 开发人员按任务确认执行
  -> 执行智能体只生成结构化修改方案
  -> 后端校验并受控写入 main-workspace
  -> 创建 main 智能体版本快照
  -> 回归测试验证新版本
  -> 长期升级后由回归影响分析判断失败原因和发布风险
  -> 长期升级后由用例治理把高价值反馈沉淀为长期回归资产
  -> 后续优化继续受历史反馈保护
```

这条链路的关键不是“自动改提示词”，而是让每一次反馈都经历可追溯、可审批、可回归、可回滚的工程闭环。

## 2. 全景流程图

当前全景可以拆成十个连续层次：

```text
[1 运行]
用户 / 外部系统
  -> POST /api/chat 或 /api/chat/stream
  -> main-agent 运行
  -> agent run 记录 run_id、session_id、agent_version_id、case_id、alert_id、输入、输出摘要、trace、usage、errors

[2 反馈]
Playground 显式反馈
SOC / 监控 / 外部治理事件
开发人员人工标注
  -> feedback signal / SOC event / pending correlation / source annotation

[3 统一反馈池]
反馈信息页
  -> 统一展示 feedback sources
  -> 人工筛选、搜索、补充标注
  -> 选择反馈生成默认回归用例或创建优化批次

[4 反馈单和证据]
feedback source
  -> feedback case
  -> evidence package
  -> 运行上下文、消息、trace、错误、版本、相关 case/alert 证据被固化

[5 归因]
feedback case
  -> attribution-analyzer profile
  -> attribution job
  -> attribution output
  -> 问题类型、根因、证据、置信度、建议方向

[6 批次优化]
多个 feedback case
  -> optimization batch
  -> proposal-generator profile
  -> batch optimization plan
  -> workspace task / external webhook task / manual review / blocked item

[7 执行确认]
开发人员在优化批次中点击具体任务执行
  -> optimization task
  -> execution-optimizer profile
  -> execution plan
  -> 只输出 operations，不直接写 main-workspace

[8 受控应用]
后端 ExecutionApplicationService
  -> 校验目标路径、操作类型、expected_sha256、大小、baseline version
  -> 创建 pre_execution 快照
  -> 写入 main-workspace 受管文件
  -> 创建 execution_optimizer_applied 快照
  -> 记录 applied_diff、execution job、optimization task

[9 回归与版本]
新 main agent version
  -> batch regression run / manual eval run
  -> eval run items
  -> 通过、失败、阻断、人工复核
  -> 版本管理支持查看、diff、恢复

[10 长期进化]
高价值 feedback-derived eval case
  -> eval-case-governor 治理建议
  -> 人工审核后进入长期回归资产
  -> 后续批次按策略持续运行
  -> regression-impact-analyzer 分析回归失败影响
  -> 防止历史问题复发
  -> 反哺 main-agent 指令、skills、MCP 配置和行为边界
```

## 3. 角色分工

### 3.1 人和系统角色

| 角色 | 责任 | 不能做什么 |
| --- | --- | --- |
| 终端用户 | 在使用 main 智能体后给出显式反馈 | 不能直接触发主智能体文件修改 |
| SOC / 监控系统 | 写入告警、异常、质量事件等隐式反馈 | 不能绕过人工确认直接生成可执行修改 |
| 开发人员 | 筛选反馈、创建批次、确认执行任务、判断回归结果 | 不应手工绕过版本与回归链路发布主智能体行为变更 |
| Runtime API | 统一采集、存储、编排反馈闭环 | 不把采集接口设计成“提交即自动改” |
| SQLite runtime store | 作为反馈、case、job、task、eval、version 关联关系的权威事实源 | 不依赖旧版 JSONL 目录作为新版事实源 |
| Feedback 工作台 | 提供反馈信息、优化批次、版本管理的统一操作面 | 不在反馈提交后立即展示未经归因的最终优化建议 |

### 3.2 智能体角色

当前运行时已经实现 4 个固定 profile；长期回归资产升级方案中还规划了 2 个新增 profile。为了避免把规划能力误读成当前已上线能力，这里分开描述。

当前已实现：

| 智能体 | Profile | Workspace | 核心职责 |
| --- | --- | --- | --- |
| main 智能体 | `main-agent` | `/main-workspace` | 处理用户真实任务，是被反馈和被优化的对象 |
| 归因分析智能体 | `attribution-analyzer` | `/attribution-analyzer-workspace` | 基于证据包分析失败原因，输出结构化归因 |
| 优化方案生成智能体 | `proposal-generator` | `/proposal-generator-workspace` | 汇总批次归因结果，生成批次优化计划 |
| 执行优化智能体 | `execution-optimizer` | `/execution-optimizer-workspace` | 基于已确认任务生成结构化执行方案 |

长期回归资产升级规划：

| 智能体 | Profile | Workspace | 核心职责 | 当前状态 |
| --- | --- | --- | --- | --- |
| 用例治理智能体 | `eval-case-governor` | `/eval-case-governor-workspace` | 判断候选 eval case 是否应进入长期回归资产，建议分层、去重、阻断策略、稳定性和人工审核要求 | 规划中 |
| 回归影响分析智能体 | `regression-impact-analyzer` | `/regression-impact-analyzer-workspace` | 分析回归失败是否由本批次改动导致，判断真实退化、用例过时、flaky 或历史问题复发，并给出阻断、重跑、修复、更新用例或回滚建议 | 规划中 |

隔离原则：

1. 归因分析智能体和优化方案生成智能体不能直接修改 `/main-workspace`。
2. 执行优化智能体也不能直接写 `/main-workspace`，只能输出结构化 operations。
3. 真正的文件写入只能由后端受控执行服务完成。
4. 用例治理智能体和回归影响分析智能体属于长期回归资产治理链路，只能给出治理、分层、影响分析或发布建议，不能直接晋级阻断用例，也不能直接修改 `/main-workspace`。
5. main 智能体的版本只管理 `/main-workspace` 中的受管行为包，不把分析智能体、方案智能体、执行智能体、用例治理智能体或回归影响分析智能体的 workspace 混入主版本。

## 4. 反馈从哪里来

反馈入口分为四类。

### 4.1 显式反馈

典型来源是 Playground 或业务页面上的反馈按钮。

```text
用户看到 main 智能体回答
  -> 点击有用 / 无用 / 问题类型 / 补充说明
  -> POST /api/feedback-signals
  -> feedback signal
```

显式反馈的关键上下文：

1. `run_id`：对应一次 main 智能体运行。
2. `session_id`：对应 API 会话。
3. `agent_version_id`：回答产生时的 main 智能体版本。
4. `case_id` / `alert_id`：业务对象上下文。
5. `source_type`、标签、备注、严重程度等人工信号。

设计约束：

```text
POST /api/feedback-signals 只采集反馈，不运行归因，不生成优化方案，不修改主智能体。
```

### 4.2 隐式反馈

典型来源是 SOC 系统、监控系统、质量审计或外部治理系统。

```text
外部系统发现异常
  -> POST /api/soc-events
  -> SOC event
  -> 如果能关联 run/session/case，则进入统一反馈池
  -> 如果暂时无法关联，则进入 pending correlation
```

隐式反馈适合表达：

1. main 智能体遗漏关键证据。
2. 输出格式不满足下游消费。
3. 告警研判结论与人工复核不一致。
4. 安全边界、权限或工具使用不符合治理要求。
5. 回归运行失败或质量指标异常。

约束：

```text
隐式反馈不能直接生成可执行 workspace 修改。
它必须先被关联、确认，并进入 evidence package / attribution 链路。
```

### 4.3 待关联反馈

当反馈中缺少 `run_id`、`session_id`、`case_id` 或 `alert_id` 时，系统不能假定它属于哪次 main 智能体运行。

```text
SOC event / feedback signal
  -> 缺少可验证关联
  -> pending correlation
  -> 人工或后续数据补齐关联
  -> resolved
  -> 才能进入后续批次
```

Feedback 工作台会禁用未 resolved 的 pending correlation 批次选择，避免把证据不完整的事件直接送入优化。

### 4.4 人工标注

开发人员可以在统一反馈源上补充标注：

```text
GET /api/feedback-sources
PATCH /api/feedback-sources/{source_kind}/{source_id}
```

人工标注用于补齐：

1. 业务影响。
2. 优先级。
3. 是否适合生成回归用例。
4. 是否应该进入某个优化批次。
5. 归因时需要关注的额外背景。

## 5. 运行记录如何成为证据

每次 main 智能体运行都会生成轻量运行记录。它是反馈闭环的第一块事实基础。

运行记录包含：

1. `run_id`、`session_id`、`sdk_session_id`。
2. `agent_version_id`。
3. 用户输入、prompt、输出摘要。
4. 消息事件、agent activity、工具活动摘要。
5. `alert_id`、`case_id`、metadata。
6. usage、成本、stop reason、错误。
7. 可选 Langfuse trace id 和 trace url。

当反馈进入 case 后，系统会围绕这次运行形成 evidence package。

证据包的作用：

1. 让归因分析智能体不依赖口头描述，而是读取冻结后的事实。
2. 让后续方案生成知道当时 main 智能体版本和输入输出。
3. 让审计人员可以复盘“为什么当时判断这个反馈有效”。
4. 让版本回滚和回归失败分析有明确来源。

证据包不是长期主存本身；权威状态仍在 SQLite runtime store。证据包是面向智能体 job 的冻结输入和可审计材料。

## 6. 统一反馈池

Feedback 工作台当前提供三个主菜单：

1. `反馈信息`
2. `优化批次`
3. `版本管理`

其中 `反馈信息` 是统一入口。它把以下对象聚合成可操作的 feedback source：

1. feedback signals。
2. SOC events。
3. pending correlations。
4. 已有关联的 feedback cases。
5. 反馈生成或关联的 eval cases。

开发人员在这里完成两个关键动作：

```text
选择反馈 -> 生成回归用例
选择反馈 -> 创建优化批次
```

这一步非常重要，因为系统不把所有反馈都默认等价处理。低质量、重复、不完整、无法复现或缺少关联的反馈应先停留在反馈池中，而不是直接进入优化批次。

## 7. 从反馈到回归用例

反馈闭环里有两类用例：

1. 当前已实现的 `eval_cases`：反馈生成或手工创建的评估用例，可被批次关联并运行。
2. 规划升级的长期回归资产：在 `eval_cases` 基础上增加资产治理、分层、晋级、阻断策略和修订历史。

当前已实现路径：

```text
反馈信息页选择反馈
  -> POST /api/feedback-sources/eval-cases/generate
  -> eval case
  -> 可在批次回归中关联、编辑、移除、归档
```

批次内用例操作：

```text
GET    /api/feedback-optimization-batches/{batch_id}/eval-cases
POST   /api/feedback-optimization-batches/{batch_id}/eval-cases
PATCH  /api/feedback-optimization-batches/{batch_id}/eval-cases/{eval_case_id}
DELETE /api/feedback-optimization-batches/{batch_id}/eval-cases/{eval_case_id}
```

长期目标：

```text
feedback-derived eval case
  -> candidate
  -> eval-case-governor 给出去重、分层、稳定性、阻断策略建议
  -> 人工审核、晋级或拒绝
  -> smoke / core_regression / safety / historical_bug / scenario_pack
  -> 后续批次自动纳入相关回归计划
```

长期回归资产页尚属于升级目标，建议作为 `反馈信息 / 优化批次 / 回归资产 / 版本管理` 中的独立侧边菜单页落地。当前已完成的底层基础是 `eval_cases`、`eval_runs`、批次关联用例和批次回归运行。

## 8. 从反馈到归因

优化批次创建后，批次内每个 feedback case 都会运行归因任务。

```text
POST /api/feedback-optimization-batches/{batch_id}/attribution-jobs
```

批次归因的输入：

1. feedback case。
2. evidence package。
3. main 智能体运行记录。
4. 当时的 main agent version。
5. 相关反馈信号、SOC 事件、人工标注。
6. 已生成或手工补充的 eval cases。

`attribution-analyzer` 输出结构化归因，重点回答：

1. 反馈是否有效。
2. 问题属于提示词、技能、工具权限、MCP、输出格式、知识缺口、任务拆解、上下文关联还是外部系统问题。
3. 哪些证据支持这个判断。
4. 修复方向是什么。
5. 是否适合变成工作区可执行任务。
6. 是否需要外部系统治理。
7. 是否需要人工复核或阻断。

归因的价值是把“用户觉得不好”转化为“工程上知道该改哪里”。没有归因的反馈不应直接驱动 main 智能体演进。

## 9. 从归因到优化批次

优化批次是多个反馈的治理单元。它解决两个问题：

1. 避免每条反馈都生成孤立、互相冲突的小修改。
2. 让同类反馈合并成一个可审计的优化计划。

批次创建：

```text
POST /api/feedback-optimization-batches
```

批次计划生成：

```text
POST /api/feedback-optimization-batches/{batch_id}/optimization-plan
```

`proposal-generator` 的输入是批次级材料：

1. 批次 source refs。
2. 各 feedback case 的 attribution outputs。
3. 关联 eval cases。
4. 目标路径策略。
5. 当前 main agent version。
6. 可选的 regeneration instruction。

它输出 batch optimization plan，计划中的任务可以分为：

| 任务类型 | 含义 | 后续动作 |
| --- | --- | --- |
| `workspace_execution` | 可以通过修改 main-workspace 受管文件完成 | 开发人员点击执行，进入 execution-optimizer |
| `external_webhook` | 应交给外部治理系统处理 | 选择 webhook alias 并通知外部系统 |
| `manual_review` | 需要人工判断或补充设计 | 停留在计划中，不自动修改 |
| `blocked` | 证据不足、目标不合法或存在风险 | 记录阻断原因，不能执行 |

计划生成后并不等于自动执行。开发人员需要在工作台中确认具体任务。

### 9.1 优化任务与三方系统的关系

优化任务并不只面对 `/main-workspace`。反馈经常指向外部 MCP server、外部知识库、SOC 流程、下游 API 或治理平台。闭环需要先判断“问题应该由谁负责改”，再决定任务类型。

三方系统在闭环中有四种角色：

| 角色 | 示例 | 与优化任务的关系 |
| --- | --- | --- |
| 反馈来源 | SOC 系统、监控系统、人工复核平台 | 通过 SOC event、feedback signal 或外部治理事件进入反馈池 |
| 证据来源 | Langfuse trace、MCP tool 调用结果、外部知识库检索结果、下游 API 错误 | 被固化进 evidence package，供归因分析读取 |
| 运行依赖 | `sec-ops-data` MCP server、`security-kb` MCP server、外部知识库 | main 智能体运行时会依赖它们；反馈可能暴露工具不可用、数据质量差或描述不清 |
| 外部治理目标 | 外部 MCP 服务、知识库内容、SOC 处置流程、三方 API | 不能由 execution-optimizer 直接修改，必须形成 external governance item 或人工复核项 |

归因输出需要区分三类边界：

| 归因边界 | 典型对象 | 应生成的任务 |
| --- | --- | --- |
| main 智能体行为包可修复 | `CLAUDE.md`、skill、subagent、output style、`.mcp.json` 中的受管配置 | `workspace_execution` |
| main 智能体使用方式可修复，但外部系统不变 | 工具选择规则、MCP 描述补充、知识库检索策略、输出格式约束 | 通常是 `workspace_execution`，修改 main-workspace 内指令、skill 或受管 MCP 配置 |
| 外部系统自身需要修复 | 外部 MCP 服务接口、知识库内容、SOC 流程、三方 API 数据质量 | `external_webhook`、`manual_review` 或 `blocked` |

判断规则：

1. 如果修复只需要改变 main 智能体如何理解、选择或调用现有工具，可以进入 `workspace_execution`。
2. 如果修复需要改外部 MCP server 的服务实现、接口 schema、数据源、知识库条目或 SOC 流程，不能进入 `workspace_execution`。
3. 如果能定位到外部系统、工具/API 和具体问题描述，应生成 `external_webhook` 任务，并创建 external governance item。
4. 如果只能泛泛判断“外部系统可能有问题”，但没有具体系统、接口、工具或问题 ID，应进入 `manual_review` 或 `blocked`。
5. `.mcp.json` 属于 main-workspace 受管行为包，允许作为 main 智能体配置变更对象；但其中不得写入 API key、MCP header 或本机私有路径。

因此，同样是 MCP 相关反馈，可能产生不同任务：

```text
MCP 工具描述不清，main 智能体误选工具
  -> workspace_execution
  -> 修改 main-workspace 中的工具使用规则、skill 或 MCP 描述

MCP server 返回字段缺失或接口报错
  -> external_webhook
  -> 通知外部 MCP 服务负责人修复服务或数据

知识库内容过期，导致回答引用旧规则
  -> external_webhook / manual_review
  -> 更新外部知识库，而不是让 main 智能体硬编码新事实

main 智能体没有在答案中标注知识库证据来源
  -> workspace_execution
  -> 修改输出规范或相关 skill
```

这层关系的核心原则是：反馈闭环可以发现并分派三方系统问题，但不能把三方系统问题伪装成 main-workspace 修改。

## 10. 从优化计划到受控执行

当前闭环的执行路径是“开发人员确认任务 + 执行智能体生成计划 + 后端受控应用”。

```text
开发人员点击任务执行
  -> prepare optimization task
  -> run_execution_job
  -> execution-optimizer 生成 execution plan
  -> ExecutionApplicationService.apply_ready_execution_job
```

执行智能体只负责输出结构化 operations，例如：

1. `create_file`
2. `replace_file`
3. `append_text`
4. `noop`

后端受控应用会检查：

1. `path` 必须是 main-workspace 下的安全相对路径。
2. 目标路径必须符合 feedback store 的 target policy。
3. `replace_file` 和 `append_text` 必须携带 `expected_sha256`。
4. 应用前实际 hash 必须匹配，防止基线漂移。
5. 单次写入不得超过 `max_write_bytes`。
6. 不支持的操作直接拒绝。
7. 当前 main agent version 必须与 execution baseline 一致。
8. 应用失败时回滚已写文件；状态同步失败时尝试恢复应用前版本。

这套设计把“智能体建议”与“系统写文件”拆开，避免执行智能体拥有直接修改主智能体行为包的权限。

## 11. main 智能体如何持续进化

main 智能体的可进化对象是 `/main-workspace` 中的受管行为包，包括：

1. `CLAUDE.md`
2. `agent.yaml`
3. `.claude/settings.json`
4. `.claude/agents/*.md`
5. `.claude/skills/*/SKILL.md`
6. `.claude/commands/*.md`
7. `.claude/rules/*`
8. `.claude/output-styles/*.md`
9. `.mcp.json`
10. `.worktreeinclude`

一次受控执行会产生至少两个关键版本点：

```text
当前版本
  -> pre_execution 快照
  -> 写入受管文件
  -> execution_optimizer_applied 快照
  -> 当前版本指向新快照
```

版本快照记录：

1. `agent_version_id`。
2. `parent_version_id`。
3. 创建原因。
4. 来源 proposal / task。
5. bundle path 与 sha256。
6. manifest。
7. 文件清单。
8. 与前一版本的 diff。

因此，main 智能体不是“自己偷偷进化”，而是通过以下闭环进化：

```text
反馈事实
  -> 归因判断
  -> 批次方案
  -> 人工确认
  -> 结构化执行计划
  -> 后端受控应用
  -> main-workspace 新版本
  -> 回归验证
```

## 12. 回归验证如何闭环

回归验证有三种层次。

### 12.1 批次回归

```text
POST /api/feedback-optimization-batches/{batch_id}/regression-runs
```

运行前置条件：

1. 批次已经产生 optimization task。
2. optimization task 已经应用并产生 `applied_agent_version_id`。
3. 批次存在可运行的 eval cases。

批次回归会调用当前 main 智能体运行关联用例，并记录 eval run。

长期升级后，批次回归不会只停在 eval run 结果展示，而会继续进入：

```text
eval run
  -> gate result
  -> regression-impact-analyzer 解释失败影响
  -> 发布 / 阻断 / 修复 / 更新用例 / 回滚建议
```

### 12.2 手工数据集评估

```text
POST /api/eval-runs
```

用途：

1. 对当前 main 智能体运行一组反馈衍生用例。
2. 在没有特定批次时复核当前版本质量。
3. 为版本发布、回滚或人工验收提供证据。

### 12.3 长期回归资产

长期升级目标是让历史高价值反馈不只服务当前批次，而是成为所有后续优化的防复发资产。

```text
历史反馈 A 产生 historical_bug 用例
  -> eval-case-governor 建议晋级为长期资产
  -> 人工确认 asset layer 和 blocking policy
  -> 批次 1 修复并通过
  -> 批次 2 修改相关能力时自动纳入回归计划
  -> 如果 A 失败，regression-impact-analyzer 判断真实退化、用例过时或 flaky
  -> 按 gate result 阻断、重跑、修复、更新用例或回滚
```

长期资产需要补充：

1. 用例分层。
2. 晋级状态。
3. 阻断策略。
4. 修订历史。
5. 相关性选择。
6. flaky 标记。
7. 失败影响分析。
8. 用例治理输出。
9. 回归影响分析输出。

## 14. 状态流转全景

### 14.1 反馈状态

```text
collected
  -> correlated / pending_correlation
  -> selected
  -> feedback case
  -> in_batch
  -> attributed
  -> planned
  -> executed
  -> regressed
  -> promoted_to_asset / archived
```

### 14.2 批次状态

```text
created
  -> attribution_running
  -> attribution_ready
  -> plan_pending
  -> plan_ready
  -> task_executing
  -> applied
  -> regression_running
  -> regression_passed / regression_failed / review_required
```

实际存储字段可能比这张概念图更细；这里表达的是业务状态。

### 14.3 版本状态

```text
bootstrap
  -> current
  -> pre_execution
  -> execution_optimizer_applied
  -> current
  -> pre_restore
  -> rollback
```

版本链路用于回答三个问题：

1. 哪个反馈或优化任务导致了这个版本。
2. 这个版本改了哪些 main-workspace 文件。
3. 如果回归失败，能否恢复到之前的版本。

### 14.4 用例资产治理状态

长期回归资产升级后，用例不应从生成直接变成阻断资产。`promotion_status` 的权威取值为：

```text
candidate
  -> needs_review
  -> approved / rejected / superseded / archived
```

`eval-case-governor` 只负责生成治理建议，不新增持久化中间态。人工或明确治理规则负责把候选用例推进到 `needs_review`、`approved`、`rejected`、`superseded` 或 `archived`。

治理建议中的 `decision=merge` 不落库为独立状态；被合并用例落为 `promotion_status=superseded`，并记录 `superseded_by_eval_case_id` 指向保留用例。

### 14.5 回归门禁状态

长期回归资产升级后，回归运行还需要形成门禁状态：

```text
plan_created
  -> eval_running
  -> eval_completed
  -> impact_analyzing
  -> gate_passed / gate_blocked / review_required
```

`regression-impact-analyzer` 只解释失败影响和处置建议；最终门禁结果必须由 `gate_policy`、阻断策略和人工确认共同决定。

## 17. 安全与治理门禁

反馈闭环必须守住以下硬边界。

### 17.1 采集边界

```text
采集接口只能采集，不能顺手归因、生成方案或写文件。
```

原因：

1. 反馈可能无效。
2. 反馈可能缺少运行关联。
3. 隐式事件可能需要人工解释。
4. 自动执行会把噪声放大成主智能体行为漂移。

### 17.2 智能体边界

```text
分析智能体和方案智能体只输出结构化结果。
执行智能体只输出结构化 operations。
用例治理智能体只输出治理建议。
回归影响分析智能体只输出影响分析和发布建议。
后端才有受控应用权限。
```

原因：

1. 便于审计。
2. 便于复现。
3. 便于校验路径和 hash。
4. 便于失败回滚和补偿。

### 17.3 版本边界

```text
每次 main-workspace 行为变更必须落到 AgentVersionStore。
```

原因：

1. 知道当前 main 智能体到底是哪一版。
2. 反馈和回归结果可关联到具体版本。
3. 回归失败时可以恢复。
4. 优化历史可以追溯。

### 17.4 回归边界

```text
已应用优化不等于可发布。
必须经过与风险匹配的回归验证。
```

当前批次至少需要跑批次关联 eval cases。长期目标是按回归资产策略自动构建 regression plan。
长期升级后，regression-impact-analyzer 的结论不能替代门禁规则本身；它只能解释失败原因和建议处置，最终发布状态仍应由 gate result、阻断策略和人工确认共同决定。

## 19. 关键问题的回答

### 19.1 用户如何反馈？

用户在 main 智能体回答后通过 Playground 或业务页面提交显式反馈。外部系统也可以通过 SOC event 或治理 webhook 输入隐式反馈。反馈必须携带或后续补齐 `run_id`、`session_id`、`case_id`、`alert_id` 等关联信息。

### 19.2 系统如何收集反馈信息？

Runtime 通过 `/api/feedback-signals`、`/api/soc-events`、`/api/pending-correlations`、`/api/feedback-sources` 等接口收集和统一展示。采集只入库，不自动归因或执行。反馈与 agent run、版本、业务 case 和 trace 建立关联后，才进入后续治理。

### 19.3 反馈如何变成优化？

开发人员从反馈信息页选择反馈创建优化批次。批次运行归因分析，方案生成智能体汇总归因输出生成优化计划。计划中的 workspace task 需要开发人员点击执行，才会进入 execution-optimizer 和后端受控应用。

### 19.4 main 智能体如何持续进化？

main 智能体通过受控修改 `/main-workspace` 的行为包持续进化。每次应用都会创建版本快照，并把任务、diff、版本、回归结果关联起来。它不是自动自我修改，而是由反馈事实驱动、由多智能体辅助、由后端门禁执行、由回归验证兜底的演进。

### 19.5 如何防止改好一个问题又引入旧问题？

当前能力是批次关联 eval cases 并运行批次回归。长期目标是引入 `eval-case-governor` 和 `regression-impact-analyzer`，把高价值反馈沉淀为长期回归资产，让后续批次按相关性和阻断策略持续运行历史用例，并在失败时判断真实退化、用例过时、flaky 或历史问题复发，形成防复发网络。

### 19.6 优化任务和外部 MCP server、外部知识库是什么关系？

优化任务首先判断责任边界。如果问题可以通过 main 智能体行为包修复，例如补充工具选择规则、MCP 描述、skill 或输出规范，就生成 `workspace_execution`，由 execution-optimizer 生成结构化 operations，再由后端受控修改 `/main-workspace`。如果问题在外部 MCP server、外部知识库、SOC 流程或三方 API 本身，就生成 `external_webhook`、`manual_review` 或 `blocked`，通过 external governance item 通知外部系统或要求人工处理。

## 20. 闭环验收标准

一次完整的反馈闭环至少应满足：

1. 反馈能追溯到明确的 agent run 或 pending correlation 处理记录。
2. 反馈进入批次前有统一来源和人工选择动作。
3. 归因输出能解释问题、证据和修复方向。
4. 优化方案能区分 workspace task、external webhook task、manual review 和 blocked item。
5. workspace task 的执行有开发人员确认动作。
6. execution-optimizer 不直接修改 main-workspace。
7. 后端应用时校验路径、hash、操作类型、大小和版本基线。
8. 应用前后都有 main 智能体版本快照。
9. 优化结果能运行对应回归用例。
10. 回归结果能反向标记批次和版本是否可继续发布。
11. 长期升级后，高价值反馈能经过 eval-case-governor 建议和人工审核转为长期回归资产，参与后续批次。
12. 长期升级后，回归失败能经过 regression-impact-analyzer 解释真实退化、用例过时、flaky 或历史问题复发，并形成可审计处置建议。
13. 涉及外部 MCP server、外部知识库、SOC 流程或三方 API 的问题能被正确分派为 workspace task、external webhook task、manual review 或 blocked item。
14. 失败、阻断、外部治理和人工复核都有可审计记录。

## 21. 最终全景

用最压缩的方式表示，完整反馈闭环是：

```text
如何反馈
  -> 显式 feedback signal / 隐式 SOC event / 外部治理事件 / 人工标注

如何收集
  -> agent run
  -> feedback source
  -> pending correlation
  -> feedback case
  -> evidence package

如何理解反馈
  -> attribution-analyzer
  -> attribution output
  -> 问题根因、证据、风险、修复方向

如何转为优化
  -> optimization batch
  -> proposal-generator
  -> batch optimization plan
  -> workspace task / external webhook / manual review / blocked item

如何处理三方系统
  -> main-workspace 可修复的问题进入 workspace task
  -> 外部 MCP server / 外部知识库 / SOC 流程 / 三方 API 问题进入 external governance 或人工复核

如何执行优化
  -> 开发人员确认任务
  -> execution-optimizer
  -> structured operations
  -> 后端受控应用
  -> main-workspace 文件变更

如何让 main 智能体进化
  -> pre_execution snapshot
  -> execution_optimizer_applied snapshot
  -> agent version diff
  -> 当前 main agent version 更新

如何验证
  -> batch regression run
  -> eval run items
  -> 通过 / 失败 / 阻断 / 人工复核

如何长期持续进化
  -> feedback-derived eval case
  -> eval-case-governor 治理建议
  -> 人工审核后进入长期回归资产
  -> 后续批次回归计划
  -> regression-impact-analyzer 回归影响分析
  -> 防复发
  -> 新反馈继续进入闭环
```

这就是当前反馈闭环的完整画像：反馈不是终点，反馈是 main 智能体版本演进、质量门禁和长期回归资产的起点。
