# 反馈闭环长期回归资产升级方案

> 用途：定义反馈闭环从“批次内回归验证”升级为“长期回归资产沉淀与持续防复发”的产品、架构和项目改造方案。  
> 状态：设计方案稿  
> 日期：2026-06-01

本文档面向 `claude-agent-runtime` 当前反馈优化闭环，描述如何把反馈生成的测试用例从当前优化批次的临时验证材料，升级为后续所有优化批次持续使用的长期回归资产。

核心结论：

```text
反馈信息
  -> 候选测试用例
  -> 用例治理
  -> 长期回归资产
  -> 后续优化批次分层回归
  -> 回归影响分析
  -> 发布 / 阻断 / 修复 / 用例库反哺
```

长期目标不是“每次多跑一些测试”，而是让每一条高价值反馈都沉淀为系统未来不会再次复发的能力保护网。

---

## 1. 背景与问题

当前反馈优化闭环已经具备以下基础能力：

```text
反馈信息
  -> 默认回归用例
  -> 优化批次
  -> 归因分析智能体
  -> 优化方案生成智能体
  -> 执行优化智能体
  -> 批次回归测试
```

现有实现中已经存在：

1. `eval_cases`：反馈生成或手工创建的评估用例。
2. `eval_runs` / `eval_run_items`：回归运行及逐条结果。
3. `feedback_optimization_batches.eval_case_ids`：批次关联用例。
4. `FeedbackEvalRunner`：按 eval case 调用主智能体并记录结果。
5. 优化批次详情中的“回归测试”区域：执行当前批次关联用例。

但当前模型仍更接近：

```text
当前批次选择了哪些 eval_case_ids
  -> 应用优化后跑这些用例
  -> 得到本批次结果
```

缺口在于：反馈生成的用例还没有被明确建模为长期回归资产。这样会出现典型退化：

```text
批次 1 修好了 A 问题，并通过 A 对应测试。
批次 2 为了解决 B 问题压缩输出模板，又把 A 的证据来源或置信度删掉。
```

如果 A 用例只绑定在批次 1 上，批次 2 不持续运行它，旧问题就会复发而不被发现。

---

## 2. 设计目标

本次升级的目标是建立“反馈 -> 用例 -> 优化 -> 回归 -> 防复发”的完整闭环。

### 2.1 产品目标

1. 反馈生成的高价值用例可以沉淀为长期资产。
2. 每个后续优化批次都能按策略选择相关历史用例持续验证。
3. 回归失败时能明确判断是历史问题复发、当前批次目标失败、用例过时，还是 flaky。
4. 开发人员可以在统一页面治理长期回归资产。
5. 批次发布前有明确阻断规则，不依赖人工临场判断。

### 2.2 工程目标

1. `SQLite eval_cases` 继续作为长期回归用例的权威存储。
2. 不把 `docker/volume/main-workspace/evals/` 作为权威用例库；它可以保留为主智能体 workspace 中的文件资产或执行优化结果。
3. 用例编辑必须保留 revision，使历史运行结果能追溯当时的用例内容。
4. 回归计划、门禁结果、影响分析结果必须可审计。
5. 新增智能体必须符合现有 Runtime Profile 隔离模型，不共享 workspace 和 claude root。
6. 必需工作流继续支持离线模式；没有外部模型配置时要有可解释的 deterministic/offline fallback。

---

## 3. 新闭环流程

升级后的正式链路为：

```text
反馈信息
  ↓
候选测试用例生成
  ↓
用例治理智能体分析
  ↓
人工审核 / 去重 / 分层 / 晋级
  ↓
长期回归资产库
  ↓
优化批次创建
  ↓
归因分析智能体
  ↓
优化方案生成智能体
  ↓
回归计划生成
  ↓
执行优化智能体
  ↓
当前批次专项测试
  ↓
分层历史回归测试
  ↓
回归影响分析智能体
  ↓
发布 / 阻断 / 继续优化 / 更新用例库
```

这里有两个核心变化：

1. 用例在进入优化批次前，先成为“候选资产”，经治理后才进入长期回归集。
2. 批次回归不再只跑 `batch.eval_case_ids`，而是跑一个可审计的 `regression_plan`。

## 4. 用例资产模型

### 4.1 权威存储

长期回归用例的权威存储为 SQLite 中的 `eval_cases` 及相关表。

选择 SQLite 主存的原因：

1. 当前系统已经围绕 `eval_cases` / `eval_runs` 建立 API、前端和回归运行链路。
2. 批次、反馈、运行结果、Agent 版本均在 SQLite 中有可关联 ID。
3. 资产治理需要过滤、排序、状态管理、修订历史和运行统计，数据库更适合作为主存。
4. 文件用例容易引入 DB 与文件双写漂移，不适合作为 v1 权威来源。

### 4.2 用例层级

每条 eval case 增加 `asset_layer`，用于表达它在长期回归资产库中的位置。

| 层级 | 含义 | 默认运行策略 |
| --- | --- | --- |
| `candidate` | 新生成、待治理的候选用例 | 不进入发布门禁 |
| `batch_specific` | 当前批次专项用例 | 当前批次必须跑 |
| `smoke` | 冒烟用例，数量少，覆盖基础能力 | 每个批次必须跑 |
| `core_regression` | 核心能力回归 | 每个非小改动批次必须跑 |
| `scenario_pack` | 场景包，如告警研判、漏洞查询、报告生成 | 相关批次选择性跑 |
| `safety` | 安全边界，如越权、危险操作、权限策略 | 安全相关改动必须跑，发布前必须跑 |
| `historical_bug` | 历史反馈沉淀的防复发用例 | 相关批次必须跑，高危项阻断 |
| `exploratory` | 边界探索、实验性用例 | 不默认阻断 |

### 4.3 晋级状态

每条用例增加 `promotion_status`：

| 状态 | 含义 |
| --- | --- |
| `candidate` | 刚生成，尚未审核 |
| `needs_review` | 治理智能体或规则认为需要人工确认 |
| `approved` | 已确认进入长期回归资产 |
| `rejected` | 不进入长期资产 |
| `superseded` | 被更好的用例替代 |
| `archived` | 已归档，不参与自动选择 |

默认策略：

```text
反馈生成的用例默认进入 candidate。
人工确认后才能进入 smoke/core_regression/scenario_pack/safety/historical_bug。
```

这样可以避免反馈用例无脑进入长期集，导致膨胀、重复、冲突和脆弱。

### 4.4 阻断策略

每条用例增加 `blocking_policy`：

| 策略 | 含义 |
| --- | --- |
| `blocking` | 失败必须阻断发布 |
| `blocking_if_relevant` | 当前批次命中相关性时失败阻断 |
| `review_required` | 失败后需要人工判断 |
| `non_blocking` | 失败只记录，不阻断 |
| `flaky_non_blocking` | 已标记 flaky，不直接阻断 |

推荐默认：

```text
safety -> blocking
smoke -> blocking
core_regression -> blocking
historical_bug(high) -> blocking_if_relevant
scenario_pack -> blocking_if_relevant
batch_specific -> blocking
exploratory -> non_blocking
candidate -> non_blocking
```

### 4.5 防复发目标

每条长期用例必须记录 `regression_goal`。

示例：

```json
{
  "eval_case_id": "evc-...",
  "asset_layer": "historical_bug",
  "source_feedback_case_id": "fbc-...",
  "origin_batch_id": "fob-...",
  "scenario_pack": "alert_triage",
  "bug_type": "missing_evidence_chain",
  "regression_goal": "防止 Agent 在横向移动分析中遗漏进程链、证据来源和置信度。",
  "severity": "high",
  "blocking_policy": "blocking_if_relevant",
  "promotion_status": "approved"
}
```

回归失败时，系统应该展示：

```text
历史问题复发：横向移动分析遗漏证据来源和置信度。
来源反馈：fbc-...
首次修复批次：fob-...
当前失败批次：fob-...
```

而不是只展示“某条测试 failed”。

## 5. 用例修订与历史追溯

当前实现允许直接覆盖 `prompt`、`expected_behavior`、`checks_json`、`labels`、`status`。长期资产化后，直接覆盖会带来审计问题：历史 eval_run 看到的是旧内容还是新内容不明确。

因此需要引入 revision。

### 5.1 修订规则

以下字段变更必须创建 revision：

```text
prompt
expected_behavior
checks_json
labels
asset_layer
blocking_policy
scenario_pack
regression_goal
severity
promotion_status
```

`eval_cases` 保存当前版本字段，`eval_case_revisions` 保存历史快照。

### 5.2 建议新增表

```text
eval_case_revisions
  revision_id
  eval_case_id
  revision_number
  created_at
  created_by
  change_reason
  content_hash
  snapshot_json
```

`snapshot_json` 至少包含：

```text
prompt
expected_behavior
checks_json
labels
asset metadata
source metadata
```

### 5.3 运行结果快照

`eval_run_items.payload_json` 应固化：

```text
eval_case_revision_id
eval_case_snapshot
regression_goal
asset_layer
blocking_policy
scenario_pack
```

这样即使用例后续被编辑，历史运行也能还原当时判断依据。

## 6. 回归计划与门禁

### 6.1 回归计划

批次回归前生成 `regression_plan`，不再只使用批次创建时的 `eval_case_ids`。

`regression_plan` 包含：

```text
plan_id
batch_id
created_at
selection_reason
case_groups
required_case_ids
optional_case_ids
blocked_case_ids
gate_policy
```

用例分组建议：

```text
current_batch_cases
smoke_cases
core_cases
safety_cases
relevant_historical_cases
scenario_pack_cases
exploratory_cases
```

### 6.2 选择策略

按批次改动影响选择回归层：

| 改动类型 | 必跑用例 |
| --- | --- |
| 小型文案或局部 skill 调整 | `smoke` + `batch_specific` + 相关 `historical_bug` |
| 主智能体指令改动 | `smoke` + `core_regression` + 相关 `historical_bug` |
| skill 改动 | `smoke` + 对应 `scenario_pack` + 相关 `historical_bug` |
| MCP 配置或工具描述改动 | `smoke` + 相关工具场景包 + `safety` 子集 |
| 权限、安全、外部调用策略改动 | `safety` + `core_regression` + 相关历史用例 |
| 发布前验证 | `smoke` + `core_regression` + `safety` + 相关场景包 + 高危历史用例 |

### 6.3 阻断规则

回归运行完成后生成 `gate_result`。

| 失败类型 | 是否阻断 |
| --- | --- |
| `safety` 用例失败 | 必须阻断 |
| `smoke` 用例失败 | 必须阻断 |
| 当前批次专项用例失败 | 必须阻断 |
| 高危历史问题复发 | 必须阻断 |
| 相关 `core_regression` 失败 | 默认阻断 |
| 相关 `scenario_pack` 失败 | 按 `blocking_policy` 判断 |
| `exploratory` 失败 | 不直接阻断 |
| 已标记 flaky 的用例失败 | 不直接阻断，但记录 |
| 用例过时 | 进入用例治理，不直接发布通过 |

`gate_result` 示例：

```json
{
  "status": "blocked",
  "summary": {
    "total": 18,
    "passed": 15,
    "failed": 3,
    "blocking_failed": 2,
    "non_blocking_failed": 1
  },
  "blocking_failures": [
    {
      "eval_case_id": "evc-001",
      "asset_layer": "historical_bug",
      "regression_goal": "防止遗漏证据来源和置信度",
      "reason": "高危历史问题复发"
    }
  ],
  "non_blocking_failures": [
    {
      "eval_case_id": "evc-009",
      "asset_layer": "exploratory",
      "reason": "探索用例失败，不阻断"
    }
  ]
}
```

## 7. 新增智能体

### 7.1 用例治理智能体：`eval-case-governor`

职责：

1. 判断候选用例是否值得进入长期回归资产。
2. 检测与已有用例的重复、冲突和覆盖重叠。
3. 建议用例层级、场景包、严重度和阻断策略。
4. 判断用例是否稳定、是否可自动判分、是否需要人工审核。
5. 给出变体、反例、干扰用例建议，降低过拟合。

输入：

```text
candidate_eval_case
source_feedback
feedback_case
attribution_summary
proposal_summary
similar_eval_cases
recent_eval_run_stats
```

输出 schema 建议：

```text
eval-case-governance-output/v1
```

关键字段：

```text
decision: promote | keep_candidate | reject | merge | needs_human_review
recommended_layer
recommended_scenario_pack
recommended_blocking_policy
severity
regression_goal
duplicate_of
conflicts
stability_risk
judgeability
recommended_variants
human_review_reason
```

治理智能体只给建议，不自动把用例放入长期阻断层。进入长期阻断层必须有人确认。

### 7.2 回归影响分析智能体：`regression-impact-analyzer`

职责：

1. 分析批次回归失败是否与本批次改动相关。
2. 判断失败是真实退化、用例过时、判分规则不合理还是 flaky。
3. 识别历史反馈问题是否复发。
4. 给出发布建议：阻断、重跑、修复、更新用例、回滚。
5. 产出可展示给开发人员的失败归因摘要。

输入：

```text
batch
optimization_plan
execution_result
regression_plan
eval_run
failed_eval_run_items
eval_case_snapshots
agent_version_diff
```

输出 schema 建议：

```text
regression-impact-analysis-output/v1
```

关键字段：

```text
decision: block_release | allow_with_notes | rerun | update_cases | rollback_or_fix
failure_classifications
related_to_batch_change
historical_regression_recurred
case_update_recommendations
batch_fix_recommendations
release_note
```

## 9. 后端 API 调整

### 9.1 Eval Case 资产治理

新增或扩展：

```text
GET    /api/eval-cases
GET    /api/eval-cases/{eval_case_id}
PATCH  /api/eval-cases/{eval_case_id}
GET    /api/eval-cases/{eval_case_id}/revisions
POST   /api/eval-cases/{eval_case_id}/governance-review
POST   /api/eval-cases/{eval_case_id}/promote
POST   /api/eval-cases/{eval_case_id}/archive
POST   /api/eval-cases/{eval_case_id}/mark-flaky
POST   /api/eval-cases/{eval_case_id}/unmark-flaky
```

`PATCH` 长期用例时必须创建 revision。

### 9.2 回归计划

新增：

```text
POST /api/feedback-optimization-batches/{batch_id}/regression-plan
GET  /api/feedback-optimization-batches/{batch_id}/regression-plan
```

批次运行回归时：

```text
POST /api/feedback-optimization-batches/{batch_id}/regression-runs
```

行为调整为：

1. 如果请求未指定 `regression_plan_id`，先生成或复用最新 plan。
2. 按 plan 中的 active 用例执行。
3. 固化用例 revision 和快照。
4. 计算 `gate_result`。
5. 必要时触发或允许手动触发回归影响分析。

### 9.3 回归影响分析

新增：

```text
POST /api/eval-runs/{eval_run_id}/impact-analysis
GET  /api/eval-runs/{eval_run_id}/impact-analysis
```

也可以在批次回归完成后自动创建分析 job，但发布门禁不能依赖外部在线服务。离线模式下必须有 deterministic fallback：

```text
有 blocking failure -> blocked
仅 non-blocking failure -> review_required 或 passed_with_notes
无 failure -> passed
```


### 10.3 迁移机制

当前 `runtime_db.py` 主要使用 `Base.metadata.create_all(engine)` 和 `schema_migrations` 记录初始版本。`create_all` 不会为已有表自动添加新列。

因此本次真正落地前必须补一层轻量迁移机制：

1. 在 `schema_migrations` 中登记每个迁移版本。
2. 启动时按版本执行幂等 `ALTER TABLE` / `CREATE TABLE`。
3. 每个迁移必须可重复检测，不依赖外部服务。
4. 测试覆盖从旧库升级到新 schema 的路径。

这是本项目需要先改造的基础设施之一。

## 12. Job 编排改造

当前 feedback job 编排已经包含 attribution、proposal、batch plan、execution 等类似流程。新增两个智能体时，如果继续复制现有模式，会扩大重复代码。

因此在实现新增智能体前，应先抽出统一 job spec：

```text
JobSpec
  job_type
  profile_name
  expected_schema_version
  build_input
  build_prompt
  complete_handler
  fail_handler
  offline_output
  result_loader
```

目标：

1. 新增智能体只注册 spec，不复制一整段 create/start/run/complete/fail 模板。
2. 统一处理超时、schema 校验、offline fallback、raw output、validated output。
3. 避免 `feedback_job_orchestrator.py` 继续变成多分支模板聚合点。

这是本次项目改造的结构前置项。

## 13. Schema 与判分能力改造

### 13.1 Eval Case Schema



注意：不要一次性引入复杂判分 DSL。v1 只扩展最常用、可确定性执行的检查项。

### 13.2 防过拟合

```text

```

例如“沙箱误报内部工具”反馈不应只生成固定样本，还应覆盖：

```text
沙箱恶意 + 白名单命中 -> 降低置信度
沙箱恶意 + EDR 恶意行为 -> 提高置信度
沙箱恶意 + 无其他证据 -> 只能说疑似
沙箱干净 + EDR 横向移动 -> 不应被沙箱低风险结论误导
```

## 18. 风险与约束

### 18.1 用例膨胀

风险：所有反馈默认进入长期回归集会导致用例数量膨胀。

约束：

```text
默认 candidate，人工确认后晋级。
```

### 18.2 用例过拟合

风险：Agent 只学会通过固定样本。

约束：

```text
高价值反馈必须考虑变体、反例和干扰样本。
```

### 18.3 双写漂移

风险：DB 与 `evals/` 文件同时作为权威来源会漂移。

约束：

```text
SQLite 为权威；文件只作为 workspace 资产或导出视图。
```

### 18.4 发布门禁依赖智能体

风险：模型不可用导致门禁无法判断。

约束：

```text
阻断门禁必须有 deterministic fallback。
```

### 18.5 编排代码重复

风险：新增两个智能体后继续复制 job 编排模板。

约束：

```text
新增智能体前先抽象 job spec。
```

## 19. 最终验收标准

升级完成后，应满足：

1. 反馈生成的测试用例不会只服务当前批次。
2. 高价值反馈可以人工确认晋级为长期回归资产。
3. 每个后续优化批次能自动选择相关历史用例。
4. 批次回归有明确 `gate_result`。
5. 历史高危问题复发会阻断发布。
6. flaky、探索和过时用例不会无条件阻断，但会进入治理。
7. 开发人员能在“回归资产”页统一管理长期用例库。
8. 历史运行结果能追溯当时的用例 revision。
9. 新增智能体符合现有 profile 隔离和离线不变量。
10. 回归资产越积累，后续优化越不容易破坏已修复能力。
