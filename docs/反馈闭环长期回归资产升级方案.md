# 反馈闭环长期回归治理升级方案

> 文档定位：typed TestDataset、EvalRun、发布门禁和 Asset Registry 的长期设计。
> 文档层级：当前实现基线到四阶段改进治理的测试发布桥接方案。
> 替代关系：旧完整稿已归档到 `docs/archive/obsolete/反馈闭环长期回归资产升级方案_旧完整稿.md`；旧稿中的全局 EvalCase、独立回归资产、场景包、批次 UI 和双轨 schema 不再作为活跃依据。
> UI 边界：工作台布局、决策卡、处理记录和视觉验收以 `docs/AgentGov_四阶段改进治理工作台UI整改方案.md` 为准。本文只定义测试资产与发布门禁。

## 1. 目标

长期回归治理解决四个问题：

1. 反馈形成的回归候选必须能沉淀为有归属、版本、来源和生命周期的 typed 测试资产。
2. 一次评估运行必须固化数据集修订和完整用例快照，不能在运行中扫描全局用例池。
3. 发布门禁必须关联数据集覆盖、失败原因、人工决策和候选版本，不能接受客户端伪造结果。
4. Asset Registry 必须投影 typed 资产及 provenance，不能复制第二套 ID、body schema 或生命周期。

## 2. 单一真相模型

| 对象 | 所有权与职责 | 明确不负责 |
| --- | --- | --- |
| `RegressionAssessment` | 四阶段“测试发布”业务产物；提供 `eval_cases` 候选、无动作原因和建议门禁阈值。 | 不持久化全局用例实体，不拥有 prompt、系统 ID、状态或时间戳。 |
| `TestDataset` | 回归测试资产；保存业务 Agent 归属、事项证据链、修订、生命周期、候选版本和有序 typed cases。 | 不序列化为通用 `AssetRecord.body`，不与第二套回归资产实体双写。 |
| `EvalRun` | 一次候选版本评估；固化 dataset revision、完整 case snapshot、执行结果和状态。 | 不接受全局用例 ID 或客户端提交的执行结果。 |
| `ReleaseGate` | 基于持久化 EvalRun 和 change set/release 事实给出发布风险判断。 | 不重新执行用例，不从 UI 临时状态推断放行。 |
| `AssetRegistry` | 统一查询和 provenance 视图。 | 不成为 TestDataset 的权威写入 store。 |

## 3. 当前领域契约

### 3.1 RegressionAssessment 候选

治理 Agent 只输出每项的 `expected_behavior`、`checks_json` 和 `labels`。原始复测输入、业务 Agent、反馈、事项、job、scope、标识、生命周期和时间戳均由后端绑定或覆盖。

`eval_cases` 是 RegressionAssessment 内的业务字段，不是旧全局 EvalCase API 或持久化实体的兼容入口。证据不足时输出 `no_action_reason`，不得编造 prompt 或 provenance。

### 3.2 TestDataset

TestDataset 是可版本化长期测试资产，最小契约包括：

| 字段 | 含义 |
| --- | --- |
| `dataset_id` | 稳定 ID，也是 Registry 投影使用的唯一数据集 ID。 |
| `agent_id` / owner | 所属业务 Agent；跨 Agent 采用和执行必须 fail closed。 |
| `name` / `description` / `scope` | 面向用户和治理流程的可读语义。 |
| `revision` | 生命周期 CAS 修订号。 |
| `cases` | 有稳定 case ID、位置、prompt、预期行为和检查点的有序集合。 |
| `provenance` | feedback、attribution、optimization、execution、change set 和 candidate version 证据链。 |
| `lifecycle_state` | `draft`、`active`、`evaluating`、`deprecated` 或 `archived`。 |
| `quality_tags` | 数据集级质量和风险标签。 |
| `created_at` / `updated_at` | 审计时间。 |

采用入口只能从已确认并且依赖关系一致的 Improvement 证据链派生这些 backend-owned 字段。客户端空请求体不能覆盖 Agent 归属、provenance、候选版本或 case 输入。

### 3.3 EvalRun

EvalRun 启动时必须在同一事务内验证并绑定：

- 数据集属于目标业务 Agent、状态为 active 且包含用例；
- 候选 Agent version 与 change set、execution 和数据集 provenance 一致；
- dataset revision 和完整有序 case snapshot 已持久化；
- 每个 item 只对应一个 snapshot case，并受到并发完成 fencing；
- 完成回归和发布门禁只读取持久化 `eval_run_id`。
- 自然语言期望和检查点不是机器断言；未提供可验证的结构化断言时只能进入人工复核，禁止以回答非空、无 runtime error 代替业务正确性。
- 候选回归的人工复核是独立 `regression_review_required` 状态和专用业务动作，不复用 generic force publish。请求必须完整覆盖当前 gate 的 review case，携带幂等 ID、操作人、理由和固定 scope；required check 失败不能被复核放行。
- 复核只更新有效门禁投影和 typed review decision，原始 EvalRunItem、answer/check 证据保持不变；EvalRun、ChangeSet 和事件审计必须同事务 CAS。
- 请求取消或任务中断必须先终结真实 EvalRun，再将同一持久化证据投影到 change set；创建前失败只记录 typed regression error，不构造伪 EvalRun。

数据集后续编辑或生命周期变化不得回写历史运行快照。

### 3.4 ReleaseGate

ReleaseGate 的目标契约至少包含：

- 使用的数据集、修订、覆盖范围和候选版本；
- 通过、失败、阻断和需要人工复核的状态；
- 失败项影响分析和回到优化执行的原因；
- 是否允许发布、需要补测或拒绝发布；
- 审批人、理由和审计记录。

当前发布链已要求绑定的持久化回归证据，并已实现自然语言语义待复核的逐 case 审计闭环。该复核不是失败豁免；对 required check 失败、runtime error 或人工拒绝仍只能修复重跑或走独立强制发布风险动作。更完整的豁免有效期和 ReleaseGate 风险解释仍是后续能力。

## 4. TestDataset 生命周期

```text
draft
  -> active -> evaluating -> active
       |           |
       +-------> deprecated -> active
       |           |
       +-----------+-------> archived
  -> archived
```

- `archived` 是终态。
- `evaluating` 只能由 EvalRun 原子进入；完成、失败、取消或 heartbeat 租约过期后的 reconciliation 会按运行 owner fencing 返回 active，UI 不提供手工进入入口；新进程启动不得失败化仍在续租的 EvalRun。
- 生命周期写入必须携带 expected revision；冲突不得覆盖。
- 归属、来源依赖或候选版本漂移时，不得采用、执行或发布。
- 同一事项返工产生不同 execution / 回归评估证据链时，重新采用生成新的确定性 dataset 版本；旧版本和已启动 EvalRun 的快照保持不可变。
- flaky、security、critical 等语义使用显式质量标签或门禁证据，不再创建独立旧资产状态机。

## 5. 公开 API

测试发布主路径只保留：

- `POST /api/improvements/{improvement_id}/test-dataset/adopt`
- `GET /api/test-datasets`
- `GET /api/test-datasets/{dataset_id}`
- `GET /api/test-datasets/{dataset_id}/revisions`
- `POST /api/test-datasets/{dataset_id}/lifecycle`
- `POST /api/agent-change-sets/{change_set_id}/regression-runs`
- `POST /api/agent-change-sets/{change_set_id}/regression-runs/{eval_run_id}/review`
- `POST /api/eval-runs`
- `GET /api/eval-runs`
- `GET /api/eval-runs/{eval_run_id}`

旧全局用例同步与修改、反馈来源直接生成用例、独立回归资产治理和场景包路由已从 FastAPI、OpenAPI、前端生成类型和主流程删除。

## 6. 历史数据迁移

SQLite migration 0040 对旧评测链执行一次原子迁移：

1. 以完整 JSON 行写入 `archived_legacy_evaluation_rows`，保留来源表、主键、归档时间和原因。
2. 归档并删除通用 Registry 中 `asset_type=regression` 的旧独立回归资产，以及旧评测生成 Agent jobs。
3. 删除旧全局用例、修订、治理事件、场景包和历史运行归档表。
4. 保留其他通用 Registry 资产、typed TestDataset、EvalRun 和 EvalRunItem 表及外键。

迁移使用显式 SQLite write transaction，DDL、归档和删除任一步失败都必须整体回滚；重复执行不得复制归档记录。迁移完成后不提供双读、双写或兼容 facade。

## 7. Asset Registry 关系

```text
Feedback / Run
  -> Attribution / OptimizationPlan
  -> ExecutionRecord / ChangeSet
  -> RegressionAssessment
  -> TestDataset
  -> EvalRun
  -> ReleaseGate
  -> Release
  -> AssetRegistry projection
```

Registry 必须能回答数据集来自哪个反馈和运行事实、保护哪个业务 Agent 和候选版本、在哪些评估与发布中起过作用。Registry 只建立查询关系，不复制 TestDataset 内容和生命周期。

## 8. 后续演进原则

1. 后端 store、API response、OpenAPI 和前端类型继续由同一 typed schema 派生。
2. 跨数据集复用应扩展 TestDataset provenance 或建立明确引用实体，不能恢复旧全局用例池。
3. 跨 Agent 方法论复用应单独证明适用范围、风险和逐 Agent 验证边界，不能恢复已删除场景包 facade。
4. 豁免审批必须绑定具体 EvalRun、失败项、审批人和有效范围。
5. UI 只在“测试发布”阶段展示数据集、运行和门禁，不恢复独立旧资产主菜单。

## 9. 验收断言

- TestDataset 有唯一 ID、业务 Agent 归属、修订、来源、生命周期和审计字段。
- RegressionAssessment 的 `eval_cases` 不产生全局用例表或旧生成 job。
- EvalRun 只执行持久化 TestDataset 的不可变完整快照。
- OpenAPI 和前端生成类型不含已删除旧路由与 schema。
- migration 0040 覆盖完整归档、幂等、DDL/归档回滚和 fresh schema。
- 发布门禁只消费与候选 change set 精确绑定的持久化 EvalRun。

## 10. 结论

TestDataset 是回归测试资产的唯一 typed 真相，EvalRun 是不可变执行证据，RegressionAssessment 只提供四阶段候选语义。长期演进围绕这三者完善发布门禁和 Registry 关系，不恢复旧全局用例、独立回归资产或场景包链。
