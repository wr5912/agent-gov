# 反馈闭环长期回归资产升级方案

> 文档定位：测试数据集、回归资产、发布门禁和资产 Registry 的长期设计。
> 文档层级：当前实现基线到四阶段改进治理的资产治理桥接方案。
> 替代关系：旧完整稿已归档到 `docs/archive/obsolete/反馈闭环长期回归资产升级方案_旧完整稿.md`；旧稿中的 UI 菜单、ASCII 草图、阶段进度和重复闭环架构不再作为活跃依据。
> 权威边界：改进治理工作台 UI、四阶段页面、决策卡、处理记录和效果图验收以 `docs/AgentGov_四阶段改进治理工作台UI整改方案.md` 为准。本文只定义测试资产和发布门禁如何沉淀。

## 1. 目标

长期回归资产治理解决四个问题：

1. 反馈不能只停留在一次性问题记录，必须能沉淀为防复发测试资产。
2. 测试数据集不能只是页面临时勾选状态，必须有归属、版本、来源和生命周期。
3. 发布门禁不能只看一次回归运行结果，必须关联覆盖场景、失败原因、豁免理由和版本风险。
4. 资产 Registry 不能只保存数据，还要关联方法论资产、执行资产和审计资产。

## 2. 治理对象矩阵

| 维度 | 设计结论 |
| --- | --- |
| 被治理对象 | 业务 Agent 的行为质量、反馈衍生测试资产、候选版本和发布风险。 |
| 治理执行者 | 后端门禁、治理 Agent、人工审核和发布审批组合执行。 |
| 资产类型 | 数据资产、方法论资产、执行资产、版本资产、审计资产。 |
| 生命周期 | `draft -> active -> evaluating -> deprecated -> archived`，必要时支持 `flaky`、`superseded` 等治理标记。 |
| 反馈归属 | 每个资产必须能追溯到 feedback、run、业务 Agent、版本、场景或人工创建来源。 |
| 当前实现边界 | 当前已有 eval case、regression asset、eval run、change set、release 等实现来源。 |
| 目标能力边界 | 四阶段改进治理测试发布阶段把 TestDataset、RegressionAsset、RegressionRun、ReleaseGate 收敛为同一治理视图。 |

## 3. 资产模型

### 3.1 TestDataset

`TestDataset` 是测试发布阶段的可版本化测试集合，不等同于一次回归运行。

最小字段：

| 字段 | 含义 |
| --- | --- |
| `dataset_id` | 数据集稳定 ID。 |
| `agent_id` | 所属业务 Agent。 |
| `name` / `description` | 面向用户和 Agent 的可读说明。 |
| `scope` | 适用场景，如 feedback-derived、release-gate、manual-eval。 |
| `version` | 数据集版本。 |
| `case_ids` | 关联 eval case 或 regression asset。 |
| `provenance` | 来源反馈、run、版本、人工导入或外部系统。 |
| `lifecycle_state` | draft / active / evaluating / deprecated / archived。 |
| `quality_tags` | flaky、critical、security、manual-review 等治理标记。 |
| `owner` | 维护方，可以是人工角色或治理 Agent。 |
| `created_at` / `updated_at` | 审计时间。 |

### 3.2 RegressionAsset

`RegressionAsset` 是从高价值反馈、人工评估或外部事件沉淀出的长期防退化资产。它可以被多个 `TestDataset` 引用。

核心要求：

- 必须有明确来源，不允许无 provenance 的长期资产进入发布门禁。
- 必须有适用 Agent 和适用版本范围。
- 必须保留修订历史和治理事件。
- 必须支持归档、flaky 标记、supersede 和人工豁免。
- 必须能说明防复发目标，而不是只保存输入输出样例。

### 3.3 RegressionRun

`RegressionRun` 是一次执行记录，绑定候选版本、测试数据集、执行环境、结果快照和失败分析。

它不替代 `TestDataset`，也不替代 `RegressionAsset`。一次运行可以失败、重跑或被豁免，但数据集和资产本身应保持可追溯。

### 3.4 ReleaseGate

`ReleaseGate` 是测试发布阶段的风险判断结果，至少包含：

- 使用的数据集和覆盖范围。
- 通过、失败、阻断、豁免和人工复核状态。
- 失败项影响分析。
- 是否允许发布、需要补测还是回到优化执行。
- 审批人、审批理由和审计记录。

## 4. 生命周期

```text
draft
  -> active
  -> evaluating
  -> deprecated
  -> archived
```

允许的治理标记：

- `critical`：发布门禁强依赖。
- `flaky`：不直接阻断发布，但必须展示风险。
- `manual_review`：需要人工确认。
- `superseded`：被更新资产替代。
- `external_source`：来源于外部系统或 Webhook。

资产晋级原则：

1. 单次反馈默认只能生成候选 eval case。
2. 只有高价值、可复现、可判分的 case 才能晋级长期资产。
3. 晋级必须记录原因、来源和预期防复发目标。
4. 发布门禁引用的是数据集和资产集合，不是随手拼出来的一次性 case 列表。

## 5. 测试发布阶段用户操作

测试发布阶段只保留这些与资产治理相关的主操作：

- 查看本次候选版本将运行的测试数据集。
- 管理数据集覆盖范围和关键场景。
- 执行回归并查看门禁结果。
- 查看失败项、影响分析和豁免理由。
- 将高价值用例晋级长期回归资产。
- 预览发布风险并决定发布、补测或回退优化执行。

旧“回归资产”独立主菜单、旧“版本管理”独立发布入口和旧批次回归区只作为迁移来源。四阶段改进治理工作台中，这些能力归入测试发布阶段。

## 6. 资产 Registry 关系

```text
Feedback / Run / ExternalEvent
  -> EvalCase
  -> RegressionAsset
  -> TestDataset
  -> RegressionRun
  -> ReleaseGate
  -> Release
  -> AssetRegistry
```

资产 Registry 应能同时回答：

- 这个资产来自哪个反馈或运行事实？
- 它保护哪个业务 Agent、版本和场景？
- 它依赖哪些方法论资产，如归因方法、评估规程、发布策略？
- 它触发过哪些执行资产，如 prompt、skill、playbook、workflow？
- 它在哪些发布门禁中起过作用？

## 7. 后续代码整改原则

1. `TestDataset`、`RegressionAsset`、`RegressionRun`、`ReleaseGate` 应按领域对象建模，避免继续把它们混成一次性 UI 状态。
2. 后端存储、API response、OpenAPI 和前端类型必须有单一真相来源。
3. 如果重构收益更大，不迁移旧回归资产页面结构、旧批次页面回归区或旧发布入口。
4. 旧字段、旧页面和旧 facade 只有在真实数据迁移需要时才保留，并必须写清退出条件。
5. UI 验收必须覆盖负向断言：测试资产入口不应回到旧独立主菜单，发布门禁不应脱离测试发布阶段。

## 8. 验收断言

- 活跃文档中只有本文定义测试资产模型，只有四阶段方案定义测试发布 UI。
- `TestDataset` 有 ID、归属、版本、来源、生命周期和审计字段。
- 回归资产晋级、flaky、归档、supersede 和豁免都有审计记录。
- 发布门禁能说明覆盖范围、失败原因、风险结论和人工决策。
- 新代码计划必须说明 API、OpenAPI、前端生成类型、SQLite schema、历史数据和测试覆盖如何同步。

## 9. 结论

测试数据集和回归资产是 AgentGov 的长期治理资产，不是一次性测试记录。后续代码整改应把它们纳入测试发布阶段和资产 Registry，以支撑业务 Agent 的持续优化、发布门禁和防复发能力。
