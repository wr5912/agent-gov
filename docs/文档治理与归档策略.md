# 文档治理与归档策略

本文档是 `docs/` 目录治理的入口说明和权威地图，用于说明文档治理时应先看哪里、由谁承载规则、如何验证结果。具体执行流程由 `docs-governance` skill 承载，活跃文档和归档文档分别由 README 索引维护。

## 治理目标

- 让活跃文档、历史文档、评审证据和工程流程各有入口。
- 消除同一主题的重复权威，避免旧设计继续影响当前整改。
- 保留有审计和迁移价值的历史文档，但移出活跃阅读路径。
- 把可复用的文档治理流程放入 skill，把可机械判定的规则放入脚本。

## 权威地图

| 治理对象 | 权威来源 | 说明 |
| --- | --- | --- |
| 活跃阅读路径 | `docs/README.md` | 新增活跃 Markdown 文档必须能从这里发现。 |
| 归档清单 | `docs/archive/README.md` | 记录原路径、归档路径、替代文档和归档日期。 |
| 术语和版本层级 | `docs/AgentGov术语与版本边界.md` | 区分长期产品术语、当前实现事实和 v2.7 规划。 |
| 改进治理工作台 UI | `docs/AgentGov_v2.7_四阶段改进治理工作台UI整改方案.md` | 四阶段主链路、效果图和代码整改原则的绝对依据。 |
| 文档治理动作流程 | `.codex/skills/docs-governance/SKILL.md` 与 `.claude/skills/docs-governance/SKILL.md` | 判断保留、合并、拆分、归档、删除或移入 skill。 |
| 文档边界矩阵 | `.codex/skills/docs-governance/references/docs-boundary.md` 与 `.claude/skills/docs-governance/references/docs-boundary.md` | 记录可复用判断标准和反例。 |
| 机械治理规则 | `scripts/check_docs_governance.py` | 检查 docs 入口、归档索引、skill 镜像和未完成标记。 |

## 使用原则

- 新增、移动、重命名、拆分、归档或删除 `docs/` 文档时，先使用 `docs-governance` skill。
- 涉及 AgentGov 产品定位、目标愿景使命、反馈闭环、多业务 Agent 或资产沉淀时，先使用 `agentgov-governance-preflight` 做内容建模，再决定文档容器动作。
- 与四阶段改进治理工作台 UI 整改方案冲突的旧 UI 设计、ASCII 草图、旧五类 Agent 工作台设计和旧验收口径，均不再作为未来主流程依据。
- 当前实现基线文档可以保留真实 API、数据库、测试和 UI 名称，但必须声明它们属于 pre-v2.7 事实，不定义 v2.7 目标状态。
- 旧文档不因“旧”而自动归档或删除；只有替代关系、引用迁移、审计价值和活跃阅读路径都清楚后才行动。
- 目录细则、归档明细和术语正文不放在本文档中，分别由 README、archive 索引和术语文档维护。

## 动作口径

文档治理输出统一使用以下动作，便于审查和复用：

| 动作 | 含义 |
| --- | --- |
| `keep` | 保留为活跃入口或当前实现基线。 |
| `merge` | 合并到现有权威文档，避免并列维护。 |
| `split` | 按读者、权威性或流程边界拆分。 |
| `move-to-skill` | 将可复用操作流程或判断标准移入 skill 或 reference。 |
| `archive` | 移入 `docs/archive/`，保留审计和迁移价值。 |
| `delete` | 删除没有长期读者、唯一信息或审计价值的临时材料。 |
| `no-op` | 保持现状，并说明本轮不改的原因。 |

## 验证

文档或 skill 变更后至少运行：

```bash
git diff --check
.venv/bin/python scripts/check_docs_governance.py
.venv/bin/python scripts/check_codex_governance.py --mode fail
```

移动、归档或删除文档前，先检查旧路径引用：

```bash
rg -n "旧文件名|旧相对路径" README.md docs .planning AGENTS.md AGENTS.override.md .codex .claude
```

验收标准：

- 活跃文档能从 `docs/README.md` 发现。
- 归档文档能从 `docs/archive/README.md` 追溯。
- 旧设计不会与四阶段 UI 整改方案并列成为未来依据。
- skill、docs 和脚本各司其职，不重复维护同一套长篇规则。
