# 文档治理与归档策略

本文档是 `docs/` 目录治理的规则权威来源，定义新文档、历史文档、权威文档和归档文档的准入、入口和归档规则。目标是让文档持续服务 AgentGov 的产品目标、工程闭环和治理沉淀，而不是把设计方案、测试用例、评审报告和历史复盘混在同一个文件里。

## 核心原则

- `docs/项目目标愿景使命.md` 是产品方向、目标、愿景和使命的权威来源。
- `docs/AgentGov核心功能测试用例.md` 是目标愿景使命的核心功能验收清单，不承载文档目录治理策略。
- `docs/README.md` 是活跃文档入口索引；新增活跃 Markdown 文档必须进入该索引。
- 设计方案、评审报告、工程过程、复盘材料和示例配置应保留各自边界。
- 旧文档不因“旧”而默认归档，只有在权威替代关系和引用迁移都清晰时才移动。
- 新文档进入 `docs/` 时，应同步说明读者、用途、权威性、替代关系和引用入口。
- `docs/AgentGov术语与版本边界.md` 是 docs 术语和版本层级的权威来源；当前实现基线与 v2.7 规划必须通过该文件说明映射关系。
- `docs/AgentGov_v2.7_四阶段改进治理工作台UI整改方案.md` 与四张效果图是改进治理工作台的绝对依据；与它冲突的 ASCII 草图、当前实现基线、旧设计评审和旧测试验收口径均须让位。
- 后续代码整改方案默认以四阶段重构为准；当重构或重新设计收益更大时，不把旧七段链路、旧四菜单、旧发布入口或旧 facade 作为兼容目标，除非用户明确批准。
- 报告和方案数量增长时，优先治理阅读路径和状态边界，不用目录搬迁掩盖权威关系不清的问题。

## 版本分层与术语治理

`docs/` 目前按四层治理：

| 层级 | 说明 | 处理方式 |
| --- | --- | --- |
| 长期产品权威 | 产品定位、目标愿景使命、核心验收用例 | 保持少量权威入口，使用长期产品术语 |
| 当前实现基线（pre-v2.7） | 解释当前代码、API、数据库、测试和部署运行态 | 保留真实实现名称，文档开头声明版本边界 |
| 下一大版本 v2.7 规划 | 跨代 UI / 领域模型 / 用户主流程设计 | 使用 v2.7 统一术语，不反向改写成旧实现口径 |
| 历史评审与复盘 | 设计审查、代码审查、配置治理反思 | 保留证据链，新增结论可引用术语边界文档 |

术语治理要求：

- `main-agent` 是当前默认业务 Agent 和首个样板，不得被写成 AgentGov 长期产品边界。
- 面向 v2.7 用户主流程时，统一使用业务 Agent、改进事项、反馈、系统理解、归因结果、优化方案、回归资产、发布、资产 Registry 等术语。
- 面向 v2.7 改进治理工作台时，用户主链路统一为反馈整理、归因分析、优化执行、测试发布四阶段；更细内部状态只能作为实现细节或迁移来源。
- 当前实现文档中的 API、数据库列、pytest nodeid、OpenAPI schema、配置项和文件路径保留真实标识符，不能为了术语统一伪造实现状态。
- 历史评审报告不做回写式改名；如存在旧词，应在新结论或引用处说明其版本层级。
- 新增 docs 时，如涉及 AgentGov 产品、反馈闭环、多业务 Agent、资产沉淀或 v2.7 UI，必须先确认应落在哪个版本层级。

## 活跃阅读路径治理

当 `docs/` 中同一主题同时存在主方案、补充方案、设计审查报告和历史复盘时，先在 `docs/README.md` 中明确阅读顺序和权威关系：

1. 主方案定义目标对象、术语和跨页面结构。
2. 补充方案只能细化主方案中的某个视角或用户任务。
3. 评审报告只作为差距清单和证据链，不提升为并列主方案。
4. 当前实现基线继续解释真实代码和运行态，不因 v2.7 规划出现而自动归档。
5. 只有替代关系、引用迁移和审计保留方式都清楚后，才移动到 `docs/archive/`。

改进治理工作台有一条特殊规则：四阶段整改方案和四张效果图优先于 ASCII 草图、旧 UI 补充方案、当前实现基线和历史评审。旧文档可以保留现状解释和审计价值，但必须在状态块中说明它不再定义未来用户主流程。

新增报告或方案前，应优先判断能否补到既有主方案、补充方案或评审报告中；只有读者、用途和权威关系不同，才新增文件。

## 新文档准入规则

新增 Markdown 文档到 `docs/` 前，先回答五个问题：

| 问题 | 判定要求 |
| --- | --- |
| 文档服务谁 | 面向产品决策、研发实现、测试验收、治理复盘还是外部集成 |
| 是否权威 | 是权威来源、阶段性方案、评审记录、历史复盘还是示例材料 |
| 与现有文档关系 | 新增、补充、替代、拆分、迁移还是归档 |
| 放在哪里 | 顶层 `docs/`、`docs/engineering/`、评审报告子目录、示例文件或未来归档目录 |
| 如何被发现 | 是否已从 `docs/README.md`、目标愿景使命、测试用例或相关方案文档链接 |

新增文档应尽量满足：

- 文件名能表达主题，不用临时命名。
- 开头说明文档定位和适用范围。
- 若替代旧文档，写明被替代文档和替代边界。
- 若只是历史记录，避免伪装成当前权威方案。
- 若包含测试、验收或治理准则，说明它绑定到哪个目标、流程或质量门。
- 若是活跃 Markdown 文档，必须在 `docs/README.md` 中出现其仓库相对路径。

## 目录放置建议

| 文档类型 | 建议位置 | 说明 |
| --- | --- | --- |
| 产品目标、愿景、使命 | `docs/` 顶层 | 作为长期权威来源 |
| 核心测试用例、验收矩阵 | `docs/` 顶层 | 直接服务目标闭环和开发验收 |
| 产品或架构方案 | `docs/` 顶层 | 若后续拆分，可从主方案链接到细分方案 |
| 工程流程、质量闭环 | `docs/engineering/` | 面向研发协作和持续治理 |
| 设计评审报告 | `docs/design_review_report/` | 保留评审证据链 |
| 代码与文档评审报告 | `docs/code_review_reports/` | 保留质量审计证据链 |
| Codex/Claude 配置治理复盘 | `docs/codex_setting_review_reports/` | 保留 agent 协作治理经验 |
| 图片和导图 | `docs/imgs/` | 由正文文档引用 |
| 过时且已替代文档 | `docs/archive/` | 仅在满足归档条件后移动 |

## 旧文档归档策略

当前不立即移动旧文档。原因是 `docs/` 中存在大量设计方案、评审报告、治理复盘和历史上下文，直接移动可能破坏引用链，也可能丢失审计价值。

v2.7 规划文档出现后，pre-v2.7 文档仍不自动归档。只要这些文档仍解释当前运行态、迁移边界、评审证据或测试基线，就应保留原路径并通过状态块声明其文档层级。

### 建议归档目录

未来需要归档时，统一使用：

```text
docs/archive/
```

可按类型继续分层：

```text
docs/archive/design/
docs/archive/review/
docs/archive/retrospective/
docs/archive/obsolete/
```

归档目录必须提供索引：

```text
docs/archive/README.md
```

索引表格至少包含：

| 原路径 | 归档路径 | 替代文档 | 归档日期 |
| --- | --- | --- | --- |
| `docs/旧文档.md` | `docs/archive/obsolete/旧文档.md` | `docs/新文档.md` | `YYYY-MM-DD` |

### 可以归档的条件

同时满足以下条件的文档，才移动到归档目录：

- 已被新的权威文档完全替代。
- README、docs、`.planning`、项目配置和代码注释中的引用已经更新。
- 文档开头已标明替代文档和归档原因，或归档索引已记录这些信息。
- 移动后不会影响当前开发、测试、部署或治理流程。
- 归档目录中有索引说明原路径、现路径、替代文档和归档日期。

### 不建议默认归档的文档

以下文档不应仅因“旧”而移动：

- 设计评审报告。
- 代码审查报告。
- 长程重构质量闭环。
- 治理反思与改进方案。
- 包含历史决策、失败案例、迁移边界或审计价值的文档。

这类文档可以在开头标注“历史记录”或“已被替代”，但保留原路径更利于追溯。

### 归档实施前验收

真正移动旧文档前，必须执行：

```bash
rg -n "被移动文件名|旧相对路径" README.md docs .planning AGENTS.override.md
```

成功标准：

- 活跃文档不再引用旧路径。
- 归档索引能指向当前权威文档。
- 设计、评审、历史复盘等审计链路仍可追踪。

## 配套 Skill

新增、移动、重命名或归档 `docs/` 文档时，应使用项目 skill `docs-governance` 做语义预检。该 skill 不替代本文档的规则权威，也不替代 `agentgov-governance-preflight` 对产品目标、愿景、反馈闭环和 Agent 治理内容的建模。

`docs-governance` 只负责文档容器治理：

- 新文档是否应进入 `docs/`，还是应写到当前线程、PR 描述、`.planning` 或代码注释。
- 文档类型、权威性和替代关系是否清楚。
- 是否把目录治理策略混进了产品目标、功能测试或实现方案文档。
- 移动或归档前是否完成引用检查。
- 是否运行了必要的文档和治理校验。

## 自动治理

可机械判定的文档治理规则由 `scripts/check_docs_governance.py` 执行，并已接入 `scripts/check_codex_governance.py`。现阶段自动检查包括：

- 新增活跃 Markdown 文档必须被 `docs/README.md` 引用。
- 新增归档 Markdown 文档必须被 `docs/archive/README.md` 引用，且归档索引包含固定字段。
- 项目专项 skill 的 Codex/Claude 镜像必须保持一致：`scripts/check_docs_governance.py` 会自动发现 `.codex/skills/*/SKILL.md` 与 `.claude/skills/*/SKILL.md` 中除明确例外外的同名项目专项 skill（比对时忽略 Claude 侧同源标注行）；`project-skill` 是通用模板、`codex-config-optimizer` 是 Codex-only 配置治理工具，不强制镜像。
- 变更的 `docs/`、`.codex/skills/`、`.claude/skills/` 下 Markdown 不得遗留未完成提示词（中英文待办、临时填充、空缺标注等），完整标记清单以 `scripts/check_docs_governance.py` 为准。
- `tests/` 不得遗留孤儿测试：不得 import `app`/`scripts` 中已删除的模块或符号；机械检查由 `scripts/check_orphan_tests.py` 执行，已接入治理硬门。

手工验证命令：

```bash
.venv/bin/python scripts/check_docs_governance.py
.venv/bin/python scripts/check_codex_governance.py --mode fail
```
