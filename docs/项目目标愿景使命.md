# 智能体治理平台 AgentGov 目标、愿景与使命

Agent Runtime · Feedback Loop · Version Governance

> 文档层级：长期产品权威。
> 术语口径：本文面向 AgentGov 长期产品边界，默认使用业务 Agent、治理 Agent、反馈闭环、版本治理和三层资产模型；当前实现或测试中出现的 `main-agent`、`feedback signal` 等名称只表示真实实现事实，映射关系见 [AgentGov术语与版本边界](./AgentGov术语与版本边界.md)。
> 归档边界：本文不因四阶段改进治理方案存在而归档；只有新的长期产品权威完全替代本文时才评估迁移。

## 系统定位

AgentGov 是一个面向通用智能体应用的治理平台。

它不是某个单一业务场景的 AI 助手，也不是只负责调用大模型的 Runtime 封装。它的准确定位是：以 Agent Runtime、Feedback Loop 和 Version Governance 为核心能力的智能体优化闭环治理系统。

AgentGov 通过统一的智能体运行时、反馈闭环和版本治理机制，支持创建不同业务 Agent，并把这些 Agent 在真实业务中的运行、反馈、归因、优化、评估和发布过程，沉淀为三类长期资产：

- 数据资产：运行记录、上下文、反馈、证据、评估结果和版本记录。
- 方法论资产：归因方法、优化 SOP、评估规程、发布准入规则和回滚策略。
- 执行资产：Agent 定义、prompt、skill、agent profile、playbook、评估用例、治理规则和自动化流程。

这三类资产共同构成一套可持续运转的 Agent 优化闭环治理体系。数据是基础，方法论提供判断框架，执行资产提供可落地的改进能力。

当前前端是 AgentGov 的开发调试与治理观察界面，用于验证 Runtime API、调试聊天链路、观察反馈闭环、管理评估与版本状态。面向最终用户的业务门户由外部业务系统承载，并通过 AgentGov API 接入智能体能力。

当前阶段聚焦 AgentGov 平台本身，以及在平台上创建、运行和持续优化智能体的核心闭环。
AgentGov 当前不提供通用协作看板、issue/task 生命周期、协作成员或 squad 模型。

当前不接入 Multica 或其他外部研发协作平台，也不定义智能体之间如何协作。研发过程使用
本地质量门和人工部署，不进入 AgentGov 产品 API。

## 使命

AgentGov 的使命是为通用智能体应用提供可靠、可治理、可集成、可持续进化的治理平台，让智能体运行中的数据、反馈、经验、方法和版本演进持续沉淀为可复用的优化闭环治理体系。

AgentGov 解决的不是“如何做一个单点 AI 助手”，而是如何让智能体在真实业务中长期可用、可评估、可改进：

- 通过 Agent 创建与配置能力，让不同业务场景定义自己的 Agent、profile、工具边界和行为包。
- 通过 Agent Runtime 记录智能体运行过程、上下文、工具调用、任务状态和输出结果。
- 通过 Feedback Loop 接收用户、业务系统、评估流程和外部治理系统中的质量信号。
- 通过归因方法和证据链，把失败案例转化为可解释的问题模式。
- 通过优化 SOP、prompt、skill 和执行计划，把问题模式转化为可落地的改进动作。
- 通过 Workspace pytest、平台测试运行和发布准入规则，验证改进是否有效且没有造成能力退化。
- 通过 Version Governance 管理智能体能力的变更、发布、回滚和审计。

## 愿景

AgentGov 的愿景是成为智能体应用的优化闭环治理基础设施。

智能体系统不应只依赖 prompt、模型调用和一次性输出，而应能把真实业务中的运行轨迹、用户反馈、失败案例、归因过程、优化动作、评估结果和版本演进持续沉淀下来。

这些沉淀不只是数据记录，更逐步形成一套组织级的 Agent 治理知识体系：

- 哪类问题应如何归因。
- 哪类失败应如何优化。
- 哪些 prompt、skill 和工具配置被证明有效。
- 哪些评估用例能够阻断能力退化。
- 哪些发布策略、回滚策略和审计规则适合当前业务风险。

外部业务系统可以像接入一套智能体治理操作系统一样接入 AgentGov：先创建适合自身场景的业务 Agent，再把任务、上下文、工具、反馈和质量信号传入 AgentGov，获得可追踪、可评估、可持续改进的智能体能力。

## 三层资产模型

AgentGov 的核心沉淀对象分为三层。

### 1. 数据资产

数据资产记录智能体运行和治理过程中发生了什么：

- 会话、任务、上下文、工具调用、trace 和 agent job。
- 用户反馈、业务系统反馈、人工复核结论和外部治理信号。
- 证据包、归因结果、优化方案、执行记录和评估结果。
- agent version、change set、release、rollback 和审计事件。

### 2. 方法论资产

方法论资产回答如何分析、如何优化、如何评估、如何治理：

- 归因分析方法。
- 反馈分级标准。
- 优化 SOP。
- 评估规程。
- 回归准入标准。
- 发布和回滚策略。
- 高风险动作审批原则。

### 3. 执行资产

执行资产让方法论可以被 Agent 和系统复用：

- Agent 定义和行为包。
- prompt 模板。
- skill。
- agent profile。
- playbook。
- evaluation case。
- regression dataset。
- governance rule。
- 自动化 workflow。

数据资产提供事实基础，方法论资产提供判断框架，执行资产提供可落地的改进能力。

## 三大核心能力

tagline `Agent Runtime · Feedback Loop · Version Governance` 不是三个并列的功能模块，而是一条治理链路上的三层：

- Agent Runtime 是事实产生层：记录智能体在真实任务中的上下文、工具、行为、结果和运行状态，为后续归因和评估提供事实基础。
- Feedback Loop 是经验转化层：把用户反馈、业务反馈、失败案例和测试结果转化为归因结论、优化 SOP、prompt/skill 改进、回归测试设计和 Workspace pytest 文件。
- Version Governance 是治理固化层：把经过验证的改进固化为版本，管理发布、回滚、审计和长期质量证据。

## 多 Agent 治理对象

AgentGov 治理所有注册业务 Agent（含 `main-agent`）。当前 `security-operations-expert` 是唯一内置、默认且受保护的业务 Agent，但这三个属性彼此独立，具体 Agent ID 不构成平台治理边界。

AgentGov 中存在两类 Agent：

- 业务 Agent：面向具体业务任务、由外部系统或平台管理员创建、配置、运行和持续优化的目标对象，是 AgentGov 的长期治理对象。
- 治理 Agent：服务于 AgentGov 自身闭环流程的系统执行者，用于完成归因分析、优化方案生成、执行优化、评估用例治理和回归影响分析，当前只作为治理流水线的一部分运行。

AgentGov 的产品目标，是用治理 Agent 支撑业务 Agent 的持续演进，而不是只让多个 Agent 并行运行。

### Agent 生命周期

每个业务 Agent 具有清晰的生命周期：

- draft：草稿定义，尚未进入生产使用。
- active：当前可运行版本。
- evaluating：待发布版本正在接受平台测试或发布检查。
- deprecated：仍可追溯，但不建议继续使用。
- archived：归档保留，只用于审计和历史复盘。

AgentGov 的治理闭环围绕 Agent 生命周期运转：创建定义、运行采集、反馈归因、优化候选、评估准入、版本发布、回滚归档。长期治理对象不是单次运行，而是 Agent 生命周期。

### 每个被治理 Agent 的资产构成

每个被治理的 Agent 至少包含：

- Agent 身份：名称、用途、适用场景、责任边界。
- 行为配置：system prompt、skills、tools、MCP、profile、模型参数和运行约束。
- 运行记录：会话、任务、上下文、工具调用、trace 和输出结果。
- 反馈闭环：用户反馈、业务系统反馈、质量问题和归因结果。
- 优化资产：prompt 改进、skill 改进、工具配置调整、SOP 和 playbook。
- 测试资产：随业务 Agent Workspace Git 版本化的 pytest 文件、派生 suite 摘要、平台运行证据和发布准入规则。
- 版本资产：change set、release、rollback、审计事件和版本说明。

### Agent 资产 Registry

AgentGov 维护 Agent 资产 Registry，记录并关联 Agent 定义、agent profile、prompt 和 system instruction、skills 和 tools、MCP 配置、SOP/playbook、evaluation case 和 regression dataset、change set/release/rollback、归因结论、优化任务和治理事件。

Registry 的价值不是简单列清单，而是建立资产之间的关系：某次反馈影响哪个 Agent，触发了哪个优化任务，修改了哪些 prompt/skill，经过哪些评估，最终进入哪个版本。

### 反馈路由与归属

多 Agent 场景下，每条反馈都能明确归属：

- 属于哪个 Agent。
- 属于哪个 Agent version。
- 来自哪个 run / session / task。
- 对应哪个业务场景或能力域。
- 是否影响 prompt、skill、tool、SOP、评估用例或版本策略。

反馈路由是多 Agent 治理的前提。没有反馈归属，归因、优化和版本治理都会退化成全局问题池，无法证明某个 Agent 的能力是否真的改进。

### 能力域与场景包

AgentGov 支持按能力域或场景包组织治理资产。一个场景包包含业务目标和适用范围、Agent 定义和行为边界、prompt/skill/tool 配置、SOP/playbook、评估用例和回归集、发布准入规则和风险等级。

由此，AgentGov 沉淀的不是单个 Agent 的零散配置，而是一套可迁移、可复制、可审计的能力包。

## 反馈到资产的闭环链路

AgentGov 把 Runtime、Feedback Loop、Version Governance 和资产沉淀串成一个系统，而不是几个并列名词：

```text
业务 Agent 运行
-> 产生 run / trace / output
-> 用户或业务系统反馈
-> 反馈归属到 Agent / version / 场景
-> 沉淀证据和归因
-> 形成优化 SOP / prompt / skill / RegressionTestDesign
-> 在同一 change set 生成待发布提交和 Workspace pytest 文件
-> 对待发布提交执行平台测试
-> 发布新版本或回滚
-> 更新 Agent 资产 Registry
```

## 核心目标

### 1. 支持创建和管理不同业务 Agent

AgentGov 提供通过 Workspace 包创建 Agent，以及配置、运行和治理 Agent 的统一能力。所有注册业务 Agent 接入同一套反馈优化和版本治理闭环；平台不以某个历史 Agent ID 建立专用路径。

### 2. 让智能体运行过程可记录、可复盘

Agent Runtime 承载智能体会话、任务执行、上下文组织、工具调用、profile 隔离、运行记录和异步 agent job。它不是简单封装模型调用，而是让每次智能体运行都能进入后续反馈、归因、评估和版本治理链路。

### 3. 让反馈进入标准化归因链路

AgentGov 不只收集“好评/差评”，而把反馈转化为问题分类、证据链、影响范围、原因判断和改进方向。

### 4. 让优化形成 SOP 和可执行资产

每次优化不停留在一次性修改，而沉淀为可复用的 prompt、skill、agent profile、playbook、评估用例或治理规则，并应用到对应业务 Agent 的后续版本中。

### 5. 让评估成为发布前的质量闸门

反馈和失败案例持续转化为评估资产，用于验证后续优化是否有效，以及已有能力是否发生退化。

### 6. 让版本治理固化能力演进

Version Governance 管理智能体行为包的候选变更、差异审查、回归验证、发布、回滚和审计，确保每次能力变化都有依据、有验证、有记录。

### 7. 通过 API 接入外部业务系统

外部系统负责自己的业务界面、权限、审批和生产流程；AgentGov 负责提供智能体运行、反馈闭环、评估和版本治理能力。

待 AgentGov 自身核心能力稳定且出现真实的多智能体协作需求后，再重新评估是否需要协作平台以及
采用何种产品。该后期方向不预设任何候选产品，当前不预先定义任务分配、成员、状态流转或
交付模型，也不属于当前阶段目标。

## 治理边界与审批

AgentGov 生成优化建议、候选变更、评估结果和发布建议，但高风险 Agent 行为变更必须经过外部系统或授权用户确认。

平台区分三类动作：

- 可自动执行的低风险优化。
- 需要人工审批的 prompt / skill / tool / SOP 变更。
- 必须由外部业务系统确认的生产动作和高风险策略变更。

AgentGov 负责让变更可解释、可评估、可追踪，不绕过业务系统的权限和责任边界。

## 治理成熟度路径

AgentGov 的多 Agent 治理分阶段演进，既保持愿景高度，也反映当前实现的真实边界：

- 第一阶段：以内置 `security-operations-expert` 验证运行、反馈、归因、优化、评估和版本治理端到端闭环，同时保持契约对业务 Agent ID 无关。
- 第二阶段：通过 Workspace 包接入更多业务 Agent，验证同一治理模型和隔离边界。
- 第三阶段：支持按场景包管理 Agent、prompt、skill、SOP、评估集和版本策略。
- 第四阶段：形成跨 Agent 的治理知识库和方法论资产，让不同业务 Agent 共享已验证的优化经验。
- 后期候选阶段：在 AgentGov 自身核心治理能力稳定、且出现真实多智能体协作需求后，重新做协作平台选型，不预设任何候选产品。具体协作对象、交互和状态模型届时基于真实需求设计。

内置业务 Agent 是可运行的参考实现和 Workspace 起点，不是平台模板，也不定义 AgentGov 的长期产品边界。

## 典型落地场景

AgentGov 是通用智能体治理平台，不绑定单一行业。典型落地场景包括：

- 安全运营：告警研判、事件响应、威胁狩猎、检测工程、SOC 反馈闭环。
- 客服与用户支持：问题归因、答案质量反馈、知识库优化、客服助手版本治理。
- 研发助手：代码建议、缺陷归因、测试生成、工程规范治理。
- 知识管理：问答质量反馈、知识缺口识别、内容更新闭环。
- 企业流程自动化：任务执行、人工审批、流程反馈和策略优化。

安全运营是 AgentGov 的重要示例场景，但不定义 AgentGov 的全部产品边界。

## 产品边界

### AgentGov 负责

- 智能体运行时和 Runtime API。
- Agent 创建、配置、profile 管理和行为包治理。
- 多 profile 智能体任务编排。
- 运行记录、trace、agent job 和上下文管理。
- 反馈信号采集与归因分析。
- 优化 SOP、prompt、skill、playbook 和执行计划沉淀。
- Workspace pytest 测试资产、`AgentTestRun` 和发布条件证据。
- agent version、change set、release、rollback 和审计事件。
- 开发调试前端和治理观察界面。
- 面向外部业务系统的 Runtime、反馈和治理能力接口。

### 外部业务系统负责

- 面向最终用户的业务界面。
- 业务流程、权限、审批和组织管理。
- 生产系统的数据源和执行系统。
- 行业特定对象模型，例如告警、工单、客户问题、代码仓库、知识库条目等。
- 高风险动作的人工确认和业务责任闭环。
- 当前阶段的通用协作看板、issue/task 生命周期、协作成员管理、squad 管理、状态流转和团队交付节奏。

## 一句话总结

AgentGov 是通用智能体应用的优化闭环治理平台：支持创建、运行、反馈优化和版本治理不同业务 Agent，并通过 Agent Runtime、Feedback Loop 和 Version Governance，把智能体运行、反馈、归因、优化、评估和发布过程持续沉淀为数据资产、方法论资产和执行资产，形成一套可复用、可评估、可版本化的 Agent 治理系统。当前重点是把智能体开发与反馈优化闭环做强。
