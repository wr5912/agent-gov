# AI 网络安全运营智能体底座目标、愿景与使命

## 系统定位

本系统不是单一的“威胁分析平台”，也不是面向终端安全运营人员的完整 SOC 前端产品。

本系统更准确的定位是：

**面向 AI 网络安全运营场景的智能体底座与 Runtime API 服务。**

它为外部网络安全运营系统提供可集成的智能体能力，包括聊天对话、任务执行、子智能体调度、MCP 数据访问、反馈采集、反馈归因、优化闭环、回归验证和版本治理。

当前前端主要是本系统的开发调试前端，用于验证 Runtime API、调试聊天链路、观察反馈闭环、管理回归与版本状态。真正面向业务用户的安全运营门户、告警工作台、工单系统、事件响应平台、知识库平台或管理驾驶舱，可以由外部网络安全运营系统承载，并通过本系统 API 接入 AI 智能体能力。

## 使命

本系统的使命是：

**为网络安全运营系统提供一个可靠、可治理、可集成、可持续进化的 AI 智能体底座，让外部安全运营平台能够通过 API 安全地接入智能体能力。**

它要解决的不是“做一个漂亮的 SOC 页面”，而是提供一套可被外部系统调用的 AI 运营能力基础设施：

- 通过聊天 API 提供面向安全运营任务的智能问答、分析和任务编排能力。
- 通过反馈 API 接收外部系统、用户和运营流程产生的质量反馈。
- 通过多智能体机制支持告警研判、威胁狩猎、事件响应、检测工程、策略规划、知识检索和报告生成等专业能力。
- 通过运行记录、证据、trace、agent job、回归用例和版本治理，让智能体能力可以被审计、验证和持续改进。
- 通过 API 边界与外部安全运营系统集成，而不是要求用户必须使用本系统自带前端完成所有业务操作。

## 愿景

本系统的愿景是成为网络安全运营智能化的基础设施层。

外部安全运营系统负责承载用户界面、业务流程、告警队列、工单流转、资产视图、事件管理、组织权限和管理报表；本系统负责提供底层 AI 智能体能力：

- 能理解安全运营任务。
- 能调用安全数据源和工具。
- 能调度专业子智能体。
- 能输出可追踪的分析、建议、报告和处置计划。
- 能接收业务系统反馈。
- 能把失败案例转化为回归用例。
- 能通过版本治理持续改进智能体能力。

理想状态下，外部系统可以像调用一个安全运营智能体平台一样调用本系统：把告警、事件、资产、IOC、漏洞、工单、用户反馈和上下文传入 Runtime API，获得结构化、可审计、可继续流转的智能体输出。

## 核心目标

### 1. 提供稳定的智能体 Runtime API

系统首先应提供稳定、清晰、可集成的 API，而不是把能力绑定在当前调试前端里。

核心 API 能力包括：

- 聊天与流式聊天。
- 会话与运行记录。
- agent、skills、配置和工具发现。
- 反馈信号采集。
- SOC 事件和外部反馈接入。
- 反馈处置单、证据包和归因分析。
- agent job 队列和异步任务状态查询。
- 优化方案、执行计划、回归评估和版本治理。

外部系统应能够通过这些 API 把本系统嵌入自己的安全运营流程。

### 2. 支撑网络安全运营多智能体能力

main 智能体是 AI SecOps 能力入口和调度中枢，负责理解任务、组织上下文、选择子智能体、调用 MCP 数据源并输出结果。

业务子智能体承担专业分工：

- `soc-analyst`：告警研判、安全事件初筛、证据链整理和升级建议。
- `threat-hunter`：IOC、TTP、进程链、网络行为和横向移动迹象的假设驱动狩猎。
- `incident-responder`：事件响应计划、隔离/阻断/恢复建议、回滚方案和验证方案。
- `detection-engineer`：检测规则缺口分析、检测逻辑设计和调优建议。
- `policy-operator`：防火墙、EDR、WAF、IAM、邮件网关等安全策略变更方案。
- `knowledge-curator`：SOP、处置手册、检测规则说明、FAQ 和历史经验沉淀。
- `report-writer`：安全运营日报、周报、事件报告、复盘报告和管理层汇报。

这些能力通过 Runtime API 暴露给外部系统，而不是只服务当前调试 UI。

### 3. 明确前端边界

当前前端的主要职责是开发调试和内部验证，包括：

- Playground 聊天调试。
- 会话、subagents、skills 和配置发现。
- 反馈信号和反馈处置链路验证。
- 归因分析、优化方案、回归评估和版本治理调试。
- API 契约和主流程状态观察。

它不是最终的网络安全运营业务前端，不应承担完整 SOC 产品的信息架构和用户流程。外部网络安全运营系统可以根据自身业务形态构建正式用户界面，并通过 API 接入本系统能力。

### 4. 以反馈 API 驱动持续进化

本系统不仅要回答问题，还要吸收外部系统和真实业务流程中的反馈。

反馈可以来自：

- 用户对聊天结果的显式反馈。
- 外部告警工作台的误报、漏报、升级或关闭结果。
- 工单系统中的人工复核结论。
- 事件复盘中的问题记录。
- 外部治理系统发现的知识库、工具、规则或流程缺口。

这些反馈进入统一闭环后，应能够形成归因、优化方案、执行计划、回归用例和版本变更，让智能体能力持续改进。

### 5. 保证智能体能力可治理

作为底座系统，智能体能力必须可控、可审计、可回归。

系统需要保证：

- 每次运行有 run、session、trace 和 agent version 关联。
- 每次反馈可以追溯到原始运行和上下文。
- 每次优化可以看到原因、方案、变更和结果。
- 每次发布前可以通过回归验证。
- 每个版本可以被识别、切换和追踪。
- 高风险动作必须由外部系统或用户确认，不能由智能体绕过审批直接执行。

### 6. 支持外部安全运营系统集成

本系统应把外部系统视为主要消费者。

典型集成对象包括：

- SOC 告警平台。
- SIEM / SOAR。
- EDR / XDR 平台。
- 漏洞管理平台。
- 资产管理平台。
- 工单系统。
- 安全知识库。
- 报告与管理驾驶舱。
- 外部治理或质量评估系统。

本系统通过 API、MCP、Webhook 或其他集成方式，为这些系统提供 AI 智能体能力和反馈优化能力。

## 产品边界

### 本系统负责

- AI 智能体运行时。
- 聊天和流式聊天 API。
- 多智能体能力组织。
- MCP 工具和安全运营数据访问。
- 反馈采集与反馈优化闭环。
- agent job 异步任务。
- 回归用例和评估运行。
- 智能体版本治理。
- 开发调试前端。

### 外部系统负责

- 面向最终用户的正式 SOC 页面。
- 告警队列和事件工作台。
- 工单流转和组织审批。
- 用户、角色和租户权限。
- 资产全景、漏洞全景和业务拓扑。
- 管理驾驶舱和业务报表。
- 企业内部数据源和生产处置系统。

## 一句话总结

**本系统的目标是成为 AI 网络安全运营的智能体底座：通过稳定的 Runtime API 向外部安全运营系统提供聊天、分析、反馈、评估和版本治理能力，让外部系统可以安全、可控、可持续进化地接入 AI SecOps 智能体。**
